# Agent Knowledge Host — Single-Operator Fork (SOLO v1)

> **Lineage.** This is a deliberate fork of the multi-tenant platform spec ([`agent-knowledge-host-spec-PLATFORM-v2.md`](agent-knowledge-host-spec-PLATFORM-v2.md)), descoped to a single operator with a public read surface. The platform spec is preserved unchanged as the blueprint for "if this ever becomes a service." This document is **almost** a strict subset: nearly every mechanism here is carried over identically or simplified, so growing back toward the platform is mostly additive, not a rewrite. The one deliberate *divergence* (not a simplification) is the **slug lifecycle** — SOLO makes slugs non-reusable where the platform frees them on delete; it is called out in §0 below and, on the platform side, at its freed-slug section, so the lineage stays auditable. Where a platform section was dropped, it is likewise listed in §0 with the reason.

## 0. What this is, and what was dropped

**The itch this actually serves.** My agents write reports and other HTML content I want to share externally with nothing more than a link, without the manual copy-into-Google-Drive-then-share dance. This is a personal tool with a public *read* surface — not a platform.

**Three actors, and only three:**
- **The operator** (me) — the sole owner. Full control: manage agents and keys, write, delete, revoke.
- **Agents** — authenticated by per-agent API key; can **read and write**. Per-agent keys exist specifically so a rogue or compromised agent can be contained (revoked) without disturbing the others.
- **Link-holders** — unauthenticated, **read-only**. The capability is *possession of the unguessable URL*. There is no login for readers.

**Dropped from the platform spec (with reason), so the descope is explicit:**
- **Multiple operators / the operator-access invariant / `can_access` matrix** — there is one operator; ownership is implicit. The entire access-control engine collapses to "mine" + "link-published."
- **Projects, share policies (`operator_only`/`writing_agent_only`/`all_*`/`explicit`), the polymorphic `shares` table** — no multi-principal sharing exists. A document is reachable by its unguessable URL or revoked. That is the whole access model.
- **The admin / Trust-&-Safety service (platform §12–§13): reports intake, enforcement_actions, platform_admins, the moderation queue, withhold-vs-delete, statement-of-reasons, DMCA repeat-infringer machinery** — there is no tenant population to police. This is the section whose weight motivated the fork; it is gone. (The residual publisher duties that *don't* vanish are in §8.)
- **Legal holds / the two-counter quota / hold-invisibility / the `held` cache and its reconciler** — all existed for the multi-tenant + preservation-duty context. Replaced by a single simple storage cap (§7) and an ordinary delete.
- **The Organization principal seam** — irrelevant at one operator.

**Carried over, because they are load-bearing even at this scope:**
- **Sanitize-and-sandbox-on-serve** (platform §6/§7). This is the one heavy piece that *survives*, and it survives *because* of the public-read requirement: anything reachable at a URL in someone's browser is published to the open web, so it must be sanitized and served in an isolated, network-restricted sandbox. The threat is not a malicious *author* (my agents aren't adversarial) — it is (a) an agent that gets prompt-injected into emitting something hostile, and (b) the simple fact that public bytes render in other people's browsers.
- **Per-agent API keys + revocation** (rogue-agent containment).
- **Versioning** (agents iterate; cheap and useful).
- **The sandbox + strict-CSP serving model** (platform §7), but **not** the platform's per-document origins. SOLO documents are served sandboxed under a strict CSP; the isolation that matters here is **document-bytes-from-the-service-origin** (achieved by serving documents from a single separate content domain — intended; see §5), *not* documents-from-each-other. The earlier "per-document origin doubling as the capability" idea is dropped: the capability is the `public_id`/`slug` URL (§4), and the serving origin no longer carries it.

**Diverged from the platform spec (the one place SOLO is not a strict subset):**
- **Slug lifecycle — non-reusable, where the platform frees slugs on delete.** The platform's "freed-slug" model (PLATFORM §3.2 / §3.5.1 / table row "Freed slugs") deliberately mirrors a web host: soft-deleting a document **frees its slug immediately** for reuse, and a restore-time collision auto-renames the *restored* doc. SOLO **reverses** this: a slug, once claimed, is **retired forever** (revoke/rename/clear move it to `slug_tombstones`, §2/§4; `/s/<slug>` 410s; a reclaim → `409 slug_retired`). Rationale (GitHub issue #6): at SOLO scale a slug is a public, shareable, cross-linkable handle, and silent reuse — a bookmarked `/s/<slug>` later serving unrelated content — is a footgun that outweighs the web-host convenience the platform optimized for. This is a true behavioral divergence, not a descope: growing back toward the platform here would mean *changing* a rule, not adding one. (For the legitimate rename/consolidation case SOLO provides a **loud, opt-in redirect** — §4 — distinct from the platform's *silent* reuse: a retired slug can forward to another document, but only via a browser click-through or an agent's explicit `follow_redirects`.)

## 1. Principals and authentication

- **Operator** — authenticates with a static **operator token** (`OPERATOR_TOKEN`, a high-entropy server secret, constant-time compared). For browser convenience there is also a **signed-cookie session** derived from that same token — log in once instead of pasting the token on every action — resolved *after* a bearer token, so programmatic callers are unchanged. The single owner; the only principal who can mint/revoke agent keys, delete or revoke documents, change a document's visibility/slug/tags, and restore a historical version. **The operator is also a first-class _author_** — it can publish and update documents directly (the JSON `POST /admin/documents` / `PUT /admin/documents/:id` door, the companion-app authoring path), not only administer what agents write. An operator-authored document records its creator as the operator principal rather than as a fake agent (see §6), so the audit trail stays honest. **The operator's two-faced surface is the same authority through two skins:** a JSON `/admin/*` API for programmatic/token-bearer use, and a server-rendered, no-JavaScript **browser console** at `/admin/console` for hands-on fleet work (the as-built console bullet below). Both resolve to the one operator principal — the console is just the cookie-session/CSRF skin over the same admin authority, not a new trust root. *(Multi-operator is a **deferred** seam: there is exactly one operator because there is one operator token, and the cookie-session signing key is derived from it — distinct operators would require per-operator authentication and an `operators` table, a platform-axis evolution. The per-document/per-version authorship discriminator added for operator authoring is the only forward-compatible artifact built now; nothing here assumes the operator is plural.)* **There is no third-party IdP / Google-OAuth login for the operator** (formally cancelled): the operator token is the sole operator trust root, and the cookie session adds no new root (rotating the token, or bumping the session epoch, ends every session). *(An OAuth 2.1 + PKCE flow does exist, but it authenticates **agent** connectors — e.g. Claude / Cowork / ChatGPT — and resolves to an agent identity, never to operator authority. A connector's agent identity is established either when its OAuth client is minted **bound** to an existing agent, or — for a client minted **unbound**, *or one that self-registered via dynamic client registration* (RFC 7591; a DCR client writes no `client_id ↔ agent_id` row, so it is unbound by definition) — at the consent screen, where the operator picks an existing agent or mints a new one; the `client_id ↔ agent_id` binding is then written and `props.agentId` is re-derived from that binding, never trusted from the submitted form. Dynamic client registration therefore confers **no authority** — it only removes the manual client_id paste from the connect flow; the agent is still chosen at the operator-gated consent screen, and a dynamically-registered client expires 90 days after registration (an absolute ceiling, not reset by use). The consent screen is itself operator-gated and session-aware (the same cookie session, or a pasted token), and it lets the operator inline-approve a connector's new redirect callback — limited to a small allowlist of pre-trusted vendor hosts that doubles as the consent page's CSP `form-action` source, so a host the browser would block on the post-grant redirect can never be approved. None of these paths confers operator authority; each resolves only to an agent identity.)*
- **Operator browser console (as-built).** Beyond the per-document **Manage** page reachable from a document shell (§4/§6), the operator has a dedicated, server-rendered **admin console** at `/admin/console` for fleet-wide work — the one place to do everything the JSON `/admin/*` API does without scripting a token. It is **cookie-session-authed (the operator session above) and entirely no-JavaScript**: every page is HTML rendered server-side, and every action is a plain `GET`/`POST` form. A logged-out visitor gets a sign-in card with **no DB hit** (the console is never an existence oracle), and bare `GET /admin` 302-redirects into it. The surface, all under `/admin/console`: a **dashboard** (`/admin/console`); **agent + key + OAuth-client management** (list/inspect agents at `/admin/console/agents` and `/admin/console/agents/:id`, mint an agent, mint a per-agent key, mint a bound OAuth client, mint an *unbound* OAuth client, and revoke an agent / a key / delete an OAuth client); a **documents browser** (`/admin/console/documents?q=&tag=&slug=&cursor=&limit=`) that reuses the existing hybrid-search + list cores untouched — it does **no** server-side visibility filtering, only renders a `Public`/`Private` **badge** per row, and offers the same tag/slug filters and keyword/semantic search the agent surface has; and a **maintenance** page (`/admin/console/maintenance`) fronting the operator Vectorize **backfill** (`POST /admin/console/vectors/backfill`). Per-document visibility / slug / tags / version-history / restore stay on the **Manage** page (§4/§6), now understood as the per-document leaf of this same console. **Two implementation notes that are deliberate, not accidental:** (1) every console mutation uses the **form-field CSRF ladder** — a pasted `operator_token` field *or* the cookie session plus a `csrf_token` form field — the same ladder the `/login`, `/logout`, and Manage-page forms use, **not** the `X-CSRF-Token` header `requireOperator` wants (a no-JavaScript form cannot send a custom header). (2) Several console actions are **`POST` twins of JSON `DELETE`s** (agent / key / OAuth-client teardown) and one is a `POST` twin of the per-doc tag write — an HTML form can only `GET`/`POST`, so the destructive verb is carried in the path + a body field (`agent_id` / `key_id` / `client_id` / `tags`) rather than in the HTTP method. The console adds **no new authority and no new core write path**: it is a thin operator-UI skin over the existing `*Core` functions and admin handlers, so the JSON `/admin/*` API is unchanged by it.
- **Agent** — authenticates with a per-agent API key: a ≥32-byte CSPRNG secret, stored as HMAC-SHA-256 under a server pepper, constant-time compared (a key-verification construction, **not** a password hash). The key resolves to an agent identity owned by the operator. Revoking a key (`revoked_at`) fails it closed on the next request — the rogue-agent kill switch.
- **Link-holder** — not authenticated. Presents an unguessable document URL; possession is the capability (§4). Read-only, always.

There is no cross-principal access logic to evaluate. A request is operator (session), agent (key), or anonymous (public read) — and what each may do is fixed by the lists above, not computed per-document.

## 2. The document model

A **document** is an agent- or operator-authored HTML artifact, versioned, with an unguessable `public_id` and an optional discovery/linking `slug` (§3). Both the sanitized **served** bytes and the **pre-sanitized source** are stored, one of each per version (convert-and-discard has been **reversed**; §5); only the sanitized bytes are ever served to a browser.

The schema below is the as-built shape (Cloudflare **D1** / SQLite, hence `text`/`integer` columns; ids are TEXT UUIDs and `public_id`s are 22-char base64url tokens):

```
documents
  id              text pk               -- internal row id (UUID)
  public_id       text unique not null  -- the UNGUESSABLE capability (§4). 22-char base64url CSPRNG; the shareable URL.
  slug            text                  -- OPTIONAL discovery/linking name (§3). NULL = no slug = the live slug.
                                        --   Uniqueness is a PARTIAL unique index (where slug is not null). NOT released on
                                        --   revoke/rename: the old value is RETIRED into slug_tombstones (§4), reserved forever.
  tags            text                  -- DOCUMENT-LEVEL classification, JSON-encoded array (nullable) (migration 0012).
                                        --   Like slug: identity/classification, NOT content — survives content updates and
                                        --   restore unless explicitly changed; changing tags does NOT bump a version.
  current_ver     integer               -- monotonic version pointer; the ETag (§6). Null only transiently at create.
  created_by      text references agents(id) on delete set null  -- creator AGENT; null if operator-authored OR the agent was deleted
  created_by_kind text not null default 'agent' check (created_by_kind in ('agent','operator'))
                                        --   creator's PRINCIPAL kind (§6, migration 0013). 'operator' when the operator authored
                                        --   the doc (created_by then null); disambiguates a null created_by meaning "operator"
                                        --   from one meaning "agent since deleted". DEFAULT 'agent' backfills legacy rows.
  revoked_at      text                  -- null = live. Set = the kill switch: public_id 404s; slug retired → /s/ 410s (§4).
  visibility      text not null default 'public' check (visibility in ('public','private'))
                                        --   per-document public/private axis (§4, migration 0011). ONLY gates the anonymous
                                        --   public browser surface (private → 404 to anon, like missing). Operator/agents read all.
                                        --   Operator-only to change. DEFAULT 'public' backfills legacy rows; NEW docs are born
                                        --   at DEFAULT_DOCUMENT_VISIBILITY (a [var], default 'private'), bound explicitly on write.
  status          text not null default 'active' check (status in ('active','deprecated','archived'))
                                        --   LIFECYCLE classification (migration 0014) — the third state axis beside revoked_at
                                        --   (existence) and visibility (anonymous read): "is it current?". 'deprecated' = still
                                        --   served + searchable (marked in hits) but EXCLUDED from context-pack fills by default.
                                        --   'archived' is RESERVED in the CHECK (unsettable, no behavior in v1). Document-level
                                        --   like tags/slug: never bumps a version; operator-only to change. NEVER gates access.
  superseded_by   text                  -- a replacement document's public_id, set only on a deprecated doc (migration 0014).
                                        --   The document-level analogue of slug_tombstones.redirect_to: surfaced LOUDLY,
                                        --   never auto-followed (packs substitute it only on explicit follow_redirects, and
                                        --   even then the original stays visibly reported). Single-hop; live-target +
                                        --   no-self-pointer validated on write; forced null when status returns to 'active'.
  created_at      text not null default (strftime now)

versions
  document_id     text not null references documents(id) on delete cascade
  version_no      integer not null      -- monotonic per document; the ETag (§6)
  r2_key          text not null         -- R2 object key for the served sanitized bytes H (§5)  [key shape <id>/v<n>]
  size_bytes      integer not null      -- rendered H byte count, for the storage cap (§7)
  source_r2_key   text                  -- R2 key for the retained pre-sanitized source S (§5)  [key shape <r2_key>.src]
                                        --   NULL = legacy/un-backfilled (loud source_unavailable on source-read; no fallback to H).
  source_size_bytes integer             -- source S byte count; also counts toward the cap (§7). NULL on legacy rows (coalesced to 0).
  source_sha256   text                  -- SHA-256 of the retained source S bytes (migration 0015, §6). NULL on pre-0015 rows.
                                        --   The currency token: surfaced so an agent can confirm a LOCAL copy is still the current
                                        --   source (compare to `sha256sum file`) and skip a source re-read before an edit.
  sanitizer_v     text not null         -- which sanitizer profile produced the served bytes (§5); the re-heal trigger/stamp
  source_format   text not null default 'html'  -- 'html' | 'markdown' — which input path produced this version; the source's language
  title           text                  -- per-version metadata (nullable)
  description     text                  -- per-version metadata (nullable)
                                        --   (tags moved to documents — document-level since migration 0012; no versions.tags column)
  author_kind     text not null default 'agent' check (author_kind in ('agent','operator'))
                                        --   this version's WRITER principal kind (§6, migration 0013). Per-version, so a doc's
                                        --   full author list is queryable (creator may differ from a later editor).
  author_agent_id text references agents(id) on delete set null
                                        --   the writing agent's id; NULL for an operator-written version (or a pre-0013/legacy
                                        --   version — not backfilled; the historical writer survives only in R2 customMetadata).
  created_at      text not null default (strftime now)
  primary key (document_id, version_no)
  -- NOTE: no sanitized_hash / model column. The pre-sanitized source IS retained (source_r2_key, §5).
  --   The per-version writer IS recorded in D1 (author_kind/author_agent_id, migration 0013) — D1 is the
  --   authoritative, queryable record; the R2 customMetadata writer tag is kept only as a best-effort echo.

  id              text pk
  name            text not null
  created_at      text not null default (strftime now)

agent_keys
  id              text pk
  agent_id        text not null references agents(id) on delete cascade
  key_prefix      text not null         -- indexed lookup
  key_hash        text not null         -- HMAC-SHA-256(secret) under server pepper
  revoked_at      text                  -- null = active
  expires_at      text                  -- null = never; non-null only on short-lived publish credentials
  created_at      text not null default (strftime now)

  slug            text pk               -- the retired name; reserved forever. A claim checks documents.slug ∪ this set.
  document_id     text references documents(id) on delete set null  -- origin doc (audit); set null if the doc row ever goes away
  retired_at      text not null default (strftime now)
  reason          text not null         -- 'revoked' | 'renamed' | 'released'
  redirect_to     text                  -- optional forwarding target (migration 0010): the TARGET document's
                                        --   public_id. NULL = plain 410 tombstone. Set on rename (auto, → same doc)
                                        --   or by the operator (cross-doc). Resolved LOUDLY at /s/ (interstitial /
                                        --   409 slug_redirected / follow_redirects); single-hop (points at a public_id).
```

`public_id` is globally unique; `slug` is optional, so its uniqueness is a **partial** unique index (`where slug is not null`) binding only live, slugged documents. A slug is **not** released on revoke — it is retired into `slug_tombstones` (above) and reserved forever, so name-uniqueness effectively spans `documents.slug ∪ slug_tombstones.slug` (§4). There is one operator, so no tenant partition. No `deleted_at`/soft-delete machinery: deletion is revoke (§4) plus, if the operator chooses, an actual purge of the bytes.

**Storage & auxiliary tables (implementation).** Metadata lives in **D1** (above); the one sanitized blob per version lives in **R2** (`DOCS` binding, key `<id>/v<n>`); the agent-OAuth provider's state lives in **KV**. Two tables exist beyond the core model: `oauth_clients` (a `client_id ↔ agent_id` join backing the agent connector OAuth door — §1; the row is **optional** — a client minted *unbound* lives only in KV until the operator binds it to an agent at the consent screen, so absence of a row *is* "unbound," and `UNIQUE(agent_id)` keeps it one-client-per-agent) and `documents_fts` (an FTS5 virtual table over `title`/`description`/`body`, one row per live document, backing full-text search — tags are **not** full-text-indexed since migration 0012; the tag *filter* runs against `documents.tags` directly). Neither changes the access model; both are noted so the schema matches the build.

**Search is hybrid — keyword + semantic (as-built).** Beyond the FTS5 keyword index, a **semantic axis** exists: each live document is embedded as N ~512-token chunk vectors (Cloudflare **Vectorize** index, 1024-dim cosine, Workers AI `@cf/qwen/qwen3-embedding-0.6b`), keyed `${document_id}#${chunk}`. The search surface (`search_documents` / the agent-reachable `GET /d/search` / its operator twin `GET /admin/documents/search`) defaults to **hybrid** — it runs both the keyword and the vector ranking and fuses them with Reciprocal Rank Fusion — with `keyword`-only and `semantic`-only modes available. **Crucially, this changes nothing about the access model:** Vectorize is a candidate *ranker*, **never an access gate**. A vector hit yields only a document id; the authoritative "is this document live / does this caller see it" check is the same D1 re-join (`revoked_at IS NULL` + the tag/slug filters) the keyword path uses, so a stale or not-yet-purged vector can never surface a revoked or filtered-out document. The vector store is **eventually-consistent and best-effort** (it is not transactional with D1; it is synced after the metadata write commits via `ctx.waitUntil`, and an operator backfill endpoint reconciles drift) — a missing vector degrades to "keyword still finds it," never a correctness or access fault. No new migration: the vectors live entirely in Vectorize, recoverable by id. *(Vectorize sits outside the capability/visibility boundary by construction — a relevance hint layered over the authoritative D1 gate, not a new read path.)*

**Context packs — budgeted bulk read (as-built; [`context-packs-design.md`](context-packs-design.md), migration 0014's consumer).** The corpus's "bring an agent up to speed" verb: one call returns full document **bodies** (always the markdown derivation of the sanitized bytes — never raw HTML, never unsanitized source) filled **best-first under a byte budget**, with everything that didn't fit **reported, never truncated** (an `omitted[]` entry carries the reference, the reason — budget / count-cap / deprecated / unavailable / revoked — and, for deprecated members, the `superseded_by` pointer). Two roots, one envelope: an **automatic** pack amplifies a search result (`include_bodies` on the search surface — pure amplification, no storage), and a **curated/ad-hoc** pack roots at a document (`load_context_pack`): its members are an explicit fenced ` ```pack ` **manifest block** in the document's retained source when present (one slug-or-public_id per line, an `[optional]` tier whose entries carry one-line hints — echoed even when omitted, so the pack doubles as a menu), else the document's **outbound `/d/`+`/s/` links** in order of appearance — any hand-written index page is a pack with zero ceremony, and a manifest wins when both exist. The lifecycle `status` axis (schema above) is the pack's trust guard: **deprecated documents are excluded from automatic fills by default**, so a superseded note that still ranks in search cannot confidently mis-onboard an agent. Packs change **nothing** about the access model: they are agent/operator-gated surfaces composing existing reads (a pack returns only documents the caller could already `read_document` one at a time — under the single-tenant whole-fleet trust model that is the entire corpus), and the anonymous render gate is untouched.

## 3. Two names, two jobs — `public_id` and `slug`

This is the crux of the design and the part most worth getting right, because it is what lets documents both be shared by link *and* hyperlink to each other.

- **`public_id` — the unguessable capability.** High-entropy (≥128 bits; UUIDv4 or a ≥22-char base62 CSPRNG token). It *is* the security boundary (§4): the public share URL is `https://<host>/d/<public_id>`. Unguessable, non-enumerable, never sequential. This is what I copy and send to someone.
- **`slug` — the discovery + linking name.** Agent-chosen, human-meaningful (`north-island-report`), `unique`. It does two jobs. **(1) Discoverability** — a slugged document is reachable at the short, typeable `https://<host>/s/<slug>` URL, which **serves the document in place** — the slug stays in the address bar, content-negotiated exactly like the `public_id` URL (§5); there is no redirect. This is for documents *meant* to be found by name: paste a clean link, print it on something physical, point a human at it without the opaque id. **(2) Inter-document links** — an agent authoring document A needs a stable, predictable name to link to document B, since it cannot use B's server-minted random `public_id`. So A links to B by its `/s/<slug>` URL, which the public `/s/<slug>` route serves in place at click/read time (§4). Because that URL is stable and predictable, A can link to B *before B exists*; the two can be authored in either order, or at once. **A slug is opt-in, not a default.** Unlike `public_id`, a slug is **low-entropy and externally visible** — the `/s/<slug>` endpoint is public (§4), so a slug is a deliberate, *weaker* capability traded for findability. **Most documents should not carry one:** a document you only ever share by its `public_id` URL is strictly more private without a slug, and that is the right default. Reserve slugs for documents that are meant to be found by name or linked to from another document.
- **`label` — optional cosmetic alias** the operator might one day set when sharing, purely for human readability. It would resolve to the same document, never be the security boundary, and carry no uniqueness-collision machinery. **Not built:** there is no `label` column in the as-built schema (§2); it is a possible later addition, not a reserved column.
- **`tags` — document-level classification (as-built, migration 0012).** A document carries an optional set of tags. Like `slug`, tags are **identity/classification, not content**: they live on `documents`, not `versions`, so they **survive content updates and restore** (a `PUT`/`edit`/restore that omits tags leaves them untouched, with no version bump) and only change when set explicitly. They are agent-settable (the `tags` field on the write tools) and operator-settable (a no-version-bump `POST /admin/documents/<public_id>/tags`); there is no agent-reachable tag-*only* endpoint in v1. Tags drive the `list_documents`/search **tag filter** (AND semantics over `documents.tags`) but are **not** full-text-indexed (§2 — they were removed from `documents_fts` in 0012).
- **`status` — lifecycle classification (as-built, migration 0014).** The third document-level classification field beside `slug` and `tags`: `active` | `deprecated` (| reserved `archived`), with an optional `superseded_by` replacement pointer. Same shape as its siblings — document-level, survives updates/restore, never bumps a version, **operator-only to change** in v1 (agent-reachable self-deprecation is a deferred follow-up the trust model already permits). A `deprecated` document keeps rendering and keeps ranking in search — the field rides every listing/read row so a reader can discount it — but is **excluded from context-pack fills by default** (§2's pack paragraph). `status` never gates access anywhere; it is a trust signal, not a boundary.

**Slug claim and addressing semantics:**
- **Claiming a name is semi-permanent.** A slug is set via the optional slug field/header on a write. `POST /d` (create) with a slug held by a **live** document → **`409 slug_taken`**; with a slug that was *ever* claimed and has since been retired (revoked/renamed/released) → **`409 slug_retired`**. Either way the agent picks a different name and retries. A slug, once used, is **reserved forever** — there is no path that returns a used name to the pool. (As-built: live slugs live on `documents.slug` under the partial unique index; retired slugs move to a `slug_tombstones` table, and a claim is checked against both sets. This reverses the original v1 "released on revoke" behavior — see issue #6.) The agent-facing rule: **don't mint slugs frivolously, and to change what a name serves, update *that* document rather than revoking and re-publishing under the same name.**
- **Reads address by `public_id` *or* slug, interchangeably.** `GET /d/<public_id>` and `GET /s/<slug>` serve the same document with the same content negotiation (§5), and the MCP `read_document` tool accepts either identifier (echoing back the resolved `public_id`). This read symmetry is the slug↔`public_id` "unification." The id-only mutation/inspection routes (`PUT /d/<public_id>`, `/source`, `/links`) are bridged for slug-holders by the discovery surface: **`GET /d?slug=<slug>` resolves a slug to its `public_id`** (returning the 0-or-1 listing row), so a headless agent that knows only the name can still address them. That listing+search surface — `GET /d` and `GET /d/search` — is reachable with an **agent key OR the operator token** (the HTTP twins of the MCP `list_documents`/`search_documents` tools; the operator-only `/admin/documents` forms are byte-identical in shape), consistent with the single-tenant whole-fleet trust model: an agent key already reads every document by id, so enumeration discloses nothing it couldn't already obtain.
- **Writes address by `public_id` only.** `PUT /d/<public_id>` revises an existing document; there is **no** create-or-update-*by-slug* verb (no `PUT /s/<slug>`). On a write the slug is a *settable attribute*, never a write-addressing key.

So the mental model is: **`POST` to create (optionally claiming a slug); `PUT /d/<public_id>` to revise; read by slug or `public_id`.**

**Forward links just work — no pending-link machinery.** Because internal links are authored as `/s/<slug>` URLs resolved at click/read time by the public `/s/<slug>` route (which serves the target in place — §4), a link's target does **not** need to exist when the link is written. A link to an unclaimed slug 404s until that slug is published, then resolves; a link to a revoked slug 410s (the slug is retired, so the link never silently re-points to a *different* document). The platform tracks no `pending_links` table and does no serve-time patching or backfill — the `/s/<slug>` lookup *is* the late binding. This is exactly what lets a cross-linked set, cycles included (A↔B), be authored in any order or all at once. The details are in §4.

**The link graph — backlinks as a first-class read (as-built, migration 0016, issue #40).** The *serving* of links stays pure late binding (above), but the system now also **records** each document's on-platform link targets at write time: every publish/update extracts the `/d/<public_id>` and `/s/<slug>` hrefs from the sanitized render into a `document_links` table, in the same atomic batch as the search-index row (revoke deletes the doc's rows). Targets are stored as the **raw addressed names** — the table inherits the late-binding philosophy rather than fighting it — and are resolved at *read* time into both directions of the wiki question:
- **Backlinks** ("what links here"): live documents whose current version addresses this one by `public_id` or live slug — a credentialed read (agent or operator, **never anonymous**: a backlink row describes *another* document, including private ones, so it belongs to the whole-fleet listing surface, which has always been credentialed). Surfaced to agents via `read_document` (`include_links`) / `GET /d/<public_id>/links`, and to the operator on the manage page.
- **Outbound link health**: each stored target classified as `live` / `redirected` (a renamed slug's loud forward — consistent with §4's never-auto-follow rule, a redirected inbound link is deliberately *not* counted as a backlink) / `retired` / `revoked` / `missing` — the broken-link report that renames and revokes (which the slug-retirement model makes loud at *serve* time) now also make **queryable** at curation time. Orphan detection (live docs nothing links to) is the operator's whole-fleet companion view.
The graph is a **read model over authored content, not a new authority**: it gates nothing, no surface auto-follows anything because of it, and deleting it would lose no canonical state (it is rebuildable from the stored renders via an operator backfill sweep — which is also how pre-graph documents get their rows).

## 4. The capability-URL security model (this is THE security boundary)

Because there is no reader login, **for a `public` document possession of the `public_id` URL is authorization** — there is no second check. This concentrates the entire read-security of the *public* surface into one mechanism, so that mechanism is treated as first-class, not a detail. (The exception is a `private` document — the visibility feature below: there the anonymous capability URL alone is *not* sufficient; the bytes serve only to the operator/fleet and the URL 404s to the open web. Possession-is-authorization remains exactly true for the public documents it was always about.)

- **High entropy, non-enumerable.** `public_id` is ≥128 bits of CSPRNG output. No sequential ids, no `/d/1`, no low-entropy tokens. It cannot be brute-forced or walked. (Contrast: in the platform spec the unguessable origin was *defense-in-depth* behind `can_access`; **here it is the sole boundary**, so the entropy requirement is load-bearing, not belt-and-suspenders.)
- **The slug is a deliberate, weaker capability — and that asymmetry is the design, not a leak.** A slugged document is reachable at the public `https://<host>/s/<slug>` endpoint, which serves the document in place with no auth (content-negotiated exactly like the `public_id` URL — §5; the slug stays in the address bar, no redirect). So slugged and unslugged documents are **not** equally private: a slug trades part of the `public_id`'s ≥128-bit unguessability for a short, typeable, linkable name. A memorable slug like `north-island-report` *is* guessable, so a slugged document is discoverable by anyone who guesses or is told the name. **This is acceptable precisely because the slug is opt-in and rare.** The default — and the posture for the overwhelming majority of documents — is *no slug*: a document shared only by its `public_id` URL keeps the full capability boundary. Give a document a slug only when it is *meant* to be found by name or to serve as a link target (next bullet); for everything else the unguessable `public_id` alone is the stronger, correct posture. The lever against an over-shared slug is the same kill switch as any document — revoke retires the slug and 410s the `/s/` lookup (§4 revoke bullet).
- **Internal links are authored as `/s/<slug>` URLs and resolved at click/read time by the public `/s/<slug>` route (served in place).** An agent writes `<a href="/s/section-1">` — a normal same-origin relative link the sanitizer passes through unchanged. There is **no write-time link rewriting, no `doc:` scheme, no stored `public_id` substitution**: the authored slug URL is exactly what gets stored and served, and the `/s/<slug>` endpoint (above) resolves it to the current target on every click or agent read. The cost is paid per *read* (a cheap serve-time lookup), not per *write*, and the served bytes carry the slug, not a resolved `public_id`. This is what makes cross-references and cycles trivial — because `/s/<slug>` is stable and predictable, an agent can write A linking to B and B linking to A in either order, or publish both at once, without knowing either `public_id` in advance. A link to a slug that isn't claimed yet simply 404s until the target is published (and 410s if the target is later revoked — the slug is retired, never re-bound to a different document); there is **no** forward-reference bookkeeping, **no** drop-and-advise pass, and **no** leaf-first ordering constraint. Net: **agents link by memorable slug; the slug URL is public and stable; resolution is a serve-time lookup, not a write-time rewrite.**
- **Per-document visibility — `public` vs `private` (as-built, migration 0011).** This reverses the original "born-live" model. A document carries a `visibility` flag that gates **only the anonymous/public browser surface**: a `private` document returns the same opaque `404` as a missing/revoked one to an unauthenticated caller — *masking discovery, never a `401`, never an existence oracle* — while a `public` one is served at its `public_id`/`slug` URLs exactly as before. The gate does **not** touch authenticated principals: in the single-tenant whole-fleet trust model the operator and any active agent key already read the entire corpus, so they read every document regardless of visibility. Visibility is purely "is this exposed to the open web," not a new confidentiality boundary among insiders.
  - **Born-private by default; operator-publish.** New documents are born at `DEFAULT_DOCUMENT_VISIBILITY` — a deploy-time `[var]`, default `private` — bound explicitly on each write (the column's own `DEFAULT 'public'` only backfills pre-0011 rows, so already-shared documents keep serving). **Only the operator changes visibility** — via the JSON `POST /admin/documents/<public_id>/visibility`, or the in-browser **Manage** page reached from the shell topbar's *Manage…* item (`POST /d/<public_id>/visibility`, session-cookie + CSRF; the signed-in toolbar also surfaces the current `Public`/`Private` state at a glance). An agent cannot, and to an agent "published is published" — visibility is not surfaced or explained on any agent-facing contract. This operator-publish rule is exactly what makes the feature the §8 compromised-agent-exfiltration mitigation: a compromised agent cannot unilaterally push its output onto the open web.
  - **Slug interaction.** Visibility is orthogonal to the slug. A private document **keeps** its slug claim (it is NOT retired — that is revoke-only); `/s/<slug>` simply returns `404` to anonymous and serves to the operator/agents, and making the document public again relights the same name. A private slugged document is *masked*, not removed.
  - **The access decision is a named function — `can_access`.** The read decision lives in one pure primitive, `canRead(principal, doc)` in `src/access.ts`: the deliberately-collapsed single-tenant descendant of the platform spec's `can_access(principal, document, level)` (PLATFORM §4.1) — same top-down, first-match shape, with shares / projects / soft-delete / owner-suspension all stripped away. What remains is `revoked → false; operator → true; agent → true; anonymous → visibility == 'public'`. Write authority and visibility-*change* authority are deliberately kept OUT of it (they stay separate operator checks), mirroring how the platform keeps admin authz out of `can_access`. This is the seam the SOLO spec previously reserved here ("a real draft/private state — born-private, explicit operator publish"): it is now **built**, and `can_access` is where future sharing/ACLs would extend rather than re-scattering the decision.
- **Revoke = the kill switch.** Setting `revoked_at` makes the `public_id` return `404` immediately (subject to the same edge-cache/TTL caveat as any served-bytes change — see §5), permanently. A revoked `public_id` is never reissued. Its **slug is retired, not released** — the `/s/<slug>` lookup flips to **`410 Gone`** and the name is reserved forever; no future document can claim it (a claim attempt → `409 slug_retired`). Because internal links are `/s/<slug>` URLs resolved at click time (above), any inbound links to a revoked document's slug 410 — they go dead, but they **never silently re-point** to a different document, because the name is never reissued. This closes the original v1 "late re-binding" footgun (a revoked-then-reclaimed slug serving unrelated content under the same URL): the as-built behavior was reversed per issue #6 precisely because slug reuse is surprising — a shared/bookmarked/cross-linked name should keep meaning what it meant or break loudly, never quietly swap. (The operator-controlled *redirect* case — repointing a retired name to a new destination via an explicit, visible redirect — is a separate, deliberate feature, distinct from silent reuse; see the deferred seam below.) This is the "nuke a live link" capability.
  - *Slug redirects (as-built).* The one legitimate "change a slug's destination" case (a branding rename, or consolidating two documents) is met not by reuse but by a **loud redirect** on the tombstone (`slug_tombstones.redirect_to`, migration 0010 — the target's `public_id`). Two ways one lands: (a) **rename auto-forwards** — when a live document's slug is changed A→B (by an agent's `update_document`/`PUT` write, OR by the operator's no-version-bump slug change on the **Manage** page / `POST /admin/documents/<public_id>/slug`), A's tombstone is created pointing at that document's own `public_id`, so `/s/A` keeps working at the new name (same-document, so it needs no separate redirect action — the operator slug change reuses the exact same retire-and-forward path as the agent write); (b) **operator action** — `POST /admin/slugs/:slug/redirect {target_public_id}` points a retired slug at a *different* live document (cross-document; operator-only). Resolution is **loud, never silent**: `/s/<slug>` gives a browser a click-through interstitial and an agent `409 slug_redirected`, which follows only when it opts in (`?follow_redirects=true`, or `follow_redirects:true` on MCP `read_document`, which then stamps `redirected_from`). The target is stored as a `public_id`, so resolution is single-hop and loop-free; a revoked target dangles and falls back to 410. Two operator levers complete it: `DELETE /admin/slugs/:slug/redirect` drops the redirect (back to a plain 410 tombstone, still retired), and `DELETE /admin/slugs/:slug` is the **escape hatch** — force-release a tombstone entirely so the name is claimable again (the only path that un-retires a slug; for a revoke-by-mistake). Operator endpoints are `requireOperator`-gated (CSRF on cookie auth).
- **`Referer` leakage is closed.** External links in served documents carry `rel="noopener noreferrer"` (carried from platform §6.5). The `noreferrer` does double duty here: besides opener-protection, it stops a document's own secret `public_id` URL from leaking to an external site via the `Referer` header when a reader clicks an outbound link. Without it, the capability secret would leak on every outbound click.

## 5. Sanitize and serve (carried from platform §6/§7, slimmed)

Anything reachable at a public URL is published to the open web, so the sanitize-then-sandbox pipeline is retained. It is simpler than the platform's only in that there is no adversarial-author threat model and no scripted tier — but the mechanics are the same and should be lifted from platform §6/§7 directly.

**Write path (validate → sanitize → store):**
- **Validate (loud, agent-facing ergonomics):** check against a named blocklist (`<script>`, `on*` handlers, `javascript:`/`data:` hrefs, `<iframe>`/`<object>`/`<embed>`/`<form>`, `<img>`/SVG `<image>`, SVG `<foreignObject>`/`<use>`, `<meta http-equiv=refresh>`, `<base>`, MathML). On a hit, reject with a structured error naming the violation, so the agent can self-correct. This is feedback, **not** the security boundary.
- **Sanitize (the security boundary — allowlist):** run an allowlist-based sanitizer (the build uses **Ammonia**, compiled to WASM, as a pinned profile) — only allowed tags/attributes/SVG-subset survive; everything else is dropped. The allowlist, not the blocklist, is what makes it safe (safe against *unknown* constructs). **The sanitizer's serialized output is the bytes stored at `r2_key`, unchanged** — no markup mutation after the sanitizer's final serialization (the platform §6.2 ordering rule; re-serialization after the last security decision manufactures mutation-XSS). Forced link attributes (`rel="noopener noreferrer"`, `target="_blank"`) are applied *inside* the sanitizer profile (a hook), never as a post-pass.
- **Document-level CSS via `<style>` is allowed (as-built).** The profile passes `<style>` **blocks** through (not just inline `style="…"` attributes), so a document can carry a class-based stylesheet / design system — selectors, `:hover`/`:focus`, `::before`/`::after`, `@media`, `@keyframes`, `prefers-color-scheme`, `@font-face` with `data:` fonts. The sanitizer does **not** parse the CSS; as with inline `style`, the defense is the render-time CSP + iframe sandbox (the serve path below), not the sanitizer — and external CSS *egress* (`@import url(https://…)`, external `url(…)`/`@font-face src`) is closed at the CSP, so such resources survive sanitize but the browser refuses to load them (a `will_not_render[]` advisory flags this on write). Still stripped at sanitize: `<script>`, external `<link rel=stylesheet>`, `<iframe>`, `<meta>` — the blocklist above is unchanged.
- **SVG** is admitted as an explicit diagram/chart **allowlist**, not "SVG minus the bad parts"; `<use>` and `<foreignObject>` dropped (platform §6.3).
- **Store** the sanitized bytes **plus the pre-sanitized source** — **two** R2 objects per version: the served form **H** at `r2_key` and the submitted **source S** at the sibling key `<r2_key>.src` (Markdown for a Markdown doc, the original HTML for an HTML doc), alongside the `sanitizer_v` stamp (so "which docs used a now-vulnerable profile" stays a `WHERE` clause). **Convert-and-discard has been reversed** (as-built): source retention is the model now, because retaining S is what lets `edit_document` patch the *source* (a Markdown doc edits its Markdown, re-renders, stays Markdown and keeps its reader theme — the bug that motivated the reversal; §6) and what unblocks lazy re-sanitization (below). The source counts toward the storage cap (§7) — single-tenant, so every stored byte is real budget. **Only H ever reaches a browser; S is never served on the public render path.** Source-read surfaces (the agent-scoped `read_document representation:"source"` MCP knob, and the credential-gated HTTP `GET /d/<public_id>/source`) are **authenticated, never public** — the HTTP route admits **any non-anonymous principal: the operator OR any agent key (operator ≥ agent)**, never operator-only (in this whole-fleet single-tenant model any active agent key already reads and overwrites every document, so a source-read discloses no authority the caller lacks) and never agent-only (the operator is the apex principal and reads everything an agent can — gating these on the agent key alone, as an earlier build did, wrongly ranked the operator *below* an agent). Each source-read response carries `unsanitized: true` plus the `stripped[]` / `will_not_render[]` advisories, so a consuming agent is told the bytes are pre-sanitization and where the live render diverges. (Legacy/un-backfilled versions retain no source and fail loud — a structured `source_unavailable`, never a silent fall back to H.)
- **Resource bounds:** per-document size cap; a cheap pre-parse depth/node/attribute gate before the heavy parser; sanitize in a **killable worker** with a global concurrency bound (a wall-clock timeout cannot preempt synchronous parse work on the Node event loop — platform §6.6). These survive because a compromised/injected agent can still submit a pathological document.

**Serve path (two readers, two representations):**
- **Browser read (public, via `public_id` or `slug`):** the request returns a tiny HTML **shell** that embeds a **sandboxed `<iframe>`**; the iframe loads the document bytes from a sibling `/raw` URL that carries the strict CSP — `connect-src 'none'` (no exfiltration), `frame-ancestors 'self'`, `base-uri 'none'`, `form-action 'none'`, `img/font/style-src` self/inline/`data:` with no external origins, `sandbox` with **no** `allow-scripts` and **no** `allow-same-origin` (static, no script, no ambient origin), `Cache-Control: no-store`. Two URLs are deliberate: `frame-ancestors` is header-only, so the bytes must come from an HTTP response, not `srcdoc`. **No per-document origin.** Documents are **not** isolated from one another (they don't need to be — my agents aren't adversarial to each other). What they *do* need is isolation **from the service's own origin** — its cookies, its session, its DOM — which matters most if a scripted tier is ever allowed. The intended hardening for that is to serve all document bytes from a **single separate content domain** (distinct registrable domain from the app/service origin), not a per-document origin. *(Current build serves same-origin under the sandbox + strict CSP, which is the load-bearing wall; the separate content domain is the planned next layer.)* A revoked `public_id` stops resolving and returns `404`; a revoked document's (now retired) `slug` returns **`410 Gone`** (§4). **Visibility-gated (§4):** this anonymous browser read serves only `public` documents — a `private` one returns the same opaque `404` to an unauthenticated caller, and serves only to the operator (whose `awh_session` cookie reaches the same-origin `/raw` subresource — `SameSite=Lax` strips only cross-*site* requests) or to an agent key. The gate lives in one place, the `/raw` byte path, so the shell and the homepage (which both embed `/raw`) are covered by it. Because a valid `public_id`/`slug` now `404`s to a logged-out operator (it can't tell its own session is missing from the doc being absent), the **browser `404` is an HTML page carrying an operator "Sign in" link** that round-trips back to the requested URL (`/login?next=…`) — so the operator signs in and lands back on the document. It is rendered **identically** for every browser-navigation `404` (absent, revoked, malformed, or private-to-anonymous), so it adds no existence oracle; authenticated/agent callers still receive the plain `Not Found` body.
- **Agent read (authenticated, via API):** an agent presenting its key on `GET /d/<public_id>` (or `GET /s/<slug>`) — or the operator presenting its token, since operator ≥ agent — receives the raw **sanitized HTML** — the same stored bytes the iframe loads — with internal links carried verbatim as their authored `/s/<slug>` URLs (§4), so an agent sees the same stable links a human would click. A **Markdown** projection (for ingest-as-context) is a *separate* endpoint — `GET /d/<public_id>/text` (and `GET /s/<slug>/text`), credential-gated (operator or agent, never anonymous) — and is **built, not reserved**: it is **derived on read** from the sanitized HTML (no second stored artifact, so no drift). The split is by **path** (`/` vs `/text`), not by `Accept` header. The MCP surface mirrors this with one `read_document` tool taking a `format: "html" | "markdown"` knob (default `markdown`).

**Re-sanitization is now unblocked, but still deferred — not built.** When the sanitizer profile hardens, **new writes self-heal** (they use the current `sanitizer_v`). Re-healing *already-stored* versions requires the pre-sanitized original, and the build now retains it (source S per version, above) — so the foreclosure is gone: reversing the no-original-retention decision was the prerequisite, and that reversal has **landed**. What remains deferred is the *re-heal mechanism itself* — **[RESEARCH NEEDED]**. The leading (operator-preferred) proposal is **lazy re-heal on first read after a profile bump**: on read, if a version's stored `sanitizer_v` is behind the current profile, re-render from the retained source, re-sanitize, re-store H, and bump the stamp — so the first reader pays the heal and everyone after gets the healed bytes, with no batch job. The open questions are where the heal fires given the streaming serve path (a re-sanitize needs buffer + WASM, which the stream can't do inline), thundering-herd idempotency on the first read after a bump, and whether a heal mints a new `versions` row or rewrites H in place under the same version with a new `sanitizer_v`. The `sanitizer_v` stamp still records which profile produced each stored blob, so "which docs used a now-vulnerable profile" remains a `WHERE` clause — and *acting* on that answer is now a buildable re-heal rather than a foreclosed one. **No link interaction either way:** internal links are stored verbatim as `/s/<slug>` URLs resolved by the public `/s/<slug>` route on every read (§4), so there is nothing to re-resolve — and no write-time link resolution ever existed to repeat.

## 6. Write semantics

- **Create:** `POST /d` with the body (`Content-Type: text/html` or `text/markdown`; Markdown is parsed to HTML before sanitizing) and an optional `slug` (plus optional `title`/`description`/`tags`). Mints `id` + `public_id` + version 1. Fails **`409 slug_taken`** if the slug is held by a live document. The create is one transaction: insert document (null `current_ver`) → insert version 1 → update the pointer (the circular-FK order from platform §3.2a). Returns the `public_id` (the shareable URL) and `version: 1`. *(No `model` field is accepted or stored.)* The **operator** authors through the parallel JSON door `POST /admin/documents` (and `PUT /admin/documents/<public_id>` to update) — same core write path, JSON body instead of raw bytes, and the operator may set birth visibility atomically. On the operator update door `If-Match` is **optional** (last-write-wins when omitted), the deliberate app-friendly divergence from `PUT /d`'s required `If-Match`.
- **Authorship attribution (who wrote it).** A write resolves to an **author principal** — `{operator}` or `{agent, id}` — *not* a bare agent id (the operator is never smuggled through as a fake agent). It is persisted in D1: `documents.created_by` + `created_by_kind` record the immutable **creator**, and each `versions` row records its own **writer** (`author_kind` + the agent id, null for the operator) — so a document's full per-version author list is queryable (e.g. an agent-created doc the operator later edits reads creator = agent, latest author = operator). The writing identity is also echoed into R2 `customMetadata` as before, but D1 is now the authoritative, queryable record.
- **Update:** `PUT /d/<public_id>` writes a new version (addressed by `public_id` only — there is no `PUT /s/<slug>`; §3). `If-Match` is **required** (omit it → `428 Precondition Required`); send `"v<version_no>"` for optimistic concurrency (stale → `412`) or `*` to force. The canonical strong tag is `"v<version_no>"`, but the lenient `v<version_no>`/`<version_no>` forms are accepted too, so the integer `version` a read returns can be sent as-is (matching the `If-None-Match` tolerances on the render path). The ETag is `version_no` (monotonic int — "has this advanced past what I based my write on?").
- **No idempotency keys.** `Idempotency-Key` is **descoped** in this single-tenant build — `If-Match`/ETag optimistic concurrency is the only write-safety mechanism. (An optional `X-Content-SHA256` byte-exact integrity check on the *received body* exists separately, for the byte-exact curl publish path.)
- **Source currency token (`source_sha256`, migration 0015).** Every version records the SHA-256 of its retained source S — the *same* bytes the `X-Content-SHA256` check hashes, so for a byte-exact publish it equals the publisher's `sha256sum file`. It is surfaced (write response; `current_source_sha256` on a body-free listing row; a `representation:"source"` read) so an agent holding a local copy can confirm it's still the current source — comparing hashes for pennies — and **skip re-downloading the source before an `edit_document`** (the read the edit contract otherwise prescribes). The match is exact only for a well-formed-UTF-8 file published as-is; a mismatch is a safe false negative (just re-read). This is ergonomics, not a trust boundary: it gates nothing.
- **No reader-facing write path.** Only the operator (token/session) and agents (key) write.
- **Delete/revoke:** operator-only. `revoke` flips `revoked_at` (the kill switch, §4), **retires the slug** (the name is reserved forever — `/s/<slug>` 410s, never reused — §4), drops the search index row, and purges the R2 bytes. There is no document-level soft-delete/undelete/hold lifecycle — those were multi-tenant/preservation concerns now dropped; revoke is permanent.
- **Version history + restore (as-built).** Because `versions` is append-only and each version's R2 bytes are retained until revoke purges them, **prior versions are readable, not just an audit trail.** The operator can view any historical version in the browser (`GET /d/<public_id>/v/<n>` shell + `/v/<n>/raw` bytes, operator-gated — an operator surface orthogonal to public/private visibility) and **restore** one from the document-management page. Agents read history over MCP (`read_document` with `version` to read a specific one, `include_history` for the newest-first manifest) — **read-only; agents cannot restore in v1** (they can surface "v5 was the good one" and propose it). **Restore is restore-as-a-new-version, never a pointer rewind:** restoring version N re-publishes N's content (and title/description) as a *new* version N+1, preserving the monotonic `version_no`/ETag invariant (§6) — a backward `current_ver` move would collide with the next write. The document's current slug **and tags are kept**, not rolled back: both are document-level (§3), not part of a version's content. This is distinct from the (absent) document-undelete above: it recovers a prior *state of a live document*, not a revoked one — revoke still purges the bytes, leaving nothing to restore.

## 7. Storage cap

A single configurable **storage cap**, value chosen at build time against the selected provider's free tier(s). Enforced at write: reject a write that would exceed the cap. Accounting is a single counter (or a `SUM(size_bytes)` — at one operator the scale doesn't warrant the platform's denormalized two-counter machinery; one denormalized integer updated in the write transaction is more than enough). No hold-aware or operator-invisible accounting — there are no holds and one operator.

## 8. The residual publisher posture (the honest, small legal note)

Descoping to single-operator removes the *host-of-third-party-content* duties (no DMCA safe-harbor apparatus needed — I am not hosting others' content; no report-intake/T&S surface — there is no adversarial author population). But publishing my own agent's output to public URLs is not zero-duty, and the honest residual is small and worth stating:

- **I am the publisher of record.** Content at a public `public_id` URL is published-by-me to the open web. That is the "personal website / I publish my own output" posture — far lighter than UGC hosting, but it means ordinary content-of-the-open-web norms apply to me as publisher.
- **CSAM law is not gated on tenancy.** It applies regardless of how few operators there are. At this scope there is no automated detection and no adversarial population feeding the surface — the realistic exposure is an agent being prompt-injected or malfunctioning into generating prohibited content, which I then would be publishing. The mitigation that exists is the **kill switch** (§4 revoke, instant) plus operator awareness. If this ever grows toward independent agents handling sensitive data or a real public population, the platform spec's §6.8/§13 preservation-and-reporting machinery is the blueprint to pull back in — *do not* grow the public surface without revisiting it.
- **Compromised-agent exfiltration** (operator-raised): a compromised/injected agent could write secrets to a document and leak the `public_id` to exfiltrate via the open web. Sized as minimal — a compromised agent with outbound network access has far easier exfil paths than this tool, and I am not yet running independent agents holding sensitive context. **The mitigation is now built (§4): documents are born private by default and only the OPERATOR can make one public**, so a compromised agent cannot unilaterally expose its output — it can write a private document, but those bytes never reach an anonymous reader without an operator action, and a private doc's `public_id` 404s to anyone but the operator/fleet, so the leaked capability URL is no longer self-sufficient for exfil. (The driver was product, not this threat — born-live is wrong for a knowledgebase — but operator-publish closes the threat as a hygiene side effect.)
- **Source retention (§5) leaves this analysis unmoved.** The retained pre-sanitized source **S** is **never published** — only the sanitized **H** ever reaches a public URL, so the publisher-of-record and CSAM postures above (which are about what is *published*, i.e. H) are unchanged. The new raw-source channel is **agent-key-gated, never public** and is strictly *less* reachable than H (which already has a public render URL), so it adds no exfiltration path the open-web `public_id` doesn't already present. The one genuinely new exposure is narrow: an agent that requests source ingests *markup-form* bytes the sanitizer would have stripped from the render — an XSS/markup-shape payload an HTML-naive context window might misread as instructions. That increment is covered by the `unsanitized: true` provenance marker + advisories on every source-read and the agent-key gate; it is not a new natural-language-injection surface (adversarial prose already survives into H and the Markdown projection today). Holding unsanitized bytes at rest creates **no** new at-rest-scanning obligation here — single-tenant, operator-owned R2 holding the operator's own agents' bytes, with no third-party UGC ingestion and no execution of stored bytes (a decided non-action, not an omission).

This section is deliberately a paragraph, not a subsystem. If any of its "if this grows" conditions start to become true, that is the signal to consult the platform spec (and, per its own §10.1, counsel) before expanding the public surface.

## 9. What to lift directly from the platform spec when building

To avoid re-deriving solved problems, these parts of [`agent-knowledge-host-spec-PLATFORM-v2.md`](agent-knowledge-host-spec-PLATFORM-v2.md) transfer almost verbatim (with multi-tenant references stripped):
- **§6.2 validate-then-sanitize** (the two-stage gate, the allowlist-is-the-boundary authority statement, the no-post-sanitize-mutation ordering rule, the `stripped[]` advisory).
- **§6.3/§6.3a** sanitizer hardening + the **pinned, versioned profile** discipline (SVG allowlist, MathML/`<use>`/`<foreignObject>` handling) — though the SOLO build realizes that profile as **Ammonia compiled to WASM** (no DOMPurify, no jsdom shim); the `sanitizer_v` stamp (§2) is the pinned-version handle.
- **§6.6** resource bounds (pre-parse SAX gate, killable worker + global concurrency bound).
- **§7.2–§7.4** the iframe sandbox, CSP-as-network-boundary, and serve-time header attachment via the edge worker — **but not §7.1's per-document origins.** SOLO does not isolate documents from each other; the only origin isolation is document-bytes-from-the-service, via a single separate content domain (§5).
- **Nothing for slug → `public_id` link resolution.** SOLO deliberately does **not** lift the platform's §3.7 `doc:` resolver in any form (neither serve-time nor write-time). Internal links are authored as `/s/<slug>` URLs and resolved by the stateless public `/s/<slug>` route, which serves the target in place on each read (§3–§4) — so the `doc:` grammar, the write-time rewrite, and the drop-and-advise machinery are *not* built. The `/s/<slug>` route is the whole resolver.
- **§11.3** write semantics (ETag = `version_no`, `If-Match`/`412`), **minus** the `Idempotency-Key` / new-attempt-new-key rule — idempotency keys are descoped at single-tenant scale (§6).

Everything else in the platform spec — access control, projects, sharing, admin/T&S, holds, quotas — is intentionally *not* carried.
