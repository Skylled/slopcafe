# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> **Operator-instance specifics — forkers, read this.** This repo is the source for Kyle's `slopcafe.com` deployment. A few obligations below are tied to *that specific instance*, not to the code: the "re-publish the live Slopcafe copy in the same commit" rules and the hard-coded `public_id`/`slug` values (e.g. `slopcafe-http-api` → `0EtsEq6cnCeuOhBKO6ICzA`, the SOLO/PLATFORM spec ids, and the `slopcafe-*` design-note slugs) point at documents in Kyle's corpus on `slopcafe.com`. If you fork this and run your own deployment, treat those as *examples* — substitute your own slugs/ids or skip the mirror step entirely; nothing in the code depends on them. Everything else here (the architecture, the conventions, and the source-↔-`docs/` sync rules that don't mention Slopcafe) is instance-independent.

## Commands

Build / deploy / dev:
```sh
npm run dev            # wrangler dev — uses .dev.vars + local D1/R2
npm run typecheck      # tsc --noEmit
npm run build:wasm     # rebuild the Rust→WASM sanitizer (auto-run via predeploy)
npm run deploy         # rebuilds WASM then `wrangler deploy`
```

Tests:
```sh
npm test                  # sanitizer corpus + advisories + metadata + pagination + search + edit + integrity + auth + session + access + conditional + vector + pack + contract + openapi
npm run test:sanitizer    # cargo test in /sanitizer (host target, not wasm)
npm run test:advisories   # node --experimental-strip-types test/advisories.test.mjs
npm run test:metadata     # node --experimental-strip-types test/metadata.test.mjs
npm run test:pagination   # cross-file ts import → uses test/register-ts-resolver.mjs
npm run test:search       # FTS5 query tokenizer (buildFtsMatchQuery) — pure function tests
npm run test:edit         # applyEdits find/replace logic (edit_document) — pure function tests
npm run test:pack         # context-pack pure logic (budget fill, manifest parser, link extractor)

# Run a single sanitizer test (Rust name match):
(cd sanitizer && PATH="$HOME/.cargo/bin:$PATH" cargo test rejects_script_tag)
```

Database (D1):
```sh
npm run db:migrate:local            # apply migrations to local D1
npm run db:migrate:remote           # apply migrations to remote D1
npm run db:console:local  "SELECT * FROM agents"
npm run db:console:remote "SELECT * FROM agents"
```

`predeploy` injects `$HOME/.cargo/bin` into PATH so `wasm-pack` resolves from a fresh shell. Rust toolchain (`rustup` + `wasm32-unknown-unknown` target + `wasm-pack`) is only required for `build:wasm`.

## Git workflow

When the operator says "commit your work" (or similar) without specifying a branch, **commit straight to `main`** — do not branch first. The operator deliberately delegates commits to the agent so the work shows up credited in the log; keep using the standard `Co-Authored-By` trailer. Still only commit/push when explicitly asked, and push only on explicit request.

## Architecture

**Single Worker, one origin.** `src/index.ts` exports an `innerHandler` doing exact-match route dispatch, and the default export is that handler wrapped in `@cloudflare/workers-oauth-provider` (see `src/oauth.ts`). The OAuth wrap intercepts `/mcp` (its `apiRoute`), validates the token, populates `ctx.props.agentId`, and dispatches to the same inner handler as `apiHandler`. Everything else (including `/authorize` and the `awh_`-bearer path on `/mcp`) goes to `defaultHandler`. **The inner handler is registered as both `apiHandler` and `defaultHandler`** — the only difference between the two paths is whether `ctx.props.agentId` is populated. The route table matches on `method === "GET"` (etc.) literally, so **`HEAD` is normalized to `GET` by the `withHeadSupport` wrapper** sitting *between* the OAuth wrap and `innerHandler` (`wrapWithOAuth(withHeadSupport(innerHandler))`): it re-issues the request as a `GET` (carrying the original headers, so the visibility/auth/`If-None-Match` gates are unchanged) and strips the response body, so `HEAD /d/:id/raw` answers with the `GET`'s `text/html`/`ETag`/CSP instead of falling through to the JSON `404` (GitHub issue #36). A private/missing doc still `HEAD`s as the same opaque `404`.

**Two auth doors converge on one identity (`AwhProps` in `src/mcp-auth.ts`):**
- **Door A (OAuth 2.1 + PKCE)** — one pre-registered client per agent. Bound mint via `POST /admin/agents/:id/oauth-clients`; or **unbound** mint via `POST /admin/oauth-clients` (KV client record, no `oauth_clients` row) with the agent chosen at consent. Clients can also **self-register via DCR** (`POST /register`, RFC 7591 — served by the provider library when `ENABLE_DCR` in `src/oauth.ts` is on; **public (PKCE-only) clients allowed** — `DISALLOW_PUBLIC_CLIENT_REGISTRATION = false`, required for native CLIs like Claude Code's `claude mcp add` which can't hold a secret; 90-day client TTL): a DCR client arrives unbound and rides the same bind-or-mint-at-consent path, which is what enables the paste-the-MCP-URL connect flow with no operator-minted client_id (docs/design/dcr-design.md, BUILT). Consent UI lives in `src/authorize.ts`; the operator approves with `OPERATOR_TOKEN` (or a browser session). Used by Cowork / claude.ai web / ChatGPT. `agentId` comes from the grant's `encryptedProps` (via `ctx.props`). **`/authorize` is operator-session-aware and offers three inline repairs** (all operator-gated, none disclose client state to non-operators): (1) **TOFU callback approval** — register an unregistered but allowlisted-host `redirect_uri` via `updateClient`; the allowlist is `APPROVABLE_CALLBACK_HOSTS` in `src/admin-oauth.ts` (the narrow **vendor-https TOFU** trust list — `claude.ai`/`claude.com`/`chatgpt.com`; `validateCallbackUri` in `session.ts` is the pure https-only gate). **This allowlist is NO LONGER coupled to the consent-page CSP `form-action`** (it used to be — that coupling is exactly what blocked native clients). `form-action` is now the standalone `CONSENT_FORM_ACTION_SOURCES` in `src/authorize.ts`, deliberately broad enough to cover *every legitimate OAuth callback shape* — `'self'`, `https:` (any vendor, no per-host list), `http://localhost|127.0.0.1|[::1]:*` loopback (RFC 8252 native/CLI clients), and IDE custom schemes (`vscode:`/`cursor:`/…) — because `form-action` is **browser defense-in-depth, not the access gate**: the real gate is the library's per-client registered-`redirect_uri` match + S256 PKCE + operator consent, so the form can only 302 to a provider-validated target. Add a new custom scheme there (only) when onboarding a client that uses one. (2) **Bind-or-mint at consent** — an *unbound* client (no `oauth_clients` row; needs **no migration** — absence == unbound) gets a pick-existing-or-mint-new-agent card; "allow" writes the binding row, then `props.agentId` is **re-derived from the binding** (never trusted from a form) before `completeAuthorization`. (3) **Login-from-`/authorize`** — an unauthed requester gets a `/login?next=<this url>` link, keyed on the requester's own auth state so it discloses nothing. The POST auth ladder mirrors `handleRevokeForm` (pasted `operator_token` → synthetic Bearer/CSRF-exempt, else session cookie + `csrf_token` field) — it does **not** call `requireOperator` (that wants an `X-CSRF-Token` header a no-JS form can't send).
- **Door B (static `awh_` bearer)** — same key shape `POST /d` uses. Plumbed into the OAuth wrap as `resolveExternalToken` so a single configured `apiRoute` covers both transports. `agentId` comes from `authenticateAgent` (HMAC-under-pepper + `revoked_at` check on `agent_keys`).

**Operator browser session (`src/session.ts`) — a SECOND door onto the *operator* check, not a third agent door.** The two doors above authenticate *agents*; the operator is separate (`authenticateOperator` = constant-time compare vs `OPERATOR_TOKEN`). The session layer lets the operator log in once in a browser (`GET/POST /login`, `GET/POST /logout` in `src/login.ts`) instead of pasting the token on every action. It is HTTP/browser-only — **`/mcp` never consults these cookies** (the OAuth wrap owns it), and **no MCP tool/`/mcp` behavior changes**. It's a **stateless signed cookie**, no D1/KV/migration: `awh_session = base64url({v,iat,exp,csrf}) "." HMAC(payload, signingKey)`, where `signingKey = hmacSha256Hex("awh-session-signing/v"+EPOCH, OPERATOR_TOKEN)`. The signing key is derived from `OPERATOR_TOKEN` **on purpose** (not `HMAC_PEPPER`): the session authenticates the operator principal, so rotating the token must end every session — don't "fix" this to the pepper. Two invalidation levers, both of which change `signingKey`: bump `SESSION_EPOCH` (a `[var]`, the cheap "log everyone out" knob — defaults to `"1"` via `sessionEpoch(env)`) or rotate `OPERATOR_TOKEN`. `authenticateOperatorRequest` resolves **Bearer-first then cookie**; the shared async `requireOperator` (in `session.ts`, used by `admin.ts` / `admin-oauth.ts` / `DELETE /d/:id`) returns 401/403 and enforces CSRF (`403 csrf_failed`) only when authed **via cookie + unsafe method**. Bearer-authed requests are CSRF-exempt — so every programmatic/curl caller is unchanged.

**Operator console (`src/console.ts`, GitHub issue #22) — the server-rendered admin UI.** A no-JS HTML console under the **`/admin/console/*`** namespace (bare `GET /admin` 302-redirects to `/admin/console`), authed by the **operator browser session** (cookie + CSRF) above — a logged-out caller gets a sign-in card, not an existence oracle. It is a UI layer ONLY: every console POST routes through the **form-field CSRF ladder** (`authorizeOperatorForm`, NOT `requireOperator` — a no-JS `<form>` can't send the `X-CSRF-Token` header `requireOperator` wants), and every write reuses the same `*Core` worker the JSON `/admin/*` handlers call (`mintAgentCore` / `mintAgentKeyCore` / `revokeAgentCore` / `revokeKeyCore` / `listAgentsCore` / `listAgentKeysCore` in `admin.ts`; `createOAuthClientCore` / `createUnboundOAuthClientCore` / `deleteOAuthClientCore` / `listBoundClientsCore` in `admin-oauth.ts`). **The JSON `/admin/*` API is UNCHANGED** — the `*Core` refactor kept its wire bytes byte-identical. The console pages are the dashboard, agents (list + detail with key minting), documents (list with `?q=`/`tag`/`slug`/`cursor`/`limit` reusing the existing list/search core filters + a Public/Private **badge** only — no server-side visibility filter), and maintenance (vectors backfill). A freshly minted agent key or OAuth-client secret is shown in a **one-shot disclosure card** (rendered once, never logged — same discipline as the MCP minted key + the OAuth token). **POST-not-DELETE divergence:** several console routes (`POST /admin/console/agents/revoke`, `/admin/console/keys/revoke`, `/admin/console/oauth-clients/delete`) are POST twins of JSON `DELETE`s (`DELETE /admin/agents/:id`, `/admin/keys/:id`, `/admin/oauth-clients/:id`) because a no-JS HTML form can only GET/POST — the target id rides a form field (`agent_id` / `key_id` / `client_id`), not the path.

**Shared write path: `src/core.ts`.** Both HTTP (`POST /d`, `PUT /d/:id` in `src/index.ts`) and MCP (`src/mcp.ts` tools) call `publishDocumentCore` / `updateDocumentCore` / `editDocumentCore` / `readDocumentCore` / `readDocumentTextCore` / `revokeDocumentCore` / `listDocumentsCore` / `findDocumentBySlugCore` / `searchDocumentsCore`. **Sanitization runs exactly once, inside core**, regardless of door. If you add a new write surface, route it through core — never duplicate the sanitize → cap-check → R2 → D1 sequence. `editDocumentCore` is the find/replace surface (MCP `edit_document` only — no HTTP `PATCH`): it does NOT re-implement the sequence — it loads the **retained source S** via `readDocumentSourceCore`, applies the string edits (pure logic in `src/edit.ts`), then DELEGATES the write to `updateDocumentCore` passing the doc's **own** `source_format` (no longer hardcoded `"html"` — getting this wrong feeds a Markdown source to the HTML identity pipeline and corrupts *silently*), so there's exactly one copy of the sanitize→cap→R2→D1→FTS batch. Match is against the STORED **source** (Markdown for md docs, HTML for html docs — so a Markdown doc edits its Markdown and the reader theme survives by construction); the response adds a `replacements` count so an agent can tell "patch landed" from "sanitizer normalized." A source-unavailable doc (un-backfilled, `source_r2_key IS NULL`) hard-fails with `source_unavailable` — no fallback to H. Listing-row reads (`listDocumentsCore`, `findDocumentBySlugCore`, and `searchDocumentsCore`) share `LISTING_SELECT_COLUMNS` / `LISTING_JOINS` constants so the projection stays in lockstep — when adding a column to `DocumentListing`, update those two constants once and all three surfaces follow. **Wire-shaping the success Result is also centralized (`src/wire.ts`, Phase 2b):** core returns `{ ok: true, … }`-tagged Results; the on-the-wire JSON strips that `ok` (revoke renames it `revoked`). `toWriteResponse` / `toEditResponse` / `toRevokeResponse` are the ONE copy of that strip (typed against `WriteResponse`/`EditResponse`/`RevokeResponse` in `contract.ts`), used by both HTTP (`createDocument`/`updateDocument`/`revokeDocument` in `index.ts`) and MCP (`publish_document`/`update_document`/`edit_document` in `mcp.ts`) — don't re-hand-list those fields in a handler; call the mapper.

**Text-read path runs at READ time, not write.** `readDocumentTextCore` fetches the sanitized HTML and pipes it through `htmlToMarkdown` (same WASM module as `sanitize`, separate exported function). No per-version markdown cache in v1 — the parse is single-digit ms for typical docs and is dwarfed by the R2 GET it shares. The markdown text-read derives from the sanitized **H** blob; conversion always runs on the *cleaned* bytes so the text channel can't surface anything the sanitizer stripped. **The retained source `.src` blob is a separate read surface, not part of this path:** `readDocumentSourceCore` fetches the unsanitized **S** (the bytes as submitted) and re-runs the advisory pass on it (markdown-or-identity → sanitize → `detectAdvisories`) so a source-read carries `unsanitized: true` plus `stripped[]`/`will_not_render[]` — "the live render differs from this source here" is surfaced, never silent. Source-read is **credential-gated — operator OR agent (operator ≥ agent), never anonymous, never public** (the `requireReader` gate in `serve.ts`, which refuses only an anonymous principal): in this single-tenant whole-fleet model any active agent key already reads and overwrites every doc, so source discloses no authority the caller lacks — and the operator, the apex principal, must never rank below an agent (an earlier build gated `/source`+`/text` on `authenticateAgent` alone and so locked the operator out entirely). Render path stays H-only and never serves S. **Exception:** the FTS5 search index in `documents_fts` is populated at WRITE time using the same `htmlToMarkdown` call — the FTS shadow tables are the sole storage of the body-as-text for search. The read endpoint still re-derives fresh from R2 on each call (no D1 dependency for the markdown view); the FTS write-time copy is a separate, write-only side channel that only exists to feed BM25 ranking. The two can drift by at most one converter-version step if the converter changes between an old write and a new read — acceptable, and the `converter_v` stamp on the read response signals when it's happened.

**Two security layers (in order):**
1. **Sandbox + strict CSP at render** (`src/serve.ts`). `GET /d/:id` returns a tiny HTML shell with `<iframe sandbox>` (empty = all restrictions on); `GET /d/:id/raw` is what the iframe loads under `default-src 'none'` + `frame-ancestors 'self'`. Two URLs is deliberate — `frame-ancestors` is header-only, so the bytes must come from an HTTP response, not `srcdoc`. **This is the load-bearing wall.**
2. **Ammonia-WASM sanitization at write** (`sanitizer/src/lib.rs` → `src/sanitizer.ts`). Strips `<script>`, `<meta http-equiv>`, `<iframe>`, dangerous URL schemes, inline event handlers. Cheap insurance behind the CSP wall (covers e.g. `<meta refresh>` which CSP can't). **`<style>` blocks are now ALLOWED through** (sanitizer v1.4 — `ammonia-v1.4`): the sanitizer does NOT parse the CSS (verbatim passthrough), exactly as it never parsed inline `style="…"` attrs — defense against CSS vectors is Wall 1 (the render-time CSP + iframe sandbox), not the allowlist (docs/design/style-support-design.md).

**Storage model.** D1 (`META` binding) holds the metadata — see `migrations/0001_init.sql` for `agents`, `agent_keys`, `documents`, `versions`; `0002_oauth_clients.sql` for `oauth_clients` (client_id ↔ agent_id join); `0003_source_format.sql` for `versions.source_format`; `0004_document_metadata.sql` for `versions.title` / `description` / `tags` (JSON-encoded array; NULL when unset — but `tags` was later **lifted to `documents.tags` and dropped from `versions` in 0012**); `0005_document_slug.sql` for `documents.slug` + partial UNIQUE INDEX where slug IS NOT NULL; `0006_documents_search.sql` for the `documents_fts` FTS5 virtual table (one row per live document, columns `document_id UNINDEXED, title, description, tags, body` with porter+unicode61+remove_diacritics tokenizer — **the `tags` column was dropped in 0012**, leaving `document_id, title, description, body`); `0007_key_expiry.sql` for `agent_keys.expires_at` (NULL = never; non-NULL only on the short-lived publish credentials minted by `create_publish_credential`); `0008_source_retention.sql` for `versions.source_r2_key` + `versions.source_size_bytes` (both nullable, no default — NULL = un-backfilled/legacy, the loud presence-flag `editDocumentCore`/`readDocumentSourceCore` hard-fail on with `source_unavailable`, never a legacy fallback); `0009_slug_tombstones.sql` for the `slug_tombstones` table (slug PK, document_id, retired_at, reason) that makes a claimed slug semi-permanent — reversing 0005's "released on revoke" (GitHub issue #6); `0010_slug_redirect.sql` adds `slug_tombstones.redirect_to` (target `public_id`) for the loud operator/rename redirect; and `0011_document_visibility.sql` adds `documents.visibility` (`public`|`private`, CHECK-pinned — column `DEFAULT 'public'` backfills legacy rows so already-shared docs keep serving, but new docs are born EXPLICITLY at `DEFAULT_DOCUMENT_VISIBILITY` ([var], default `"private"`), so the column default only ever covers pre-migration rows; GitHub issue #7); and `0012_document_tags.sql` **lifts `tags` from per-version to document-level** — adds `documents.tags` (backfilled from each doc's current version), `DROP`s `versions.tags`, and rebuilds `documents_fts` WITHOUT its `tags` column (the body text is carried through a temp table since it's WASM-derived and lives only in the FTS index); and `0013_authorship.sql` records the **author principal** so the operator is a first-class author rather than a fake agent — `documents.created_by_kind` (the creator's kind, `agent`|`operator`, CHECK-pinned, `DEFAULT 'agent'` backfills legacy) plus per-version `versions.author_kind` + `versions.author_agent_id` (the **queryable** writer, replacing the R2-customMetadata-only writer tag; legacy rows default to `agent`/NULL — the true historical writer survives only in customMetadata, not backfilled), so an operator-authored doc/version is honest and a document's full per-version author list is queryable; and `0014_document_status.sql` adds the **lifecycle axis** — `documents.status` (`active`|`deprecated`|reserved `archived`, CHECK-pinned, `DEFAULT 'active'` is both backfill and product default) plus `documents.superseded_by` (a replacement `public_id`, the document-level analogue of `slug_tombstones.redirect_to` — loud, never auto-followed, single-hop, live-target/no-self validated; GitHub issue #21); and `0015_source_sha256.sql` adds `versions.source_sha256` (nullable, no default — NULL = pre-0015/un-backfilled, same presence-flag posture as the 0008 source columns), the SHA-256 of the retained source S the Worker stamps at write time (`sha256Hex(prep.sourceBytes)` — the *same* bytes `X-Content-SHA256` hashes, so for a byte-exact publish it equals the publisher's `sha256sum file`). Surfaced as `current_source_sha256` on listing rows + `source_sha256` on the write response and source-read so an agent can confirm a local copy is still the current source and skip a source re-read before an edit (GitHub issue #35; ergonomics only — gates nothing); and `0016_document_links.sql` adds the **link graph** (GitHub issue #40) — the `document_links` table (`src_doc_id`, `position`, CHECK-pinned `target_kind` `public_id`|`slug`, `target_value`; PK on the (src, kind, value) triple + a `(target_kind, target_value)` index): one row per distinct on-platform link in a doc's CURRENT render, extracted at write time from H via the same `extractOutboundLinks` packs use, with the target stored as the RAW addressed name (late binding — resolved at read time by `documentLinksCore` into `live`/`redirected`/`retired`/`revoked`/`missing`), synced inside the same `META.batch()` as the FTS row (see the link-sync convention below). Status is the third state axis (revoked = existence, visibility = anonymous read, status = currency): a `deprecated` doc still serves and still ranks in search (marked in the row) but is excluded from context-pack fills by default; `archived` is unsettable until its behavior is wired. Tags join `slug`/`visibility`/`status` on the identity/classification side: they survive content updates and restores, a tag change never bumps a version, and they're no longer FTS-indexed (the `?tags=` filter is a LIKE on `documents.tags`, never FTS). R2 (`DOCS` binding) holds **two** blobs per version: the sanitized render **H** at key `<docId>/v<n>` and the retained source **S** at the dot-suffix sibling `<docId>/v<n>.src` (content-type from `source_format`). Source bytes count toward the fleet storage cap (`size_bytes` stays "rendered H bytes"; `source_size_bytes` is the separate per-version S size — `checkStorageCap` SUMs both, coalescing NULL legacy rows to 0). `versions` rows are append-only and survive revoke as an audit trail (only the R2 bytes are purged — **both** H and the `.src` sibling). KV (`OAUTH_KV` binding) is owned by the OAuth provider library (clients, grants, tokens). **Vectorize (`VECTORIZE` binding) + Workers AI (`AI` binding)** back semantic search (docs/design/vector-search-design.md, BUILT): the `agent-web-host-docs` index (1024-dim cosine) holds **N chunk vectors per document** keyed `${documents.id}#${i}`, embedded with `@cf/qwen/qwen3-embedding-0.6b`. **No D1 migration** — vectors live entirely in Vectorize; the chunk ID yields the doc id, so there's no per-vector metadata column and no chunk-count column (the fixed `#0…#${MAX_CHUNKS-1}` range-delete handles a shrunk chunk count). The one stored metadata field, `preview`, is a non-indexed display snippet. Sync is **best-effort, eventually-consistent, OUTSIDE the D1 batch** (see the vector-sync convention below). On revoke, `documents.revoked_at` is flipped **before** the R2 batch delete so the doc is unreachable instantly even if R2 cleanup hangs; the same batch clears `documents.slug`, **retires that slug into `slug_tombstones`** (so the name is reserved forever — migration 0009, not freed), and deletes the FTS row, so kill + slug-retire + search visibility all land atomically (the chunk-vector delete rides `ctx.waitUntil` after the batch — not the kill switch, just index reclaim; the read-path D1 re-join enforces `revoked_at is null` regardless).

**Single-tenant trust model.** Any active agent key under one operator can `PUT` to any document — `core.ts` deliberately does not scope updates by `created_by`. **The write cores take an `Author` (`{kind:"operator"} | {kind:"agent";agentId}` — the `Principal` union from `access.ts`, minus anonymous), NOT a bare `agentId: string`** (migration 0013): the operator is a first-class author recorded as the operator principal, never smuggled through as a fake agent (the old `restoreVersionCore("operator")` string hack is gone). `documents.created_by`/`created_by_kind` retain the original creator (`created_by` is NULL for an operator-created doc — its `created_by_kind` is `"operator"`); each new version records its own writer in `versions.author_kind`/`author_agent_id` (queryable, not just R2 `customMetadata`). The operator authors through `POST /admin/documents` / `PUT /admin/documents/:id` (operator JSON write door, `admin.ts`), same core path as the agent door. `listDocumentsCore` shows the whole fleet to every caller. If per-agent scoping ever becomes needed, add a `createdBy?` arg to core functions. **Unsanitized source bytes are now also at rest in R2** (the `.src` blob per version, since migration 0008): it is auth-gated (credential-gated — operator or agent, never anonymous) and **never served to a browser** — the render path is still H-only behind the iframe+CSP wall. Source retention changes only the *ingestion-as-context* surface (an agent asking for `representation:"source"` can pull markup-form bytes the sanitizer would have neutralized on the render path); the provenance tag + credential auth cover exactly that increment, and revoke purges the `.src` siblings alongside H so the kill switch leaves no unsanitized source resident.

**Per-document visibility — a two-axis read model (migration 0011, GitHub issue #7).** Orthogonal to revoke: every document is `public` or `private`, and `src/access.ts` is the single pure chokepoint. `canRead(principal, doc)` is top-down first-match-wins (revoked→false, operator→true, agent→true, anonymous→`visibility === "public"`); `resolvePrincipal` resolves operator → agent → anonymous from the request. **Visibility gates ONLY the anonymous browser surface** — `serveShell` / `serveRaw` / `serveBySlug` / `serveHomepage`, all enforced at the **`serveRaw` chokepoint** (`resolvePrincipal` → `canRead`). The operator (cookie session OR bearer) and any active agent key read everything, so the credentialed read surfaces (`/text`, `/source` — gated operator-or-agent via the `requireReader` helper, operator ≥ agent; MCP read/list/search — agent-scoped) are unaffected by visibility, and `listDocumentsCore` still shows the whole fleet to every authenticated caller. A private doc denies as the **same opaque 404** a missing/revoked doc gives — **never a 401, no existence oracle**. New docs are born at `DEFAULT_DOCUMENT_VISIBILITY` ([var] in `env.ts`/`wrangler.toml`, default `"private"`), bound EXPLICITLY by the write path so the column's own `DEFAULT 'public'` only ever covers legacy rows; the operator flips it via `POST /admin/documents/:id/visibility` (JSON) or the Manage page (`setDocumentVisibilityCore`, no version bump). Visibility-CHANGE authority is a separate operator check (`requireOperator`), deliberately kept OUT of `canRead` (which governs read on the public surface and nothing else).

**Sanitizer is bundled as WASM.** `src/sanitizer.ts` imports `sanitizer/pkg/sanitizer_bg.wasm` directly as a `WebAssembly.Module` (wrangler's `[[rules]] type = "CompiledWasm"` rule) and calls `initSync`. **Do not use the wasm-pack `--target web` glue's `init()`** — it tries to fetch via `import.meta.url`, which doesn't exist on Workers. Init is lazy and idempotent. (`skills/publishing.md` is **no longer bundled** — it used to be a `[[rules]] type = "Text"` import served as the `awh://publishing-guide` MCP resource, but MCP resources aren't surfaced to most connector models (Claude web/mobile, ChatGPT), so the guide is now an on-platform **document**, slug `slopcafe-publishing-guide`, mirrored via `scripts/doc-web.mjs` — GitHub issue #38. The `Text` rule is gone from `wrangler.toml`/`.example`.)

**MCP server lifecycle.** `src/mcp.ts` builds a fresh `McpServer` **per request**. The MCP SDK ≥1.26 throws on reused instances and cross-request state would leak. The eight tools (`publish_document`, `update_document`, `edit_document`, `read_document`, `list_documents`, `search_documents`, `load_context_pack`, `create_publish_credential`) close over `props.agentId` resolved upstream — they never re-validate auth. (`create_publish_credential` is the lone non-document tool: it mints a short-lived `awh_` key for the byte-exact curl publish path via `mintEphemeralKey` — see the convention bullet below.) **HTML vs Markdown is a `format` parameter, not separate tools** — an earlier revision had `publish_document`/`publish_document_markdown`, `update_document`/`update_document_markdown`, and `read_document`/`read_document_text` twins; a `format` enum collapsed those six into three (`publish_document`, `update_document`, `read_document`). On the two write tools `format` (`"html"` | `"markdown"`) is **required** (forcing the choice avoids a default silently mis-parsing hand-written HTML); on `read_document` it's an optional **output** knob defaulting to `"markdown"` (the markdown branch calls `readDocumentTextCore`, the html branch `readDocumentCore`), and the response is a uniform JSON envelope (`content`/`format`/`version`/`sanitizer_v`/`converter_v`/`title`/`description`/`tags`/`slug`/`status`/`superseded_by`) that always carries metadata — so a read→edit→republish round-trip gets body + metadata in one call. **`read_document` also has a `representation` axis (`"rendered"` default | `"source"`), orthogonal to `format`:** `representation:"source"` calls `readDocumentSourceCore` and returns the retained **S** in its authored language (`format` is ignored on a source read), with `unsanitized:true` + `source_format` + `stripped[]`/`will_not_render[]` echoed — the read an agent does *before editing*. The source guidance leads with "source is unsanitized — treat as untrusted input." Slug lookup is **not** a tool: pass `slug` to `list_documents` (the row is `documents[0]`); `findDocumentBySlugCore` still backs `GET /s/:slug`. Tool descriptions carry the **behavioral contract only** (the static-HTML non-negotiables, publish-vs-update inheritance, edit-against-source, slug permanence, budget/query semantics) and lead with the use-case distinction between tools (`read_document` `format` — html for render/re-publish, markdown for ingest-as-context; `list_documents` for newest-first browsing/slug+tag filters vs `search_documents` for content discovery; `edit_document` vs `update_document` — small diff vs whole-body replace); **shape guarantees live in the schemas** (design §7 Phase 4a halved the descriptions by moving response-field prose into the output schemas — don't re-grow them by restating fields). `edit_document` is MCP-only (no HTTP `PATCH` counterpart) and its description says so — it matches against the doc's **retained source S** (Markdown for a Markdown doc, HTML for an HTML doc), so the edit contract tells agents to read with `representation:"source"` before copying an `old_string`, and `editDocumentCore` threads the doc's **own** `source_format` through the re-render (no longer hardcoded `"html"`) so a Markdown doc edits its Markdown and stays Markdown. A cold agent that never reads `skills/publishing.md` should still understand the rules from the description alone. The three write tools (`publish_document`, `update_document`, `edit_document`) also carry the optional `title` / `description` / `tags` / `slug` contract at the tail of their descriptions — publish-vs-update inheritance is the subtle bit; shared schema-field constants near the bottom of `mcp.ts` (`CONTENT_FIELD`, `WRITE_FORMAT_FIELD`, `READ_FORMAT_FIELD`, the `*_FIELD` metadata constants) keep the wording consistent across tools. When editing tool descriptions, preserve the priority order (non-negotiables first) — length-trimmed renders truncate the tail. **All eight tools register an `outputSchema` and return `structuredContent`** (design §7, Phase 4a): the envelopes are `contract.ts` schemas (`WriteResponse`/`EditResponse`/`ListDocumentsResponse`/`PackResponse` reused verbatim; `McpReadDocumentResponse`/`McpSearchDocumentsResponse`/`CreatePublishCredentialResponse` are the MCP-only ones), the SDK **hard-fails any non-error result that doesn't validate** (envelope drift = a broken tool call, not silent drift), and every success path goes through the `structuredOk` helper (the same JSON also rides the legacy text block; `textError` results are exempt from validation). Two union-shaped outputs are encoded as optionality because a tool output schema must be ONE object: `read_document` (document envelope ∪ unfollowed-redirect report) and `search_documents` (`pack`/`omitted` present only with `include_bodies`).

## Conventions and gotchas

- **`If-Match` is required on `PUT /d/:id`** (428 if missing). Send `"v<n>"` for optimistic concurrency or `*` to skip. Strong tags only; no weak tags, no multi-tag lists.
- **`POST /d` / `PUT /d/:id` accept `Content-Type: text/html` or `text/markdown`** (`text/x-markdown` too; anything else → 415; the gate is `parseInputFormat` in `src/index.ts`). A Markdown body is parsed (CommonMark + GFM) to HTML before sanitizing; either way the stored bytes are sanitized `text/html; charset=utf-8`, and which input path produced the version is recorded in `versions.source_format`.
- **Optional `X-Content-SHA256` byte-exact integrity check on `POST /d` / `PUT /d/:id`** (`src/integrity.ts`, wired via `readVerifiedBody` in `src/index.ts`). Hashes the **raw received body before sanitization** — verifies the wire, not the sanitizer's transformation, so it's orthogonal to the `modified` flag. Malformed header → 400 `bad_integrity_header`; mismatch → 422 `integrity_mismatch`. **HTTP-only by design** (the companion to the `curl --data-binary @file` byte-exact publish path): the hash must come from the same deterministic tool that streams the file, so there is no MCP equivalent — the two primary MCP write-tool descriptions say so. `docs/design/byte-exact-publish-design.md` records the rationale (the binding constraint at 66–85 KB is model regeneration of the `content` argument, not transport; chunked-append doesn't fix it).
- **Short-lived publish credentials (`agent_keys.expires_at`, migration 0007).** The byte-exact curl path needs a bearer at the shell, but Claude's connector settings can't store one and the connector's OAuth access token isn't exposed to the model/shell. So the already-OAuth-authenticated MCP session mints one on demand: the `create_publish_credential` tool → `mintEphemeralKey` (`src/admin.ts`) inserts a normal `awh_` key with a non-NULL `expires_at` (default 15 min, max 60, clamped). `authenticateAgent` (`src/auth.ts`) rejects expired keys exactly like revoked ones via the pure `isKeyExpired` helper. `expires_at = NULL` = never expires (every operator-minted key + legacy rows), so the migration is a no-op for them. **No scope column in v1** — single-tenant, the damaging vector (overwriting docs) isn't blocked by a write-only scope anyway, and the minted key grants nothing beyond the MCP session's existing authority; scope-limiting is the additive multi-tenant follow-up. The minted **`key`** is the deliberate disclosure surface — never log it (same discipline as the OAuth token). **The returned `recipe` is NOT a secret** (GitHub issue #34): it references the key via an `$AWH_KEY` env var (the agent `export`s the `key` once — prefix the line with a space to skip shell history) instead of inlining it, so the token never reaches the curl command line and the recipe is safe to echo. Don't "re-harden" by interpolating `result.key` back into the recipe — that's the leak #34 removed. Revoke early via `DELETE /admin/keys/:id`.
- **Optional metadata on POST/PUT /d**: `X-Doc-Title`, `X-Doc-Description`, `X-Doc-Tags` (comma-separated), `X-Doc-Slug`. MCP write tools take the same shape as schema fields. **Header values are read as UTF-8** (`decodeHeaderUtf8` in `metadata.ts`, issue #31) — an HTTP header value arrives as a Latin-1 byte view (Fetch ByteString: one JS char per raw byte), so without the decode a non-ASCII title (`Café — résumé`, an em-dash, CJK) would store as mojibake. The decode is *fatal-with-fallback*: pure ASCII is identity, valid multi-byte UTF-8 is recovered, and a lone Latin-1 byte (e.g. a browser `fetch` sending `é` as `0xE9`) is invalid UTF-8 so it's preserved as-is. This makes the header channel UTF-8 like the body and MCP JSON args, and like the served `charset=utf-8` render path — so agents send non-ASCII literally instead of entity-encoding (the sanitizer normalizes character entities to literal UTF-8 anyway; only `& < > "` + `&nbsp;` stay encoded). On UPDATE, an *omitted* field inherits the prior version's value (or, for slug, the current document's value); an explicit `""` clears (and for title, re-derives from the new content's H1; for slug, releases it). Tags charset is restricted to `[A-Za-z0-9_-]` — invalid chars are silently stripped, not rejected. Slug charset is `/^[a-z0-9](?:[a-z0-9_-]{0,62}[a-z0-9])?$/` — invalid input is **rejected** (`invalid_slug` → 422), not silently sanitized (uniqueness means mutating the input could surprise-collide). Slug is unique across live docs (partial UNIQUE INDEX), conflict → `slug_taken` → 409. The render path applies anti-phishing normalization to title only (bidi/zero-width strip); description stores bidi for fidelity since it doesn't reach the browser tab.
- **Slug AND tags live on `documents`, not `versions`.** Title/description evolve per-version because they describe content; **slug and tags are document-level** (tags since migration 0012) — slug is identity-adjacent (uniqueness per-document), tags are classification (a document's place in the collection, not a revision's wording). Both survive content updates and restores and never bump a version when changed. It survives version increments unchanged unless the agent explicitly changes it (atomic claim-new + retire-old in the same D1 batch). On revoke (and on rename/clear) `documents.slug` is set NULL alongside `revoked_at`, but the slug is **NOT reusable** — it is retired into `slug_tombstones` (migration 0009) and reserved forever; `/s/<slug>` then returns **410 Gone** and a reclaim attempt → `slug_retired`. `documents.slug` therefore always means "the live slug"; the permanent reservation is the separate tombstone table (a claim checks both via `resolveSlug`; `serveBySlug`/`read_document` consult `findSlugTombstoneCore` on a live-lookup miss to choose 410-vs-404). This reverses the original "released on revoke" contract per GitHub issue #6 — slug reuse is surprising (a shared/linked `/s/<slug>` must not silently start serving unrelated content). Pulling slug from `d.slug` in the joined read/list queries — not from prior_metadata — keeps the per-version `resolveMetadata` helper unaware of cross-document state. **A retired slug can REDIRECT (migration 0010, `slug_tombstones.redirect_to` = target `public_id`), but only LOUDLY:** a rename auto-forwards the old slug to the same doc's `public_id`; the operator can cross-redirect via `POST /admin/slugs/:slug/redirect`; `serveBySlug`/`read_document` never auto-follow — browser gets a click-through interstitial, agent gets `409 slug_redirected` (HTTP) / a `{redirected:true}` envelope (MCP) and must opt in with `follow_redirects`. Single-hop (target is a `public_id`, so no chains/loops; a dead target falls back to 410). Operator escape hatch `DELETE /admin/slugs/:slug` force-releases a tombstone (the only un-retire path); `DELETE /admin/slugs/:slug/redirect` drops just the redirect.
- **Operator document-management page + browser login-link 404 (`src/serve.ts`).** `GET /d/:id/manage` (`serveManagePage`) is the cookie-session-gated operator page folding three actions onto one doc: visibility toggle (`handleVisibilityForm`, `POST /d/:id/visibility`), slug add/rename/clear (`handleSlugForm`, `POST /d/:id/slug`), and revoke. A logged-out caller gets a sign-in card rendered with **no DB hit** (so the page is no existence oracle). The shared `authorizeOperatorForm` mirrors `handleRevokeForm`'s ladder (pasted `operator_token` OR cookie + `csrf_token`). The shell topbar's old "Revoke…" kebab item is now **"Manage…"** (revoke folded into the manage page); a signed-in topbar also shows a Public/Private badge. JSON operator twins live in `admin.ts`: `setDocumentVisibility` (`POST /admin/documents/:id/visibility`) and `setDocumentSlug` (`POST /admin/documents/:id/slug`); the slug path reuses `setDocumentSlugCore` (no version bump; `resolveSlug` + `tombstoneSlug` so a rename auto-forwards — identical to the agentic `update_document` slug change). **Browser login-link 404:** because private docs exist, a valid `/d/:id` or `/s/:slug` URL 404s to a logged-out operator — so `notFoundBrowser` (+ `NOTFOUND_CSP`) returns an HTML 404 card with a "Sign in" link that round-trips back via `/login?next=<requested URL>`. It **never reads doc state** (only echoes the requested path into `next`), so a private doc's 404 is byte-identical to a nonexistent one's for the same URL. Wired into `serveShell` (all three 404s), `serveBySlug` (private-to-anon; malformed + never-claimed-slug pick HTML-vs-plain by the `Authorization` header), and `serveRetiredSlug`'s no-tombstone case. Agent/API 404s (`/raw`, `/text`, `/source`, anything carrying `Authorization`, the `/manage` POST handlers) keep the plain `text/plain` `notFound()` body.
- **Operator version history + restore-as-new-version (`src/serve.ts`, `src/core.ts`, MCP).** No migration — `versions` already retains every version's H (and `.src`) bytes until revoke purges them, so "history" is pure additive read code. Operator browser surface: `GET /d/:id/v/:n` (shell) + `/d/:id/v/:n/raw` (bytes), `authenticateOperatorRequest`-gated (Bearer OR cookie), and a Version history table on the manage page with a Restore button per non-current version (`POST /d/:id/restore`). This is an **operator** axis, deliberately NOT folded into `canRead`/visibility — a public doc's history is as operator-only as a private one's; a non-operator gets the same opaque 404. Agents read history through MCP `read_document` (`version` to read one, `include_history` for the manifest) — **read-only; there is no agent restore in v1.** **Restore is mandatorily restore-as-NEW-version, never a `current_ver` rewind:** `updateDocumentCore` computes `nextVer = current_ver + 1`, so pointing `current_ver` backward would make the next ordinary update collide on the `(document_id, version_no)` PK. `restoreVersionCore` reads version N's retained source S and re-publishes it forward through `updateDocumentCore`, restoring body + title/description/tags but keeping the doc's current slug (slug is identity, not content). A pre-0008 version with no retained source **hard-fails `source_unavailable`** — there is deliberately NO fall-back-to-H legacy branch (identical to `edit_document`'s contract); the manage table shows a muted "no source" in place of the Restore button, and such docs are revoke-and-republished (the pre-launch no-legacy-code stance). The `test/`-style suite has no D1 harness, so this path is covered by typecheck + the local `wrangler dev` E2E (publish → update×2 → restore v1 → confirm the next update lands monotonically, no collision).
- **`public_id` regex is fixed**: `/^[A-Za-z0-9_-]{22}$/` (`PUBLIC_ID_RE` in `src/serve.ts`). It's safe to interpolate into HTML templates after this check; `serveShell` does so.
- **Storage cap is best-effort.** `checkStorageCap` runs `SUM(size_bytes)` outside the insert batch; two concurrent writes can both pass. v1 accepts the slight overrun.
- **List endpoints are cursor-paginated** (`src/pagination.ts`). The four list surfaces — `GET /admin/agents`, `GET /admin/agents/:id/keys`, `GET /admin/documents`, and the MCP `list_documents` tool — share `parseHttpListParams` / `parseMcpListArgs` + the `paginate()` peek helper. Default page size 50, max 200. Ordering is `(created_at DESC, id DESC)` — the `id` tiebreaker is mandatory because D1's strftime stamps to ms and collisions are rare-but-possible; without it cursors could skip a row at a page boundary. Cursors are opaque base64url(JSON `{ts, id}`) — never construct them by hand. When you add a new list endpoint, route it through `pagination.ts` rather than introducing a new style.
- **Search is hybrid (keyword + semantic), NOT cursor-paginated.** `GET /admin/documents/search` and the MCP `search_documents` tool reuse the filter parser (`parseHttpListParams` / `parseMcpListArgs`) for `tags` / `slug` / `limit`, but ignore the cursor field — relevance rank (BM25 or the fused RRF score) is not a stable cursor key the way `(created_at, id)` is. Results are capped at `limit` (default 50, max 200) and the response has no `next_cursor`. A `mode` param (`hybrid` default | `keyword` | `semantic`) selects the legs: `keyword` is FTS-only (today's behavior), `semantic` is vector-only, `hybrid` fuses both via `reciprocalRankFusion` (docs/design/vector-search-design.md §10). **`searchDocumentsCore` now takes the RAW query** (not a pre-tokenized FTS match) — it runs `buildFtsMatchQuery` internally for the keyword leg AND embeds the un-tokenized query for the semantic leg, so the callers no longer pre-tokenize. `bad_query` (422) now fires only when **no leg can run** (keyword mode with no usable tokens, or hybrid/semantic when the embed failed AND there are no tokens) — a tokenizable-but-zero-results query is a legit empty set, not an error. The query embed is best-effort: an AI failure degrades hybrid/semantic to the keyword leg, never a hard error. `matched_field` gains `"semantic"`; `score` scale differs by mode (fused/BM25/cosine), comparable only within one result set.
- **Document filtering** (`tags`, `slug`) lives on the document list/search surfaces — the parser collects them for every endpoint that uses `ParsedListParams`, but only `listDocumentsCore` and `searchDocumentsCore` consult them in SQL (the agents/keys lists silently ignore the fields). `tags` is AND semantics: one `tags LIKE ? ESCAPE '\\'` per requested tag against the JSON-encoded `documents.tags` column (document-level since migration 0012 — the filter never used FTS). The underscore in the tag charset collides with LIKE's `_` wildcard, so `tagLikePattern()` in `core.ts` escapes it with `\_`. Tag inputs are silently sanitized to the stored shape at parse time (matching write-time semantics); slug inputs are validated and reject with `bad_slug` on invalid charset (matching the write-time reject-not-sanitize rule). `findDocumentBySlugCore` is the dedicated single-row lookup behind the `GET /s/:slug` HTTP redirect (it validates the slug via `validateSlugInput` upstream of the DB hit). There is no longer a `find_document_by_slug` MCP tool — the slug-lookup path on MCP is `list_documents` with a `slug` filter (returns 0 or 1 rows; the caller reads `documents[0]`).
- **FTS sync is write-path-local.** No D1 triggers — `publishDocumentCore` / `updateDocumentCore` / `revokeDocumentCore` each include the `documents_fts` INSERT/UPDATE/DELETE in their existing `META.batch()` so the FTS row tracks the document's current version atomically. The body column is the markdown derivation of the sanitized HTML, computed once per write via `htmlToMarkdown(prep.cleanedHtml)`. Since migration 0012 the FTS row indexes only `title`/`description`/`body` — **tags are NOT FTS-indexed** (the `setDocumentTagsCore` mutator and the `?tags=` LIKE filter both touch `documents.tags` directly, so a tag-only write touches no FTS row). The FTS row is the SOLE storage of body-as-text — no `body_text` column on `versions`. If you add a new write surface, mirror the FTS sync inside the batch (or extract a helper) — don't ship a write path that bypasses FTS, or search will silently lag.
- **Link-graph sync is write-path-local too (migration 0016, GitHub issue #40).** `documentLinkStatements` (core.ts) emits the DELETE-then-INSERT `document_links` statements and the publish/update batches splice them in alongside the FTS row (edit/restore ride `updateDocumentCore`; revoke deletes the doc's outbound rows in the revoke batch — the FK cascade never fires because revoke tombstones rather than deletes the `documents` row). Targets are stored RAW (a link may name an unclaimed slug — legal; that's the late binding), self-links are filtered, rows cap at 200/doc, and the multi-row INSERT is chunked under D1's bind limit. Read-time resolution lives in `documentLinksCore` (backlinks = live docs addressing this doc's public_id or CURRENT slug — a link to a since-renamed slug is deliberately NOT a backlink, it reports as `redirected` on the source doc's outbound list; outbound states = `live`/`redirected`/`retired`/`revoked`/`missing`). **Backlinks are credentialed reads, never anonymous** (`GET /d/:id/links` via `requireReader`; MCP `read_document include_links`; the manage page's Link graph panel) — a backlink row is a listing row for ANOTHER (possibly private) doc. Inbound rows pointing at revoked/renamed targets are NOT cleaned up — they resolve to broken states at read time, which IS the broken-link report. Pre-0016 docs have no rows until the operator runs `POST /admin/links/backfill` (R2-reading sweep, always rebuild-semantics, resumable; console Maintenance form twin `POST /admin/console/links/backfill`) — and `GET /admin/links/orphans` over-reports until that one-time run. See docs/design/backlinks-design.md.
- **Vector sync is write-path-adjacent, NOT in the batch.** Unlike FTS, the Vectorize sync is **eventually-consistent and best-effort** — Vectorize is NOT transactional with D1 (async mutations, 5–10 s visibility lag), so it *cannot* join the `META.batch()`. The write cores (`publishDocumentCore` / `updateDocumentCore` / `editDocumentCore` / `restoreVersionCore`) and `revokeDocumentCore` take an optional trailing `waitUntil?: WaitUntil` arg; callers pass `ctx.waitUntil.bind(ctx)` (HTTP handlers in `index.ts`, the MCP tool closures, the operator-authoring handlers in `admin.ts`) and after the batch commits, schedule `syncDocumentVector` / `deleteDocumentVector` (in `src/vector-io.ts`) off the request's lifetime. **When `waitUntil` is absent (unit tests, any un-plumbed caller) the sync is skipped silently** — the doc is fully functional, just not yet indexed; the next update or the operator backfill heals it. The I/O helpers swallow + log (code only, never the body — it's user HTML) so an AI/Vectorize hiccup never fails a publish. **Self-healing assumes a next write**, and much of this corpus is write-once — so the real durability net for dropped publish-time syncs is the operator-run **`POST /admin/vectors/backfill`** (`mode:"missing"` is incremental — embeds only docs whose `#0` chunk is absent). The **Queue-backed durable sync** (retrying consumer) is the documented phase-4 upgrade seam (docs/design/vector-search-design.md §6/§13). **Chunk-ID invariant:** vectors are keyed `${docId}#${i}`; write/delete/collapse all share that convention via `src/vector.ts` (`chunkVectorId` / `docIdFromChunkId` / `chunkVectorIdRange`), and the write path **delete-then-upserts the whole fixed `#0…#${MAX_CHUNKS-1}` range** so a shrunk chunk count leaves no orphan tail (no chunk-count column needed). If you add a new write surface, thread `waitUntil` through to core the same way — don't call `syncDocumentVector` from a route handler directly.
- **Operator session cookies + CSRF (`src/session.ts`).** Two cookies, both **host-only** (no `Domain`) and `SameSite=Lax`: `awh_session` (HttpOnly — the signed credential) and `awh_csrf` (readable — delivers the CSRF nonce to the page). `Secure` is **conditional on request protocol** (`isSecureRequest`): set behind https custom domains, omitted over `http://localhost` (a blanket `Secure` is silently rejected on http, breaking `wrangler dev`). CSRF is **stateless signed double-submit**: the nonce lives inside the *signed* session payload, and a cookie-authed mutating request must echo it (`X-CSRF-Token` header for JSON/admin; `csrf_token` form field for the HTML `/login`, `/logout`, `/d/:id/revoke` forms) — the server constant-time-compares the echo to the verified session's nonce, so a cookie-vs-cookie injection can't forge it. The pure crypto/cookie core takes explicit `nowMs`/`operatorToken`/`epoch` (no `Date.now()`, no env) so it's unit-tested in `test/session.test.mjs`; only the thin env wrappers read `env`. Login/logout/revoke set **two** Set-Cookie headers via `headers.append` (not `.set`, which clobbers). `next` redirects are origin-whitelisted via `validateNext` (rejects `//evil.com`, `/\evil.com`, absolute URLs, CRLF). Never log the operator token or the derived signing key (same discipline as the OAuth token / minted keys).
- **Every server-rendered HTML response carries a CSP header — no exceptions.** Each HTML surface has a matching constant (`SHELL_CSP` / `RAW_CSP` / `REVOKE_CSP` / `NOTFOUND_CSP` in `src/serve.ts`; the console/login/authorize pages have their own) and the response MUST set `content-security-policy` even when the page is fully static today (`NOTFOUND_CSP` covers the static link-only cards: HTML 404, retired-slug 410, redirect interstitial). The pre-OSS audit found two pages that had silently shipped without one — the header is cheap defense-in-depth and the discipline only works if it's universal, so when you add an HTML response, pick (or add) a CSP constant in the same change.
- **Any API-surface change must update the MCP tool descriptions AND the HTTP API reference in the same commit.** New inputs, new response fields, new headers, changed status codes, changed inheritance/derivation semantics — all of these are part of the contract a cold MCP agent sees only via tool descriptions (a connector client surfaces tools, but rarely fetches the on-platform `slopcafe-publishing-guide` doc on its own), and the contract a consumer building against HTTP (e.g. the Flutter app) sees only via `docs/http-api.md`. Walk the eight tools in `src/mcp.ts` and confirm: (1) every inputSchema field for the changed surface exists with an up-to-date `.describe()`; (2) the prose description mentions the new field/behavior at the priority level it deserves (non-negotiables first, optional bits at the tail — length-trimmed renders truncate); (3) the JSON response shape documented in the description matches what the handler actually returns; (4) `skills/publishing.md` covers the same change so the on-platform `slopcafe-publishing-guide` doc (re-published from it via `scripts/doc-web.mjs`) and the tool descriptions don't drift; (5) **`docs/http-api.md` reflects the change** — the affected endpoint's request/response/headers/status-code table, plus any `DocumentListing`/`SearchHit` shape edits in its "Shared response shapes" section. An HTTP-only surface (no MCP equivalent) still belongs in `docs/http-api.md`; an MCP-only surface (e.g. `edit_document`, `create_publish_credential`) is noted as such in `docs/http-api.md`'s MCP section. If a change is HTTP-only by design, say so explicitly in the MCP tool description so an agent on MCP knows not to look for it. (6) **`docs/http-api.md` is also published live on Slopcafe** (slug `slopcafe-http-api`, public_id `0EtsEq6cnCeuOhBKO6ICzA`) so any connected agent can read it without repo access — re-publish that copy when the doc changes (byte-exact `PUT` of the file via `create_publish_credential` + `curl --data-binary`, **not** MCP `update_document` — regenerating a ~28 KB body as a tool argument is slow and truncation-prone; `docs/README.md` records the exact recipe), or it silently drifts. (7) **Bump `OPENAPI_INFO_VERSION`** (`src/openapi.ts`) when the wire shape moves, then regenerate (`npm run build:openapi`). The contract is **stable `1.x`** (cut `1.0.0` at the 2026-06 public launch): strict semver — **MINOR** for additive/backward-compatible shape changes, **PATCH** for docs/clarification-only edits, **MAJOR** for any break (removed/retyped field, changed code/status). This bump is a **human obligation, not test-enforced** — the freshness gate (`test/openapi.test.mjs` + `git diff --exit-code openapi.json`) only checks *spec-matches-source*, so a shape change can land with the version untouched and tests green (how `1.0.0` drifted). See `docs/design/api-contract-design.md` §14. The write path runs it as a **cheap pre-screen BEFORE `sanitize()`** to reject depth-bombs (GitHub issue #42) — both `sanitize()` and the precise `maxDomDepth` are ~**O(n²)** in nesting (html5ever tree-build + `RcDom` drop), so a near-byte-cap bomb burned seconds before the `too_deep` 422 could fire. The scan converts markdown first (pulldown-cmark is O(n)), then screens the HTML's open-tag depth ~1000× faster (9 ms vs ~58 s at depth 50k). It's an **approximate** measure (void/self-closing leaves don't count; comments + raw-text element content are skipped; quoted attrs handled) — the precise `maxDomDepth` post-`sanitize` check (issue #41) stays the authoritative converter-overflow guard, now always on shallow, pre-screened input. **Two-stage depth gate: cheap approximate pre-scan (perf) + precise post-check (correctness/crash-safety).**
