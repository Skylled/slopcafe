# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Build / deploy / dev:
```sh
npm run dev            # wrangler dev — uses .dev.vars + local D1/R2
npm run typecheck      # tsc --noEmit
npm run build:wasm     # rebuild the Rust→WASM sanitizer (auto-run via predeploy)
npm run deploy         # rebuilds WASM then `wrangler deploy`
```

Tests:
```sh
npm test                  # sanitizer corpus + advisories + metadata + pagination + search + edit + integrity + auth + session + access + conditional
npm run test:sanitizer    # cargo test in /sanitizer (host target, not wasm)
npm run test:advisories   # node --experimental-strip-types test/advisories.test.mjs
npm run test:metadata     # node --experimental-strip-types test/metadata.test.mjs
npm run test:pagination   # cross-file ts import → uses test/register-ts-resolver.mjs
npm run test:search       # FTS5 query tokenizer (buildFtsMatchQuery) — pure function tests
npm run test:edit         # applyEdits find/replace logic (edit_document) — pure function tests

# Run a single sanitizer test (Rust name match):
(cd sanitizer && PATH="$HOME/.cargo/bin:$PATH" cargo test rejects_script_tag)
```

Database (D1):
```sh
npm run db:migrate:local            # apply migrations to local D1
npm run db:migrate:remote           # apply migrations to remote D1
npm run db:console:local  "SELECT * FROM agents"
npm run db:console:remote "SELECT * FROM agents"
```

`predeploy` injects `$HOME/.cargo/bin` into PATH so `wasm-pack` resolves from a fresh shell. Rust toolchain (`rustup` + `wasm32-unknown-unknown` target + `wasm-pack`) is only required for `build:wasm`.

## Git workflow

When the operator says "commit your work" (or similar) without specifying a branch, **commit straight to `main`** — do not branch first. The operator deliberately delegates commits to the agent so the work shows up credited in the log; keep using the standard `Co-Authored-By` trailer. Still only commit/push when explicitly asked, and push only on explicit request.

## Architecture

**Single Worker, one origin.** `src/index.ts` exports an `innerHandler` doing exact-match route dispatch, and the default export is that handler wrapped in `@cloudflare/workers-oauth-provider` (see `src/oauth.ts`). The OAuth wrap intercepts `/mcp` (its `apiRoute`), validates the token, populates `ctx.props.agentId`, and dispatches to the same inner handler as `apiHandler`. Everything else (including `/authorize` and the `awh_`-bearer path on `/mcp`) goes to `defaultHandler`. **The inner handler is registered as both `apiHandler` and `defaultHandler`** — the only difference between the two paths is whether `ctx.props.agentId` is populated.

**Two auth doors converge on one identity (`AwhProps` in `src/mcp-auth.ts`):**
- **Door A (OAuth 2.1 + PKCE)** — one pre-registered client per agent. Bound mint via `POST /admin/agents/:id/oauth-clients`; or **unbound** mint via `POST /admin/oauth-clients` (KV client record, no `oauth_clients` row) with the agent chosen at consent. Consent UI lives in `src/authorize.ts`; the operator approves with `OPERATOR_TOKEN` (or a browser session). Used by Cowork / claude.ai web / ChatGPT. `agentId` comes from the grant's `encryptedProps` (via `ctx.props`). **`/authorize` is operator-session-aware and offers three inline repairs** (all operator-gated, none disclose client state to non-operators): (1) **TOFU callback approval** — register an unregistered but allowlisted-host `redirect_uri` via `updateClient`; the allowlist is `APPROVABLE_CALLBACK_HOSTS` in `src/admin-oauth.ts`, the **single source of truth** also feeding the consent-page CSP `form-action` (widen in one place only — `validateCallbackUri` in `session.ts` is the pure gate). (2) **Bind-or-mint at consent** — an *unbound* client (no `oauth_clients` row; needs **no migration** — absence == unbound) gets a pick-existing-or-mint-new-agent card; "allow" writes the binding row, then `props.agentId` is **re-derived from the binding** (never trusted from a form) before `completeAuthorization`. (3) **Login-from-`/authorize`** — an unauthed requester gets a `/login?next=<this url>` link, keyed on the requester's own auth state so it discloses nothing. The POST auth ladder mirrors `handleRevokeForm` (pasted `operator_token` → synthetic Bearer/CSRF-exempt, else session cookie + `csrf_token` field) — it does **not** call `requireOperator` (that wants an `X-CSRF-Token` header a no-JS form can't send).
- **Door B (static `awh_` bearer)** — same key shape `POST /d` uses. Plumbed into the OAuth wrap as `resolveExternalToken` so a single configured `apiRoute` covers both transports. `agentId` comes from `authenticateAgent` (HMAC-under-pepper + `revoked_at` check on `agent_keys`).

**Operator browser session (`src/session.ts`) — a SECOND door onto the *operator* check, not a third agent door.** The two doors above authenticate *agents*; the operator is separate (`authenticateOperator` = constant-time compare vs `OPERATOR_TOKEN`). The session layer lets the operator log in once in a browser (`GET/POST /login`, `GET/POST /logout` in `src/login.ts`) instead of pasting the token on every action. It is HTTP/browser-only — **`/mcp` never consults these cookies** (the OAuth wrap owns it), and **no MCP tool/`/mcp` behavior changes**. It's a **stateless signed cookie**, no D1/KV/migration: `awh_session = base64url({v,iat,exp,csrf}) "." HMAC(payload, signingKey)`, where `signingKey = hmacSha256Hex("awh-session-signing/v"+EPOCH, OPERATOR_TOKEN)`. The signing key is derived from `OPERATOR_TOKEN` **on purpose** (not `HMAC_PEPPER`): the session authenticates the operator principal, so rotating the token must end every session — don't "fix" this to the pepper. Two invalidation levers, both of which change `signingKey`: bump `SESSION_EPOCH` (a `[var]`, the cheap "log everyone out" knob — defaults to `"1"` via `sessionEpoch(env)`) or rotate `OPERATOR_TOKEN`. `authenticateOperatorRequest` resolves **Bearer-first then cookie**; the shared async `requireOperator` (in `session.ts`, used by `admin.ts` / `admin-oauth.ts` / `DELETE /d/:id`) returns 401/403 and enforces CSRF (`403 csrf_failed`) only when authed **via cookie + unsafe method**. Bearer-authed requests are CSRF-exempt — so every programmatic/curl caller is unchanged.

**Shared write path: `src/core.ts`.** Both HTTP (`POST /d`, `PUT /d/:id` in `src/index.ts`) and MCP (`src/mcp.ts` tools) call `publishDocumentCore` / `updateDocumentCore` / `editDocumentCore` / `readDocumentCore` / `readDocumentTextCore` / `revokeDocumentCore` / `listDocumentsCore` / `findDocumentBySlugCore` / `searchDocumentsCore`. **Sanitization runs exactly once, inside core**, regardless of door. If you add a new write surface, route it through core — never duplicate the sanitize → cap-check → R2 → D1 sequence. `editDocumentCore` is the find/replace surface (MCP `edit_document` only — no HTTP `PATCH`): it does NOT re-implement the sequence — it loads the **retained source S** via `readDocumentSourceCore`, applies the string edits (pure logic in `src/edit.ts`), then DELEGATES the write to `updateDocumentCore` passing the doc's **own** `source_format` (no longer hardcoded `"html"` — getting this wrong feeds a Markdown source to the HTML identity pipeline and corrupts *silently*), so there's exactly one copy of the sanitize→cap→R2→D1→FTS batch. Match is against the STORED **source** (Markdown for md docs, HTML for html docs — so a Markdown doc edits its Markdown and the reader theme survives by construction); the response adds a `replacements` count so an agent can tell "patch landed" from "sanitizer normalized." A source-unavailable doc (un-backfilled, `source_r2_key IS NULL`) hard-fails with `source_unavailable` — no fallback to H. Listing-row reads (`listDocumentsCore`, `findDocumentBySlugCore`, and `searchDocumentsCore`) share `LISTING_SELECT_COLUMNS` / `LISTING_JOINS` constants so the projection stays in lockstep — when adding a column to `DocumentListing`, update those two constants once and all three surfaces follow.

**Text-read path runs at READ time, not write.** `readDocumentTextCore` fetches the sanitized HTML and pipes it through `htmlToMarkdown` (same WASM module as `sanitize`, separate exported function). No per-version markdown cache in v1 — the parse is single-digit ms for typical docs and is dwarfed by the R2 GET it shares. The markdown text-read derives from the sanitized **H** blob; conversion always runs on the *cleaned* bytes so the text channel can't surface anything the sanitizer stripped. **The retained source `.src` blob is a separate read surface, not part of this path:** `readDocumentSourceCore` fetches the unsanitized **S** (the bytes as submitted) and re-runs the advisory pass on it (markdown-or-identity → sanitize → `detectAdvisories`) so a source-read carries `unsanitized: true` plus `stripped[]`/`will_not_render[]` — "the live render differs from this source here" is surfaced, never silent. Source-read is **agent-key gated** (never operator-only, never public): in this single-tenant whole-fleet model any active agent key already reads and overwrites every doc, so source discloses no authority the caller lacks. Render path stays H-only and never serves S. **Exception:** the FTS5 search index in `documents_fts` is populated at WRITE time using the same `htmlToMarkdown` call — the FTS shadow tables are the sole storage of the body-as-text for search. The read endpoint still re-derives fresh from R2 on each call (no D1 dependency for the markdown view); the FTS write-time copy is a separate, write-only side channel that only exists to feed BM25 ranking. The two can drift by at most one converter-version step if the converter changes between an old write and a new read — acceptable, and the `converter_v` stamp on the read response signals when it's happened.

**Two security layers (in order):**
1. **Sandbox + strict CSP at render** (`src/serve.ts`). `GET /d/:id` returns a tiny HTML shell with `<iframe sandbox>` (empty = all restrictions on); `GET /d/:id/raw` is what the iframe loads under `default-src 'none'` + `frame-ancestors 'self'`. Two URLs is deliberate — `frame-ancestors` is header-only, so the bytes must come from an HTTP response, not `srcdoc`. **This is the load-bearing wall.**
2. **Ammonia-WASM sanitization at write** (`sanitizer/src/lib.rs` → `src/sanitizer.ts`). Strips `<script>`, `<style>`, `<meta http-equiv>`, `<iframe>`, dangerous URL schemes, inline event handlers. Cheap insurance behind the CSP wall (covers e.g. `<meta refresh>` which CSP can't).

**Storage model.** D1 (`META` binding) holds the metadata — see `migrations/0001_init.sql` for `agents`, `agent_keys`, `documents`, `versions`; `0002_oauth_clients.sql` for `oauth_clients` (client_id ↔ agent_id join); `0003_source_format.sql` for `versions.source_format`; `0004_document_metadata.sql` for `versions.title` / `description` / `tags` (JSON-encoded array; NULL when unset — but `tags` was later **lifted to `documents.tags` and dropped from `versions` in 0012**); `0005_document_slug.sql` for `documents.slug` + partial UNIQUE INDEX where slug IS NOT NULL; `0006_documents_search.sql` for the `documents_fts` FTS5 virtual table (one row per live document, columns `document_id UNINDEXED, title, description, tags, body` with porter+unicode61+remove_diacritics tokenizer — **the `tags` column was dropped in 0012**, leaving `document_id, title, description, body`); `0007_key_expiry.sql` for `agent_keys.expires_at` (NULL = never; non-NULL only on the short-lived publish credentials minted by `create_publish_credential`); `0008_source_retention.sql` for `versions.source_r2_key` + `versions.source_size_bytes` (both nullable, no default — NULL = un-backfilled/legacy, the loud presence-flag `editDocumentCore`/`readDocumentSourceCore` hard-fail on with `source_unavailable`, never a legacy fallback); `0009_slug_tombstones.sql` for the `slug_tombstones` table (slug PK, document_id, retired_at, reason) that makes a claimed slug semi-permanent — reversing 0005's "released on revoke" (GitHub issue #6); `0010_slug_redirect.sql` adds `slug_tombstones.redirect_to` (target `public_id`) for the loud operator/rename redirect; and `0011_document_visibility.sql` adds `documents.visibility` (`public`|`private`, CHECK-pinned — column `DEFAULT 'public'` backfills legacy rows so already-shared docs keep serving, but new docs are born EXPLICITLY at `DEFAULT_DOCUMENT_VISIBILITY` ([var], default `"private"`), so the column default only ever covers pre-migration rows; GitHub issue #7); and `0012_document_tags.sql` **lifts `tags` from per-version to document-level** — adds `documents.tags` (backfilled from each doc's current version), `DROP`s `versions.tags`, and rebuilds `documents_fts` WITHOUT its `tags` column (the body text is carried through a temp table since it's WASM-derived and lives only in the FTS index). Tags join `slug`/`visibility` on the identity/classification side: they survive content updates and restores, a tag change never bumps a version, and they're no longer FTS-indexed (the `?tags=` filter is a LIKE on `documents.tags`, never FTS). R2 (`DOCS` binding) holds **two** blobs per version: the sanitized render **H** at key `<docId>/v<n>` and the retained source **S** at the dot-suffix sibling `<docId>/v<n>.src` (content-type from `source_format`). Source bytes count toward the fleet storage cap (`size_bytes` stays "rendered H bytes"; `source_size_bytes` is the separate per-version S size — `checkStorageCap` SUMs both, coalescing NULL legacy rows to 0). `versions` rows are append-only and survive revoke as an audit trail (only the R2 bytes are purged — **both** H and the `.src` sibling). KV (`OAUTH_KV` binding) is owned by the OAuth provider library (clients, grants, tokens). On revoke, `documents.revoked_at` is flipped **before** the R2 batch delete so the doc is unreachable instantly even if R2 cleanup hangs; the same batch clears `documents.slug`, **retires that slug into `slug_tombstones`** (so the name is reserved forever — migration 0009, not freed), and deletes the FTS row, so kill + slug-retire + search visibility all land atomically.

**Single-tenant trust model.** Any active agent key under one operator can `PUT` to any document — `core.ts` deliberately does not scope updates by `created_by`. `documents.created_by` retains the original creator; new version metadata records the writer's `agent_id` only in R2 `customMetadata`. `listDocumentsCore` shows the whole fleet to every caller. If per-agent scoping ever becomes needed, add a `createdBy?` arg to core functions. **Unsanitized source bytes are now also at rest in R2** (the `.src` blob per version, since migration 0008): it is auth-gated (agent-key) and **never served to a browser** — the render path is still H-only behind the iframe+CSP wall. Source retention changes only the *ingestion-as-context* surface (an agent asking for `representation:"source"` can pull markup-form bytes the sanitizer would have neutralized on the render path); the provenance tag + agent-key auth cover exactly that increment, and revoke purges the `.src` siblings alongside H so the kill switch leaves no unsanitized source resident.

**Per-document visibility — a two-axis read model (migration 0011, GitHub issue #7).** Orthogonal to revoke: every document is `public` or `private`, and `src/access.ts` is the single pure chokepoint. `canRead(principal, doc)` is top-down first-match-wins (revoked→false, operator→true, agent→true, anonymous→`visibility === "public"`); `resolvePrincipal` resolves operator → agent → anonymous from the request. **Visibility gates ONLY the anonymous browser surface** — `serveShell` / `serveRaw` / `serveBySlug` / `serveHomepage`, all enforced at the **`serveRaw` chokepoint** (`resolvePrincipal` → `canRead`). The operator (cookie session OR bearer) and any active agent key read everything, so the agent-key-gated surfaces (`/text`, `/source`, MCP read/list/search) are unchanged, and `listDocumentsCore` still shows the whole fleet to every authenticated caller. A private doc denies as the **same opaque 404** a missing/revoked doc gives — **never a 401, no existence oracle**. New docs are born at `DEFAULT_DOCUMENT_VISIBILITY` ([var] in `env.ts`/`wrangler.toml`, default `"private"`), bound EXPLICITLY by the write path so the column's own `DEFAULT 'public'` only ever covers legacy rows; the operator flips it via `POST /admin/documents/:id/visibility` (JSON) or the Manage page (`setDocumentVisibilityCore`, no version bump). Visibility-CHANGE authority is a separate operator check (`requireOperator`), deliberately kept OUT of `canRead` (which governs read on the public surface and nothing else).

**Sanitizer is bundled as WASM.** `src/sanitizer.ts` imports `sanitizer/pkg/sanitizer_bg.wasm` directly as a `WebAssembly.Module` (wrangler's `[[rules]] type = "CompiledWasm"` rule) and calls `initSync`. **Do not use the wasm-pack `--target web` glue's `init()`** — it tries to fetch via `import.meta.url`, which doesn't exist on Workers. Init is lazy and idempotent. `skills/publishing.md` is bundled the same way via the `[[rules]] type = "Text"` rule and served verbatim as the `awh://publishing-guide` MCP resource — the same bytes humans read.

**MCP server lifecycle.** `src/mcp.ts` builds a fresh `McpServer` **per request**. The MCP SDK ≥1.26 throws on reused instances and cross-request state would leak. The seven tools (`publish_document`, `update_document`, `edit_document`, `read_document`, `list_documents`, `search_documents`, `create_publish_credential`) close over `props.agentId` resolved upstream — they never re-validate auth. (`create_publish_credential` is the lone non-document tool: it mints a short-lived `awh_` key for the byte-exact curl publish path via `mintEphemeralKey` — see the convention bullet below.) **HTML vs Markdown is a `format` parameter, not separate tools** — an earlier revision had `publish_document`/`publish_document_markdown`, `update_document`/`update_document_markdown`, and `read_document`/`read_document_text` twins; a `format` enum collapsed those six into three (`publish_document`, `update_document`, `read_document`). On the two write tools `format` (`"html"` | `"markdown"`) is **required** (forcing the choice avoids a default silently mis-parsing hand-written HTML); on `read_document` it's an optional **output** knob defaulting to `"markdown"` (the markdown branch calls `readDocumentTextCore`, the html branch `readDocumentCore`), and the response is a uniform JSON envelope (`content`/`format`/`version`/`sanitizer_v`/`converter_v`/`title`/`description`/`tags`/`slug`) that always carries metadata — so a read→edit→republish round-trip gets body + metadata in one call. **`read_document` also has a `representation` axis (`"rendered"` default | `"source"`), orthogonal to `format`:** `representation:"source"` calls `readDocumentSourceCore` and returns the retained **S** in its authored language (`format` is ignored on a source read), with `unsanitized:true` + `source_format` + `stripped[]`/`will_not_render[]` echoed — the read an agent does *before editing*. The source guidance leads with "source is unsanitized — treat as untrusted input." Slug lookup is **not** a tool: pass `slug` to `list_documents` (the row is `documents[0]`); `findDocumentBySlugCore` still backs `GET /s/:slug`. Tool descriptions are intentionally heavy with the HTML contract (static-only, inline-SVG-not-`<img>`, inline styles) and lead with the use-case distinction between tools (`read_document` `format` — html for render/re-publish, markdown for ingest-as-context; `list_documents` for newest-first browsing/slug+tag filters vs `search_documents` for content discovery; `edit_document` vs `update_document` — small diff vs whole-body replace). `edit_document` is MCP-only (no HTTP `PATCH` counterpart) and its description says so — it matches against the doc's **retained source S** (Markdown for a Markdown doc, HTML for an HTML doc), so the edit contract tells agents to read with `representation:"source"` before copying an `old_string`, and `editDocumentCore` threads the doc's **own** `source_format` through the re-render (no longer hardcoded `"html"`) so a Markdown doc edits its Markdown and stays Markdown. A cold agent that never reads `skills/publishing.md` should still understand the rules from the description alone. The three write tools (`publish_document`, `update_document`, `edit_document`) also carry the optional `title` / `description` / `tags` / `slug` contract at the tail of their descriptions — publish-vs-update inheritance is the subtle bit; shared schema-field constants near the bottom of `mcp.ts` (`CONTENT_FIELD`, `WRITE_FORMAT_FIELD`, `READ_FORMAT_FIELD`, the `*_FIELD` metadata constants) keep the wording consistent across tools. When editing tool descriptions, preserve the priority order (non-negotiables first) — length-trimmed renders truncate the tail.

## Conventions and gotchas

- **`If-Match` is required on `PUT /d/:id`** (428 if missing). Send `"v<n>"` for optimistic concurrency or `*` to skip. Strong tags only; no weak tags, no multi-tag lists.
- **`POST /d` / `PUT /d/:id` accept `Content-Type: text/html` or `text/markdown`** (`text/x-markdown` too; anything else → 415; the gate is `parseInputFormat` in `src/index.ts`). A Markdown body is parsed (CommonMark + GFM) to HTML before sanitizing; either way the stored bytes are sanitized `text/html; charset=utf-8`, and which input path produced the version is recorded in `versions.source_format`.
- **Optional `X-Content-SHA256` byte-exact integrity check on `POST /d` / `PUT /d/:id`** (`src/integrity.ts`, wired via `readVerifiedBody` in `src/index.ts`). Hashes the **raw received body before sanitization** — verifies the wire, not the sanitizer's transformation, so it's orthogonal to the `modified` flag. Malformed header → 400 `bad_integrity_header`; mismatch → 422 `integrity_mismatch`. **HTTP-only by design** (the companion to the `curl --data-binary @file` byte-exact publish path): the hash must come from the same deterministic tool that streams the file, so there is no MCP equivalent — the two primary MCP write-tool descriptions say so. `byte-exact-publish-design.md` records the rationale (the binding constraint at 66–85 KB is model regeneration of the `content` argument, not transport; chunked-append doesn't fix it).
- **Short-lived publish credentials (`agent_keys.expires_at`, migration 0007).** The byte-exact curl path needs a bearer at the shell, but Claude's connector settings can't store one and the connector's OAuth access token isn't exposed to the model/shell. So the already-OAuth-authenticated MCP session mints one on demand: the `create_publish_credential` tool → `mintEphemeralKey` (`src/admin.ts`) inserts a normal `awh_` key with a non-NULL `expires_at` (default 15 min, max 60, clamped). `authenticateAgent` (`src/auth.ts`) rejects expired keys exactly like revoked ones via the pure `isKeyExpired` helper. `expires_at = NULL` = never expires (every operator-minted key + legacy rows), so the migration is a no-op for them. **No scope column in v1** — single-tenant, the damaging vector (overwriting docs) isn't blocked by a write-only scope anyway, and the minted key grants nothing beyond the MCP session's existing authority; scope-limiting is the additive multi-tenant follow-up. The minted key/recipe is a deliberate disclosure surface — never log it (same discipline as the OAuth token). Revoke early via `DELETE /admin/keys/:id`.
- **Optional metadata on POST/PUT /d**: `X-Doc-Title`, `X-Doc-Description`, `X-Doc-Tags` (comma-separated), `X-Doc-Slug`. MCP write tools take the same shape as schema fields. On UPDATE, an *omitted* field inherits the prior version's value (or, for slug, the current document's value); an explicit `""` clears (and for title, re-derives from the new content's H1; for slug, releases it). Tags charset is restricted to `[A-Za-z0-9_-]` — invalid chars are silently stripped, not rejected. Slug charset is `/^[a-z0-9](?:[a-z0-9_-]{0,62}[a-z0-9])?$/` — invalid input is **rejected** (`invalid_slug` → 422), not silently sanitized (uniqueness means mutating the input could surprise-collide). Slug is unique across live docs (partial UNIQUE INDEX), conflict → `slug_taken` → 409. The render path applies anti-phishing normalization to title only (bidi/zero-width strip); description stores bidi for fidelity since it doesn't reach the browser tab.
- **Slug AND tags live on `documents`, not `versions`.** Title/description evolve per-version because they describe content; **slug and tags are document-level** (tags since migration 0012) — slug is identity-adjacent (uniqueness per-document), tags are classification (a document's place in the collection, not a revision's wording). Both survive content updates and restores and never bump a version when changed. It survives version increments unchanged unless the agent explicitly changes it (atomic claim-new + retire-old in the same D1 batch). On revoke (and on rename/clear) `documents.slug` is set NULL alongside `revoked_at`, but the slug is **NOT reusable** — it is retired into `slug_tombstones` (migration 0009) and reserved forever; `/s/<slug>` then returns **410 Gone** and a reclaim attempt → `slug_retired`. `documents.slug` therefore always means "the live slug"; the permanent reservation is the separate tombstone table (a claim checks both via `resolveSlug`; `serveBySlug`/`read_document` consult `findSlugTombstoneCore` on a live-lookup miss to choose 410-vs-404). This reverses the original "released on revoke" contract per GitHub issue #6 — slug reuse is surprising (a shared/linked `/s/<slug>` must not silently start serving unrelated content). Pulling slug from `d.slug` in the joined read/list queries — not from prior_metadata — keeps the per-version `resolveMetadata` helper unaware of cross-document state. **A retired slug can REDIRECT (migration 0010, `slug_tombstones.redirect_to` = target `public_id`), but only LOUDLY:** a rename auto-forwards the old slug to the same doc's `public_id`; the operator can cross-redirect via `POST /admin/slugs/:slug/redirect`; `serveBySlug`/`read_document` never auto-follow — browser gets a click-through interstitial, agent gets `409 slug_redirected` (HTTP) / a `{redirected:true}` envelope (MCP) and must opt in with `follow_redirects`. Single-hop (target is a `public_id`, so no chains/loops; a dead target falls back to 410). Operator escape hatch `DELETE /admin/slugs/:slug` force-releases a tombstone (the only un-retire path); `DELETE /admin/slugs/:slug/redirect` drops just the redirect.
- **Operator document-management page + browser login-link 404 (`src/serve.ts`).** `GET /d/:id/manage` (`serveManagePage`) is the cookie-session-gated operator page folding three actions onto one doc: visibility toggle (`handleVisibilityForm`, `POST /d/:id/visibility`), slug add/rename/clear (`handleSlugForm`, `POST /d/:id/slug`), and revoke. A logged-out caller gets a sign-in card rendered with **no DB hit** (so the page is no existence oracle). The shared `authorizeOperatorForm` mirrors `handleRevokeForm`'s ladder (pasted `operator_token` OR cookie + `csrf_token`). The shell topbar's old "Revoke…" kebab item is now **"Manage…"** (revoke folded into the manage page); a signed-in topbar also shows a Public/Private badge. JSON operator twins live in `admin.ts`: `setDocumentVisibility` (`POST /admin/documents/:id/visibility`) and `setDocumentSlug` (`POST /admin/documents/:id/slug`); the slug path reuses `setDocumentSlugCore` (no version bump; `resolveSlug` + `tombstoneSlug` so a rename auto-forwards — identical to the agentic `update_document` slug change). **Browser login-link 404:** because private docs exist, a valid `/d/:id` or `/s/:slug` URL 404s to a logged-out operator — so `notFoundBrowser` (+ `NOTFOUND_CSP`) returns an HTML 404 card with a "Sign in" link that round-trips back via `/login?next=<requested URL>`. It **never reads doc state** (only echoes the requested path into `next`), so a private doc's 404 is byte-identical to a nonexistent one's for the same URL. Wired into `serveShell` (all three 404s), `serveBySlug` (private-to-anon; malformed + never-claimed-slug pick HTML-vs-plain by the `Authorization` header), and `serveRetiredSlug`'s no-tombstone case. Agent/API 404s (`/raw`, `/text`, `/source`, anything carrying `Authorization`, the `/manage` POST handlers) keep the plain `text/plain` `notFound()` body.
- **Operator version history + restore-as-new-version (`src/serve.ts`, `src/core.ts`, MCP).** No migration — `versions` already retains every version's H (and `.src`) bytes until revoke purges them, so "history" is pure additive read code. Operator browser surface: `GET /d/:id/v/:n` (shell) + `/d/:id/v/:n/raw` (bytes), `authenticateOperatorRequest`-gated (Bearer OR cookie), and a Version history table on the manage page with a Restore button per non-current version (`POST /d/:id/restore`). This is an **operator** axis, deliberately NOT folded into `canRead`/visibility — a public doc's history is as operator-only as a private one's; a non-operator gets the same opaque 404. Agents read history through MCP `read_document` (`version` to read one, `include_history` for the manifest) — **read-only; there is no agent restore in v1.** **Restore is mandatorily restore-as-NEW-version, never a `current_ver` rewind:** `updateDocumentCore` computes `nextVer = current_ver + 1`, so pointing `current_ver` backward would make the next ordinary update collide on the `(document_id, version_no)` PK. `restoreVersionCore` reads version N's retained source S and re-publishes it forward through `updateDocumentCore`, restoring body + title/description/tags but keeping the doc's current slug (slug is identity, not content). A pre-0008 version with no retained source **hard-fails `source_unavailable`** — there is deliberately NO fall-back-to-H legacy branch (identical to `edit_document`'s contract); the manage table shows a muted "no source" in place of the Restore button, and such docs are revoke-and-republished (the pre-launch no-legacy-code stance). The `test/`-style suite has no D1 harness, so this path is covered by typecheck + the local `wrangler dev` E2E (publish → update×2 → restore v1 → confirm the next update lands monotonically, no collision).
- **`public_id` regex is fixed**: `/^[A-Za-z0-9_-]{22}$/` (`PUBLIC_ID_RE` in `src/serve.ts`). It's safe to interpolate into HTML templates after this check; `serveShell` does so.
- **Storage cap is best-effort.** `checkStorageCap` runs `SUM(size_bytes)` outside the insert batch; two concurrent writes can both pass. v1 accepts the slight overrun.
- **List endpoints are cursor-paginated** (`src/pagination.ts`). The four list surfaces — `GET /admin/agents`, `GET /admin/agents/:id/keys`, `GET /admin/documents`, and the MCP `list_documents` tool — share `parseHttpListParams` / `parseMcpListArgs` + the `paginate()` peek helper. Default page size 50, max 200. Ordering is `(created_at DESC, id DESC)` — the `id` tiebreaker is mandatory because D1's strftime stamps to ms and collisions are rare-but-possible; without it cursors could skip a row at a page boundary. Cursors are opaque base64url(JSON `{ts, id}`) — never construct them by hand. When you add a new list endpoint, route it through `pagination.ts` rather than introducing a new style.
- **Search is NOT cursor-paginated.** `GET /admin/documents/search` and the MCP `search_documents` tool both reuse the filter parser (`parseHttpListParams` / `parseMcpListArgs`) for `tags` / `slug` / `limit`, but ignore the cursor field — BM25 rank is not a stable cursor key the way `(created_at, id)` is. Results are capped at `limit` (default 50, max 200) and the response has no `next_cursor`. If the top N doesn't include what the caller wants, the answer is to refine the query, not paginate further. Query input is sanitized through `buildFtsMatchQuery` in `src/search.ts` — empty result → `bad_query` (422); the FTS MATCH never receives raw user input.
- **Document filtering** (`tags`, `slug`) lives on the document list/search surfaces — the parser collects them for every endpoint that uses `ParsedListParams`, but only `listDocumentsCore` and `searchDocumentsCore` consult them in SQL (the agents/keys lists silently ignore the fields). `tags` is AND semantics: one `tags LIKE ? ESCAPE '\\'` per requested tag against the JSON-encoded `documents.tags` column (document-level since migration 0012 — the filter never used FTS). The underscore in the tag charset collides with LIKE's `_` wildcard, so `tagLikePattern()` in `core.ts` escapes it with `\_`. Tag inputs are silently sanitized to the stored shape at parse time (matching write-time semantics); slug inputs are validated and reject with `bad_slug` on invalid charset (matching the write-time reject-not-sanitize rule). `findDocumentBySlugCore` is the dedicated single-row lookup behind the `GET /s/:slug` HTTP redirect (it validates the slug via `validateSlugInput` upstream of the DB hit). There is no longer a `find_document_by_slug` MCP tool — the slug-lookup path on MCP is `list_documents` with a `slug` filter (returns 0 or 1 rows; the caller reads `documents[0]`).
- **FTS sync is write-path-local.** No D1 triggers — `publishDocumentCore` / `updateDocumentCore` / `revokeDocumentCore` each include the `documents_fts` INSERT/UPDATE/DELETE in their existing `META.batch()` so the FTS row tracks the document's current version atomically. The body column is the markdown derivation of the sanitized HTML, computed once per write via `htmlToMarkdown(prep.cleanedHtml)`. Since migration 0012 the FTS row indexes only `title`/`description`/`body` — **tags are NOT FTS-indexed** (the `setDocumentTagsCore` mutator and the `?tags=` LIKE filter both touch `documents.tags` directly, so a tag-only write touches no FTS row). The FTS row is the SOLE storage of body-as-text — no `body_text` column on `versions`. If you add a new write surface, mirror the FTS sync inside the batch (or extract a helper) — don't ship a write path that bypasses FTS, or search will silently lag.
- **Operator session cookies + CSRF (`src/session.ts`).** Two cookies, both **host-only** (no `Domain`) and `SameSite=Lax`: `awh_session` (HttpOnly — the signed credential) and `awh_csrf` (readable — delivers the CSRF nonce to the page). `Secure` is **conditional on request protocol** (`isSecureRequest`): set behind https custom domains, omitted over `http://localhost` (a blanket `Secure` is silently rejected on http, breaking `wrangler dev`). CSRF is **stateless signed double-submit**: the nonce lives inside the *signed* session payload, and a cookie-authed mutating request must echo it (`X-CSRF-Token` header for JSON/admin; `csrf_token` form field for the HTML `/login`, `/logout`, `/d/:id/revoke` forms) — the server constant-time-compares the echo to the verified session's nonce, so a cookie-vs-cookie injection can't forge it. The pure crypto/cookie core takes explicit `nowMs`/`operatorToken`/`epoch` (no `Date.now()`, no env) so it's unit-tested in `test/session.test.mjs`; only the thin env wrappers read `env`. Login/logout/revoke set **two** Set-Cookie headers via `headers.append` (not `.set`, which clobbers). `next` redirects are origin-whitelisted via `validateNext` (rejects `//evil.com`, `/\evil.com`, absolute URLs, CRLF). Never log the operator token or the derived signing key (same discipline as the OAuth token / minted keys).
- **Logging discipline in MCP tools** (`src/mcp.ts`): log tool name + error code only. Never args (may contain user HTML), never request headers (may contain the bearer), never the OAuth token.
- **Any API-surface change must update the MCP tool descriptions AND the HTTP API reference in the same commit.** New inputs, new response fields, new headers, changed status codes, changed inheritance/derivation semantics — all of these are part of the contract a cold MCP agent sees only via tool descriptions (the client may never expose `awh://publishing-guide`), and the contract a consumer building against HTTP (e.g. the Flutter app) sees only via `docs/http-api.md`. Walk the seven tools in `src/mcp.ts` and confirm: (1) every inputSchema field for the changed surface exists with an up-to-date `.describe()`; (2) the prose description mentions the new field/behavior at the priority level it deserves (non-negotiables first, optional bits at the tail — length-trimmed renders truncate); (3) the JSON response shape documented in the description matches what the handler actually returns; (4) the bundled `skills/publishing.md` covers the same change so the resource and the descriptions don't drift; (5) **`docs/http-api.md` reflects the change** — the affected endpoint's request/response/headers/status-code table, plus any `DocumentListing`/`SearchHit` shape edits in its "Shared response shapes" section. An HTTP-only surface (no MCP equivalent) still belongs in `docs/http-api.md`; an MCP-only surface (e.g. `edit_document`, `create_publish_credential`) is noted as such in `docs/http-api.md`'s MCP section. If a change is HTTP-only by design, say so explicitly in the MCP tool description so an agent on MCP knows not to look for it. (6) **`docs/http-api.md` is also published live on Slopcafe** (slug `slopcafe-http-api`, public_id `0EtsEq6cnCeuOhBKO6ICzA`) so any connected agent can read it without repo access — re-publish that copy when the doc changes (byte-exact `PUT` of the file via `create_publish_credential` + `curl --data-binary`, **not** MCP `update_document` — regenerating a ~28 KB body as a tool argument is slow and truncation-prone; `docs/README.md` records the exact recipe), or it silently drifts.
- **Code that deviates from or builds upon the SOLO spec must update `agent-knowledge-host-spec-SOLO-v1.md` in the same commit.** The spec is the *conceptual* contract — how an external agent with **no repo access** (a Claude reasoning about this app in general) understands the principals, the `public_id`/`slug` document model, the capability-URL security boundary, the sanitize-and-serve pipeline, and the deliberate v1 omissions. It is distinct from the *wire* contract above (`docs/http-api.md` + MCP tool descriptions): a change can leave the wire untouched yet still move the model the spec describes — a new trust boundary, a changed identity-binding rule, or a "deferred" seam that got built. The spec deliberately mixes **as-built** statements with **aspirational** ones (marked *deferred* / *reserved* / *not built*): when reality moves, fix the matching as-built sentence; when you build something the spec filed as deferred, promote it from the aspirational column to as-built. `agent-knowledge-host-spec-PLATFORM-v2.md` is the **frozen** multi-tenant "if this becomes a service" blueprint the SOLO spec forked from (see its §0 lineage note) — do **not** edit it for SOLO-scope changes; touch it only to keep the lineage auditable if a SOLO change diverges from what it documents. Both specs are **published live on Slopcafe** (SOLO slug `slopcafe-spec-solo`, public_id `ClcgZMaOEcworHzhr17gVQ`; PLATFORM slug `slopcafe-spec-platform`, public_id `Wo2Lm4pKlLEOqbhg7X4VQw`) so a connected agent can read them without repo access — re-publish the changed copy in the same change (byte-exact `PUT` via `create_publish_credential` + `curl --data-binary`, same recipe + rationale as `docs/http-api.md`), or it silently drifts.
- **Keep this `CLAUDE.md` current too.** It isn't a strict wire/spec sync target like `docs/http-api.md` or the SOLO spec, but future agents read it FIRST to orient — so treat it as auto-updated alongside those docs: when a change is noteworthy (a new migration, a new `src/` module, a new route or trust boundary, a changed convention/gotcha), update the matching part in the **same commit** — the relevant **Where things live** bullet, the **Storage model** migration list, and an Architecture paragraph or **Conventions and gotchas** bullet when the change moves the model. Stale guidance here misleads silently and costs the next agent more than the minute it takes to fix.
- **Adding a sanitizer test:** the ~40 negative assertions live inline at the bottom of `sanitizer/src/lib.rs`. Each tweak to `make_builder()` should add a case there.
- **Bypass corpus (`sanitizer/tests/bypass_corpus.rs` + `sanitizer/tests/corpus/`).** A separate, data-driven regression net distinct from the inline `lib.rs` cases: it runs `sanitize()` (and the `markdown_to_html → sanitize` ingress) over a curated **long-tail** of known sanitizer-bypass vectors (exotic event handlers, `javascript:`/`data:` obfuscation, the SVG `href`/`xlink:href` surface we add, mXSS / namespace confusion, markup-mutation breakouts) and asserts no **script-execution sink** survives. The check is **structural** — it re-parses the cleaned output with html5ever and inspects element names / `on*` attrs / URL-attr schemes, so a bare fragment that sanitizes to inert *text* doesn't false-positive (a substring scan would). Deliberately **out of scope** (documented in the test header): inline-`style` CSS (ammonia allowlists `style` but doesn't parse CSS — neutralized by the render wall's `default-src 'none'` CSP) and ARIA-tree hijack (pinned by the inline `lib.rs` aria tests). Auto-discovered by `cargo test`, so it's already inside `npm run test:sanitizer` — no wiring. Two guards travel with it: `predicate_self_check` (proves the detector fires on real sinks and stays silent on benign output, so the corpus can't pass vacuously) and `known_exceptions.txt` (exact-payload quarantine of reviewed-safe survivors, each with a `>>> reason:`). The `.txt` format reserves only the `>>>` prefix, so vectors paste in verbatim; the re-sync loop + provenance + licensing live in `sanitizer/tests/corpus/SOURCES.md`. Sized to probe **our** allowlist (unknown-unknowns in `make_builder()`) + act as an **ammonia-upgrade tripwire**, NOT to mirror an upstream dump — bulk generic vectors mostly re-test what ammonia already guarantees. Triage a survivor either by fixing `make_builder()` (+ bump `sanitizer_version()`, keep `skills/publishing.md` in lockstep) or by quarantining it with a written reason.
- **Ammonia's defaults silently omit several tags/attributes our contract allows.** The default tag set ships `article`/`aside`/`header`/`footer`/`nav`/`hgroup`/`figure`/`figcaption` but NOT `<section>` or `<tfoot>`, and its generic-attribute set is only `lang`/`title` (no `dir`; lists get only `start` on `<ol>`). Don't assume a tag/attribute is allowed just because a sibling is — the allowlist is `ammonia defaults ∪ make_builder() additions`, nothing more. `make_builder()` re-adds the gaps the publishing contract names: `<section>`/`<tfoot>` via `EXTRA_TAGS`, `dir` via `HTML_GENERIC_ATTRS`, and `reversed`/`value`/`type` per list element. `<main>`/`<address>` stay stripped by design (documented as such in `skills/publishing.md`). The `contract_*` corpus tests in `lib.rs` pin the entire documented surface (every allowed tag/attr survives, structural wrappers unwrap, `<main>`/`<address>` strip) so an ammonia upgrade or allowlist edit that drops a documented item fails loudly. Keep them in lockstep with `skills/publishing.md`.
- **Adding an advisory:** `src/advisories.ts` is mostly regex-based on (input, cleaned) pairs — false negatives are acceptable, false positives are not. Behind the specific regex rules sits a generic element-diff (`detectStrippedElements`) that reports any open-tag the output lost and no specific rule named — the catch-all that surfaces an allowlist gap nobody wrote a rule for (how `<section>` vanished silently). Its skip sets (`SPECIFICALLY_COVERED`, `UNWRAPPED_OK`, `CONTEXT_SENSITIVE_OK`) keep it false-positive-free; the entity-encoded-script case plus the generic/wrapper/comment/script-body cases in `test/advisories.test.mjs` are the guards.
- **Wrangler version + Workers types** are pinned in `package.json`. `compatibility_date = "2026-05-26"` with `nodejs_compat`.
- **Secrets** (`HMAC_PEPPER`, `OPERATOR_TOKEN`) live in `.dev.vars` for local dev (gitignored) and `wrangler secret put` for production — never `[vars]` in `wrangler.toml`. `SESSION_EPOCH` is the exception: a non-secret rotation counter, so it lives in `[vars]` (default `"1"`); bump it to invalidate all browser sessions.

## Where things live

- `src/index.ts` — route dispatch + thin HTTP wrappers for `/d` and `/d/:id`. Default export wraps `innerHandler` with the OAuth provider.
- `src/contract.ts` — **the single source of truth for the API contract** (Phase 1 of `api-contract-design.md`). A leaf module (only runtime dep: `zod`; the `import type` of `access`/`metadata` is erased, so it's standalone-testable like `search.ts`/`edit.ts`) that declares every response/data shape as a Zod schema — `DocumentListing` / `SearchHit` / `WriteOk` / `EditOk` / `RestoreOk` / `ReadOk` / `ReadTextOk` / `ReadSourceOk` / `VersionListing` / `ListVersionsOk` / `RevokeOk` / `RedirectTarget` / `SlugTombstone` — all **re-exported from `core.ts` as `z.infer<>`** so every existing `import { … } from "./core.js"` is unchanged.
