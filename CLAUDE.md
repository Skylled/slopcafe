# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Build / deploy / dev:
```sh
npm run dev            # wrangler dev — uses .dev.vars + local D1/R2
npm run typecheck      # tsc --noEmit
npm run build:wasm     # rebuild the Rust→WASM sanitizer (auto-run via predeploy)
npm run deploy         # rebuilds WASM then `wrangler deploy`
```

Tests:
```sh
npm test                  # sanitizer corpus + advisories + metadata + pagination + search + edit + integrity + auth
npm run test:sanitizer    # cargo test in /sanitizer (host target, not wasm)
npm run test:advisories   # node --experimental-strip-types test/advisories.test.mjs
npm run test:metadata     # node --experimental-strip-types test/metadata.test.mjs
npm run test:pagination   # cross-file ts import → uses test/register-ts-resolver.mjs
npm run test:search       # FTS5 query tokenizer (buildFtsMatchQuery) — pure function tests
npm run test:edit         # applyEdits find/replace logic (edit_document) — pure function tests

# Run a single sanitizer test (Rust name match):
(cd sanitizer && PATH="$HOME/.cargo/bin:$PATH" cargo test rejects_script_tag)
```

Database (D1):
```sh
npm run db:migrate:local            # apply migrations to local D1
npm run db:migrate:remote           # apply migrations to remote D1
npm run db:console:local  "SELECT * FROM agents"
npm run db:console:remote "SELECT * FROM agents"
```

`predeploy` injects `$HOME/.cargo/bin` into PATH so `wasm-pack` resolves from a fresh shell. Rust toolchain (`rustup` + `wasm32-unknown-unknown` target + `wasm-pack`) is only required for `build:wasm`.

## Git workflow

When the operator says "commit your work" (or similar) without specifying a branch, **commit straight to `main`** — do not branch first. The operator deliberately delegates commits to the agent so the work shows up credited in the log; keep using the standard `Co-Authored-By` trailer. Still only commit/push when explicitly asked, and push only on explicit request.

## Architecture

**Single Worker, one origin.** `src/index.ts` exports an `innerHandler` doing exact-match route dispatch, and the default export is that handler wrapped in `@cloudflare/workers-oauth-provider` (see `src/oauth.ts`). The OAuth wrap intercepts `/mcp` (its `apiRoute`), validates the token, populates `ctx.props.agentId`, and dispatches to the same inner handler as `apiHandler`. Everything else (including `/authorize` and the `awh_`-bearer path on `/mcp`) goes to `defaultHandler`. **The inner handler is registered as both `apiHandler` and `defaultHandler`** — the only difference between the two paths is whether `ctx.props.agentId` is populated.

**Two auth doors converge on one identity (`AwhProps` in `src/mcp-auth.ts`):**
- **Door A (OAuth 2.1 + PKCE)** — one pre-registered client per agent, pinned at mint time via `POST /admin/agents/:id/oauth-clients`. Consent UI lives in `src/authorize.ts`; the operator approves with `OPERATOR_TOKEN`. Used by Cowork / claude.ai web. `agentId` comes from the grant's `encryptedProps` (via `ctx.props`).
- **Door B (static `awh_` bearer)** — same key shape `POST /d` uses. Plumbed into the OAuth wrap as `resolveExternalToken` so a single configured `apiRoute` covers both transports. `agentId` comes from `authenticateAgent` (HMAC-under-pepper + `revoked_at` check on `agent_keys`).

**Operator browser session (`src/session.ts`) — a SECOND door onto the *operator* check, not a third agent door.** The two doors above authenticate *agents*; the operator is separate (`authenticateOperator` = constant-time compare vs `OPERATOR_TOKEN`). The session layer lets the operator log in once in a browser (`GET/POST /login`, `GET/POST /logout` in `src/login.ts`) instead of pasting the token on every action. It is HTTP/browser-only — **`/mcp` never consults these cookies** (the OAuth wrap owns it), and **no MCP tool/`/mcp` behavior changes**. It's a **stateless signed cookie**, no D1/KV/migration: `awh_session = base64url({v,iat,exp,csrf}) "." HMAC(payload, signingKey)`, where `signingKey = hmacSha256Hex("awh-session-signing/v"+EPOCH, OPERATOR_TOKEN)`. The signing key is derived from `OPERATOR_TOKEN` **on purpose** (not `HMAC_PEPPER`): the session authenticates the operator principal, so rotating the token must end every session — don't "fix" this to the pepper. Two invalidation levers, both of which change `signingKey`: bump `SESSION_EPOCH` (a `[var]`, the cheap "log everyone out" knob — defaults to `"1"` via `sessionEpoch(env)`) or rotate `OPERATOR_TOKEN`. `authenticateOperatorRequest` resolves **Bearer-first then cookie**; the shared async `requireOperator` (in `session.ts`, used by `admin.ts` / `admin-oauth.ts` / `DELETE /d/:id`) returns 401/403 and enforces CSRF (`403 csrf_failed`) only when authed **via cookie + unsafe method**. Bearer-authed requests are CSRF-exempt — so every programmatic/curl caller is unchanged.

**Shared write path: `src/core.ts`.** Both HTTP (`POST /d`, `PUT /d/:id` in `src/index.ts`) and MCP (`src/mcp.ts` tools) call `publishDocumentCore` / `updateDocumentCore` / `editDocumentCore` / `readDocumentCore` / `readDocumentTextCore` / `revokeDocumentCore` / `listDocumentsCore` / `findDocumentBySlugCore` / `searchDocumentsCore`. **Sanitization runs exactly once, inside core**, regardless of door. If you add a new write surface, route it through core — never duplicate the sanitize → cap-check → R2 → D1 sequence. `editDocumentCore` is the find/replace surface (MCP `edit_document` only — no HTTP `PATCH`): it does NOT re-implement the sequence — it loads the stored bytes via `readDocumentCore`, applies the string edits (pure logic in `src/edit.ts`), then DELEGATES the write to `updateDocumentCore`, so there's exactly one copy of the sanitize→cap→R2→D1→FTS batch. Match is against the STORED sanitized bytes (mirroring Claude Code's `Edit` tool); the response adds a `replacements` count so an agent can tell "patch landed" from "sanitizer normalized." Listing-row reads (`listDocumentsCore`, `findDocumentBySlugCore`, and `searchDocumentsCore`) share `LISTING_SELECT_COLUMNS` / `LISTING_JOINS` constants so the projection stays in lockstep — when adding a column to `DocumentListing`, update those two constants once and all three surfaces follow.

**Text-read path runs at READ time, not write.** `readDocumentTextCore` fetches the sanitized HTML and pipes it through `htmlToMarkdown` (same WASM module as `sanitize`, separate exported function). No per-version markdown cache in v1 — the parse is single-digit ms for typical docs and is dwarfed by the R2 GET it shares. Stored R2 layout is unchanged (one HTML blob per version). Conversion always runs on the *cleaned* bytes so the text channel can't surface anything the sanitizer stripped. **Exception:** the FTS5 search index in `documents_fts` is populated at WRITE time using the same `htmlToMarkdown` call — the FTS shadow tables are the sole storage of the body-as-text for search. The read endpoint still re-derives fresh from R2 on each call (no D1 dependency for the markdown view); the FTS write-time copy is a separate, write-only side channel that only exists to feed BM25 ranking. The two can drift by at most one converter-version step if the converter changes between an old write and a new read — acceptable, and the `converter_v` stamp on the read response signals when it's happened.

**Two security layers (in order):**
1. **Sandbox + strict CSP at render** (`src/serve.ts`). `GET /d/:id` returns a tiny HTML shell with `<iframe sandbox>` (empty = all restrictions on); `GET /d/:id/raw` is what the iframe loads under `default-src 'none'` + `frame-ancestors 'self'`. Two URLs is deliberate — `frame-ancestors` is header-only, so the bytes must come from an HTTP response, not `srcdoc`. **This is the load-bearing wall.**
2. **Ammonia-WASM sanitization at write** (`sanitizer/src/lib.rs` → `src/sanitizer.ts`). Strips `<script>`, `<style>`, `<meta http-equiv>`, `<iframe>`, dangerous URL schemes, inline event handlers. Cheap insurance behind the CSP wall (covers e.g. `<meta refresh>` which CSP can't).

**Storage model.** D1 (`META` binding) holds the metadata — see `migrations/0001_init.sql` for `agents`, `agent_keys`, `documents`, `versions`; `0002_oauth_clients.sql` for `oauth_clients` (client_id ↔ agent_id join); `0003_source_format.sql` for `versions.source_format`; `0004_document_metadata.sql` for `versions.title` / `description` / `tags` (JSON-encoded array; NULL when unset); `0005_document_slug.sql` for `documents.slug` + partial UNIQUE INDEX where slug IS NOT NULL; `0006_documents_search.sql` for the `documents_fts` FTS5 virtual table (one row per live document, columns `document_id UNINDEXED, title, description, tags, body` with porter+unicode61+remove_diacritics tokenizer); and `0007_key_expiry.sql` for `agent_keys.expires_at` (NULL = never; non-NULL only on the short-lived publish credentials minted by `create_publish_credential`). R2 (`DOCS` binding) holds one sanitized blob per version at key `<docId>/v<n>`; `versions` rows are append-only and survive revoke as an audit trail (only the R2 bytes are purged). KV (`OAUTH_KV` binding) is owned by the OAuth provider library (clients, grants, tokens). On revoke, `documents.revoked_at` is flipped **before** the R2 batch delete so the doc is unreachable instantly even if R2 cleanup hangs; the same batch clears `documents.slug` and deletes the FTS row so slug + search visibility all release atomically with the kill.

**Single-tenant trust model.** Any active agent key under one operator can `PUT` to any document — `core.ts` deliberately does not scope updates by `created_by`. `documents.created_by` retains the original creator; new version metadata records the writer's `agent_id` only in R2 `customMetadata`. `listDocumentsCore` shows the whole fleet to every caller. If per-agent scoping ever becomes needed, add a `createdBy?` arg to core functions.

**Sanitizer is bundled as WASM.** `src/sanitizer.ts` imports `sanitizer/pkg/sanitizer_bg.wasm` directly as a `WebAssembly.Module` (wrangler's `[[rules]] type = "CompiledWasm"` rule) and calls `initSync`. **Do not use the wasm-pack `--target web` glue's `init()`** — it tries to fetch via `import.meta.url`, which doesn't exist on Workers. Init is lazy and idempotent. `skills/publishing.md` is bundled the same way via the `[[rules]] type = "Text"` rule and served verbatim as the `awh://publishing-guide` MCP resource — the same bytes humans read.

**MCP server lifecycle.** `src/mcp.ts` builds a fresh `McpServer` **per request**. The MCP SDK ≥1.26 throws on reused instances and cross-request state would leak. The seven tools (`publish_document`, `update_document`, `edit_document`, `read_document`, `list_documents`, `search_documents`, `create_publish_credential`) close over `props.agentId` resolved upstream — they never re-validate auth. (`create_publish_credential` is the lone non-document tool: it mints a short-lived `awh_` key for the byte-exact curl publish path via `mintEphemeralKey` — see the convention bullet below.) **HTML vs Markdown is a `format` parameter, not separate tools** — an earlier revision had `publish_document`/`publish_document_markdown`, `update_document`/`update_document_markdown`, and `read_document`/`read_document_text` twins; a `format` enum collapsed those six into three (`publish_document`, `update_document`, `read_document`). On the two write tools `format` (`"html"` | `"markdown"`) is **required** (forcing the choice avoids a default silently mis-parsing hand-written HTML); on `read_document` it's an optional **output** knob defaulting to `"markdown"` (the markdown branch calls `readDocumentTextCore`, the html branch `readDocumentCore`), and the response is a uniform JSON envelope (`content`/`format`/`version`/`sanitizer_v`/`converter_v`/`title`/`description`/`tags`/`slug`) that always carries metadata — so a read→edit→republish round-trip gets body + metadata in one call. Slug lookup is **not** a tool: pass `slug` to `list_documents` (the row is `documents[0]`); `findDocumentBySlugCore` still backs `GET /s/:slug`. Tool descriptions are intentionally heavy with the HTML contract (static-only, inline-SVG-not-`<img>`, inline styles) and lead with the use-case distinction between tools (`read_document` `format` — html for render/re-publish, markdown for ingest-as-context; `list_documents` for newest-first browsing/slug+tag filters vs `search_documents` for content discovery; `edit_document` vs `update_document` — small diff vs whole-body replace). `edit_document` is MCP-only (no HTTP `PATCH` counterpart) and its description says so — and because the merged `read_document` defaults to Markdown, the edit contract tells agents to read with `format:"html"` before copying an `old_string`. A cold agent that never reads `skills/publishing.md` should still understand the rules from the description alone. The three write tools (`publish_document`, `update_document`, `edit_document`) also carry the optional `title` / `description` / `tags` / `slug` contract at the tail of their descriptions — publish-vs-update inheritance is the subtle bit; shared schema-field constants near the bottom of `mcp.ts` (`CONTENT_FIELD`, `WRITE_FORMAT_FIELD`, `READ_FORMAT_FIELD`, the `*_FIELD` metadata constants) keep the wording consistent across tools. When editing tool descriptions, preserve the priority order (non-negotiables first) — length-trimmed renders truncate the tail.

## Conventions and gotchas

- **`If-Match` is required on `PUT /d/:id`** (428 if missing). Send `"v<n>"` for optimistic concurrency or `*` to skip. Strong tags only; no weak tags, no multi-tag lists.
- **`POST /d` requires `Content-Type: text/html`** (415 otherwise). Body is sanitized in-process and stored as `text/html; charset=utf-8`.
- **Optional `X-Content-SHA256` byte-exact integrity check on `POST /d` / `PUT /d/:id`** (`src/integrity.ts`, wired via `readVerifiedBody` in `src/index.ts`). Hashes the **raw received body before sanitization** — verifies the wire, not the sanitizer's transformation, so it's orthogonal to the `modified` flag. Malformed header → 400 `bad_integrity_header`; mismatch → 422 `integrity_mismatch`. **HTTP-only by design** (the companion to the `curl --data-binary @file` byte-exact publish path): the hash must come from the same deterministic tool that streams the file, so there is no MCP equivalent — the two primary MCP write-tool descriptions say so. `byte-exact-publish-design.md` records the rationale (the binding constraint at 66–85 KB is model regeneration of the `content` argument, not transport; chunked-append doesn't fix it).
- **Short-lived publish credentials (`agent_keys.expires_at`, migration 0007).** The byte-exact curl path needs a bearer at the shell, but Claude's connector settings can't store one and the connector's OAuth access token isn't exposed to the model/shell. So the already-OAuth-authenticated MCP session mints one on demand: the `create_publish_credential` tool → `mintEphemeralKey` (`src/admin.ts`) inserts a normal `awh_` key with a non-NULL `expires_at` (default 15 min, max 60, clamped). `authenticateAgent` (`src/auth.ts`) rejects expired keys exactly like revoked ones via the pure `isKeyExpired` helper. `expires_at = NULL` = never expires (every operator-minted key + legacy rows), so the migration is a no-op for them. **No scope column in v1** — single-tenant, the damaging vector (overwriting docs) isn't blocked by a write-only scope anyway, and the minted key grants nothing beyond the MCP session's existing authority; scope-limiting is the additive multi-tenant follow-up. The minted key/recipe is a deliberate disclosure surface — never log it (same discipline as the OAuth token). Revoke early via `DELETE /admin/keys/:id`.
- **Optional metadata on POST/PUT /d**: `X-Doc-Title`, `X-Doc-Description`, `X-Doc-Tags` (comma-separated), `X-Doc-Slug`. MCP write tools take the same shape as schema fields. On UPDATE, an *omitted* field inherits the prior version's value (or, for slug, the current document's value); an explicit `""` clears (and for title, re-derives from the new content's H1; for slug, releases it). Tags charset is restricted to `[A-Za-z0-9_-]` — invalid chars are silently stripped, not rejected. Slug charset is `/^[a-z0-9](?:[a-z0-9_-]{0,62}[a-z0-9])?$/` — invalid input is **rejected** (`invalid_slug` → 422), not silently sanitized (uniqueness means mutating the input could surprise-collide). Slug is unique across live docs (partial UNIQUE INDEX), conflict → `slug_taken` → 409. The render path applies anti-phishing normalization to title only (bidi/zero-width strip); description stores bidi for fidelity since it doesn't reach the browser tab.
- **Slug lives on `documents`, not `versions`.** Title/description/tags evolve per-version because they describe content; slug is identity-adjacent and uniqueness is per-document. It survives version increments unchanged unless the agent explicitly changes it (atomic release-old + claim-new in the same D1 batch). It's released to NULL on revoke alongside `revoked_at` being set, so the slug is reusable instantly. Pulling slug from `d.slug` in the joined read/list queries — not from prior_metadata — keeps the per-version `resolveMetadata` helper unaware of cross-document state.
- **`public_id` regex is fixed**: `/^[A-Za-z0-9_-]{22}$/` (`PUBLIC_ID_RE` in `src/serve.ts`). It's safe to interpolate into HTML templates after this check; `serveShell` does so.
- **Storage cap is best-effort.** `checkStorageCap` runs `SUM(size_bytes)` outside the insert batch; two concurrent writes can both pass. v1 accepts the slight overrun.
- **List endpoints are cursor-paginated** (`src/pagination.ts`). The four list surfaces — `GET /admin/agents`, `GET /admin/agents/:id/keys`, `GET /admin/documents`, and the MCP `list_documents` tool — share `parseHttpListParams` / `parseMcpListArgs` + the `paginate()` peek helper. Default page size 50, max 200. Ordering is `(created_at DESC, id DESC)` — the `id` tiebreaker is mandatory because D1's strftime stamps to ms and collisions are rare-but-possible; without it cursors could skip a row at a page boundary. Cursors are opaque base64url(JSON `{ts, id}`) — never construct them by hand. When you add a new list endpoint, route it through `pagination.ts` rather than introducing a new style.
- **Search is NOT cursor-paginated.** `GET /admin/documents/search` and the MCP `search_documents` tool both reuse the filter parser (`parseHttpListParams` / `parseMcpListArgs`) for `tags` / `slug` / `limit`, but ignore the cursor field — BM25 rank is not a stable cursor key the way `(created_at, id)` is. Results are capped at `limit` (default 50, max 200) and the response has no `next_cursor`. If the top N doesn't include what the caller wants, the answer is to refine the query, not paginate further. Query input is sanitized through `buildFtsMatchQuery` in `src/search.ts` — empty result → `bad_query` (422); the FTS MATCH never receives raw user input.
- **Document filtering** (`tags`, `slug`) lives on the document list/search surfaces — the parser collects them for every endpoint that uses `ParsedListParams`, but only `listDocumentsCore` and `searchDocumentsCore` consult them in SQL (the agents/keys lists silently ignore the fields). `tags` is AND semantics: one `tags LIKE ? ESCAPE '\\'` per requested tag against the JSON-encoded `versions.tags` column. The underscore in the tag charset collides with LIKE's `_` wildcard, so `tagLikePattern()` in `core.ts` escapes it with `\_`. Tag inputs are silently sanitized to the stored shape at parse time (matching write-time semantics); slug inputs are validated and reject with `bad_slug` on invalid charset (matching the write-time reject-not-sanitize rule). `findDocumentBySlugCore` is the dedicated single-row lookup behind the `GET /s/:slug` HTTP redirect (it validates the slug via `validateSlugInput` upstream of the DB hit). There is no longer a `find_document_by_slug` MCP tool — the slug-lookup path on MCP is `list_documents` with a `slug` filter (returns 0 or 1 rows; the caller reads `documents[0]`).
- **FTS sync is write-path-local.** No D1 triggers — `publishDocumentCore` / `updateDocumentCore` / `revokeDocumentCore` each include the `documents_fts` INSERT/UPDATE/DELETE in their existing `META.batch()` so the FTS row tracks the document's current version atomically. The body column is the markdown derivation of the sanitized HTML, computed once per write via `htmlToMarkdown(prep.cleanedHtml)`. Tags are stored space-joined in the FTS column (FTS5 will re-tokenize at index time; underscores inside tags are split there too). The FTS row is the SOLE storage of body-as-text — no `body_text` column on `versions`. If you add a new write surface, mirror the FTS sync inside the batch (or extract a helper) — don't ship a write path that bypasses FTS, or search will silently lag.
- **Operator session cookies + CSRF (`src/session.ts`).** Two cookies, both **host-only** (no `Domain`) and `SameSite=Lax`: `awh_session` (HttpOnly — the signed credential) and `awh_csrf` (readable — delivers the CSRF nonce to the page). `Secure` is **conditional on request protocol** (`isSecureRequest`): set behind https custom domains, omitted over `http://localhost` (a blanket `Secure` is silently rejected on http, breaking `wrangler dev`). CSRF is **stateless signed double-submit**: the nonce lives inside the *signed* session payload, and a cookie-authed mutating request must echo it (`X-CSRF-Token` header for JSON/admin; `csrf_token` form field for the HTML `/login`, `/logout`, `/d/:id/revoke` forms) — the server constant-time-compares the echo to the verified session's nonce, so a cookie-vs-cookie injection can't forge it. The pure crypto/cookie core takes explicit `nowMs`/`operatorToken`/`epoch` (no `Date.now()`, no env) so it's unit-tested in `test/session.test.mjs`; only the thin env wrappers read `env`. Login/logout/revoke set **two** Set-Cookie headers via `headers.append` (not `.set`, which clobbers). `next` redirects are origin-whitelisted via `validateNext` (rejects `//evil.com`, `/\evil.com`, absolute URLs, CRLF). Never log the operator token or the derived signing key (same discipline as the OAuth token / minted keys).
- **Logging discipline in MCP tools** (`src/mcp.ts`): log tool name + error code only. Never args (may contain user HTML), never request headers (may contain the bearer), never the OAuth token.
- **Any API-surface change must update MCP tool descriptions in the same commit.** New inputs, new response fields, new headers, changed status codes, changed inheritance/derivation semantics — all of these are part of the contract a cold MCP agent sees only via tool descriptions (the client may never expose `awh://publishing-guide`). Walk the seven tools in `src/mcp.ts` and confirm: (1) every inputSchema field for the changed surface exists with an up-to-date `.describe()`; (2) the prose description mentions the new field/behavior at the priority level it deserves (non-negotiables first, optional bits at the tail — length-trimmed renders truncate); (3) the JSON response shape documented in the description matches what the handler actually returns; (4) the bundled `skills/publishing.md` covers the same change so the resource and the descriptions don't drift. If the change is HTTP-only by design, say so explicitly in the description so an agent on MCP knows not to look for it.
- **Adding a sanitizer test:** the ~40 negative assertions live inline at the bottom of `sanitizer/src/lib.rs`. Each tweak to `make_builder()` should add a case there.
- **Ammonia's defaults silently omit several tags/attributes our contract allows.** The default tag set ships `article`/`aside`/`header`/`footer`/`nav`/`hgroup`/`figure`/`figcaption` but NOT `<section>` or `<tfoot>`, and its generic-attribute set is only `lang`/`title` (no `dir`; lists get only `start` on `<ol>`). Don't assume a tag/attribute is allowed just because a sibling is — the allowlist is `ammonia defaults ∪ make_builder() additions`, nothing more. `make_builder()` re-adds the gaps the publishing contract names: `<section>`/`<tfoot>` via `EXTRA_TAGS`, `dir` via `HTML_GENERIC_ATTRS`, and `reversed`/`value`/`type` per list element. `<main>`/`<address>` stay stripped by design (documented as such in `skills/publishing.md`). The `contract_*` corpus tests in `lib.rs` pin the entire documented surface (every allowed tag/attr survives, structural wrappers unwrap, `<main>`/`<address>` strip) so an ammonia upgrade or allowlist edit that drops a documented item fails loudly. Keep them in lockstep with `skills/publishing.md`.
- **Adding an advisory:** `src/advisories.ts` is mostly regex-based on (input, cleaned) pairs — false negatives are acceptable, false positives are not. Behind the specific regex rules sits a generic element-diff (`detectStrippedElements`) that reports any open-tag the output lost and no specific rule named — the catch-all that surfaces an allowlist gap nobody wrote a rule for (how `<section>` vanished silently). Its skip sets (`SPECIFICALLY_COVERED`, `UNWRAPPED_OK`, `CONTEXT_SENSITIVE_OK`) keep it false-positive-free; the entity-encoded-script case plus the generic/wrapper/comment/script-body cases in `test/advisories.test.mjs` are the guards.
- **Wrangler version + Workers types** are pinned in `package.json`. `compatibility_date = "2026-05-26"` with `nodejs_compat`.
- **Secrets** (`HMAC_PEPPER`, `OPERATOR_TOKEN`) live in `.dev.vars` for local dev (gitignored) and `wrangler secret put` for production — never `[vars]` in `wrangler.toml`. `SESSION_EPOCH` is the exception: a non-secret rotation counter, so it lives in `[vars]` (default `"1"`); bump it to invalidate all browser sessions.

## Where things live

- `src/index.ts` — route dispatch + thin HTTP wrappers for `/d` and `/d/:id`. Default export wraps `innerHandler` with the OAuth provider.
- `src/core.ts` — sanitize/cap-check/R2/D1 sequence shared by HTTP and MCP, plus `searchDocumentsCore` (BM25 over `documents_fts`) and `editDocumentCore` (find/replace that loads stored bytes, applies `src/edit.ts`, then delegates to `updateDocumentCore`). **Add new write surfaces here, not in route handlers.** Write paths sync the FTS row inside the same D1 batch as the metadata write.
- `src/search.ts` — `buildFtsMatchQuery` tokenizes raw query input into a safe FTS5 MATCH expression: word-character tokens (≥2 chars), optional trailing `*` for prefix match, lowercased to neutralize FTS5 operators (AND/OR/NOT/NEAR are case-sensitive uppercase keywords). Phrase queries and column filters are deliberately not supported.
- `src/edit.ts` — pure find/replace logic (`applyEdits` / `countOccurrences`) behind `edit_document`. Standalone (no D1/R2/WASM imports) so `test/edit.test.mjs` can import it under the Node strip-types runner, exactly like `search.ts`. Match semantics mirror Claude Code's `Edit` tool; replacement is literal (no `$`-substitution).
- `src/serve.ts` — the two render URLs (shell + raw), plus the `/s/:slug` 302 redirect (slug lookup + cross-document link target). CSP and sandbox flags are defined here. The revoke confirm/submit form is session-aware: a live operator session swaps the token-paste field for a hidden CSRF token.
- `src/session.ts` — operator browser-session layer. Pure crypto/cookie core (`mintSessionCookieValue` / `verifySessionCookieValue` / `mintCsrfNonce` / `parseCookies` / `serializeSetCookie` / `csrfMatches` / `validateNext`, unit-tested in `test/session.test.mjs`) + thin env wrappers (`authenticateOperatorRequest`, `requireOperator` — the shared operator guard with the CSRF gate, `isSecureRequest`, `sessionEpoch`). Imports `auth.ts` (never the reverse).
- `src/login.ts` — `GET/POST /login` (validate `OPERATOR_TOKEN` → mint session + CSRF cookies → 302 to validated `next`) and `GET/POST /logout` (POST-only state change behind a CSRF-token confirm form). Reuses the `/authorize` card chrome.
- `src/oauth.ts` — OAuth provider config. `resolveExternalToken` is the Door B integration point.
- `src/mcp.ts` — MCP server + seven tools (`publish_document`, `update_document`, `edit_document`, `read_document`, `list_documents`, `search_documents`, `create_publish_credential`), per-request lifecycle. HTML/Markdown is a `format` param on the write tools and an output knob on `read_document` — not separate tools. Imports `skills/publishing.md` as a bundled resource.
- `src/mcp-auth.ts` — the `AwhProps` type both doors converge on.
- `src/authorize.ts` — Door A consent UI (GET form + POST verify against `OPERATOR_TOKEN`).
- `src/admin.ts`, `src/admin-oauth.ts` — operator endpoints; `revokeAgent` cascades both keys and OAuth clients. Also exports `mintEphemeralKey` (the agent-callable short-lived-key minter behind the MCP `create_publish_credential` tool — not an HTTP route).
- `src/auth.ts` — `authenticateAgent` (HMAC + revoked + **expiry** check), `authenticateOperator` (constant-time string compare), `bearerToken`, `hmacSha256Hex`, the exported `timingSafeEqual` (reused by the session/CSRF compares), and the pure expiry helpers `isKeyExpired` / `computeExpiresAt` (unit-tested in `test/auth.test.mjs`).
- `src/ids.ts` — `newUuid`, `newPublicId` (22-char URL-safe base64), API key mint/parse.
- `src/integrity.ts` — optional `X-Content-SHA256` byte-exact handshake (`sha256Hex`, `normalizeExpectedSha256`, `verifyContentIntegrity`). HTTP-only; consumed by `readVerifiedBody` in `src/index.ts`. Pure functions, unit-tested in `test/integrity.test.mjs`.
- `src/sanitizer.ts` — WASM init shim. Exports both `sanitize`/`sanitizerVersion` (write) and `htmlToMarkdown`/`converterVersion` (read).
- `src/advisories.ts` — regex-based `stripped[]` / `will_not_render[]` detection for write responses.
- `src/metadata.ts` — title/description/tags validation, derivation (first H1 / first-N text), display-time anti-phishing normalization, and `parseMetadataHeaders` for the HTTP layer. Strip-range regexes built programmatically from hex tables so no invisible chars live in source.
- `src/pagination.ts` — cursor encode/decode, `parseHttpListParams` / `parseMcpListArgs`, and the `paginate()` peek-row helper. Shared across the four list endpoints; `MAX_LIMIT = 200`, `DEFAULT_LIMIT = 50`.
- `sanitizer/src/lib.rs` — Ammonia allowlist + ~40 corpus tests. Re-exports the markdown emitter as `html_to_markdown`.
- `sanitizer/src/markdown.rs` — HTML→GFM Markdown emitter (~40 corpus tests). Runs on sanitized bytes at read time; never on raw input.
- `migrations/` — D1 schema. `0001_init.sql` (agents/keys/docs/versions), `0002_oauth_clients.sql`, `0003_source_format.sql`, `0004_document_metadata.sql` (title/description/tags on versions), `0005_document_slug.sql` (documents.slug + partial UNIQUE INDEX), `0006_documents_search.sql` (`documents_fts` FTS5 virtual table), `0007_key_expiry.sql` (`agent_keys.expires_at` for short-lived publish credentials).
- `skills/publishing.md` — agent-facing authoring contract; also bundled as the `awh://publishing-guide` MCP resource. **Keep in sync with the sanitizer allowlist.**
- `skills/connector-guide.md` — human-facing guide for wiring Claude/Gemini connectors.
- `action-plan-v1.md` — design rationale, security model, deliberate v1 omissions.
- `byte-exact-publish-design.md` — design note on byte-exact large-document publishing: weighs key-disclosure vs chunked-append vs fetch-by-URL vs the integrity handshake, recommends the curl/REST path + `X-Content-SHA256` (implemented), and records what's deferred to an operator decision.
