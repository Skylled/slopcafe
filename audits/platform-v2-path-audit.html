Agent Web Host — Path to the PLATFORM v2 Spec

Gap audit · 2026-05-29 · commit b6009fe · compared against agent-knowledge-host-spec-PLATFORM-v2.md
Methodology: compared against the codebase via a multi-agent gap analysis; each finding independently verified against source.
Executive summary

The codebase today is a faithful, well-built single-tenant public-read tool. There is exactly one operator (a static OPERATOR_TOKEN, src/auth.ts:74-78), every document is globally readable by possession of its unguessable public_id, and every authenticated agent key can read and write the whole fleet by design (src/core.ts:113, :825). The SOLO fork deliberately collapsed the platform's identity, access-control, sharing, retention, and Trust & Safety machinery — and did so cleanly, with the seams documented.

Growing this into the PLATFORM v2 spec is not a rewrite. It is two sequenced expansions on top of an intact sanitize/serve/version/search core:

  1. Phase A — Private Trusted Testers. Build the multi-tenant identity + access-control core: an operators table, operator_id on every agent/document, the can_access() ladder as the sole data path, projects, polymorphic shares, per-version provenance, an append-only audit log, soft-delete/restore, per-key rate limits, and idempotency. This is what lets a known, vetted population each run an isolated tenant without leaking across boundaries.
  2. Phase B — Public Release. Add the duties of a host of untrusted third-party content: a Trust & Safety apparatus (reports, enforcement log, legal holds, CSAM auto-hold flow, platform-admin service), GDPR/CCPA erasure, two-counter quota and scheduled pruning, per-document origin isolation, an RLS/defense-in-depth backstop, WAF/DDoS edge protection, an adversarial sanitizer corpus, and counsel sign-off.

Phase philosophy. The dividing line is the population, not the feature count. Phase A asks: "can two vetted tenants share infrastructure without seeing each other's data?" — an isolation problem. Phase B asks: "can we host content authored by parties we do not trust and meet our legal and abuse obligations toward the world?" — a policing problem. A capability belongs in Phase A if a small trusted-tester population still needs it; it belongs in Phase B only if its justification is an untrusted/open audience.

What already exists and carries over present

The platform is built on this foundation, not instead of it. Each survives the multi-tenant transition essentially unchanged:

  • Ammonia-WASM sanitizer (write-time allowlist) + iframe sandbox & strict per-document CSP at render (the load-bearing wall, src/serve.ts:68-76). The script-execution threat is already contained.
  • Advisory engine — stripped[] / will_not_render[] feedback on write (src/advisories.ts).
  • Shared write/read core — both HTTP and MCP doors converge on src/core.ts; this is exactly the single mediated data path the spec wants can_access() and the audit hook to live in.
  • Nine-tool MCP server + agent API, append-only versioning, cursor pagination, and FTS5 search — all reused; the deltas scope them, they are not replaced.

Phase A — Private Trusted Testers private_testers

The multi-tenant identity + access-control core. Goal: a vetted population of operators, each isolated, with no cross-tenant leakage. No content-policing apparatus is built here — the population is trusted.

Deferred / out of scope deferred

Explicitly out of even platform v1 by spec. The only v1 obligation for most is a seam built earlier (noted), so retrofitting is a resolution/data change, not a migration:

  • Organization principal (cross-operator project ownership) — seam satisfied by the Phase A project-ownership resolver function; the org table itself is later.
  • Agent groups as share targets — the Phase A polymorphic shares.principal_type column leaves room; no migration needed to retrofit.
  • Public (no-sign-in) link sharing + cacheable CDN body path — the feature that converts this into an open publishing platform and triggers the bulk of the §10.1 legal load. (Distinct from SOLO's public-read-by-unguessable-URL.)
  • Scripted/JS document tier (Phase 2) — per-document security_tier='scripted', sandbox allow-scripts (never with allow-same-origin), required viewer interstitial.
  • doc: internal-link resolution + ref= read shortcut — no doc: handling exists anywhere; only relevant once per-document origins break relative URLs. Slug "required-on-row + server generation" follows this work.
  • Killable sanitization worker pool — the spec's Node-worker_thread requirement is largely redundant against the per-request-isolated Workers CPU model; a conscious non-build gated on the pre-parse gate + differential review proving insufficient.
  • Soft-deleted-target dangling-link rendering — a property of the unbuilt doc: resolver, not the deletion path.

Ordered implementation plan

Loose dependency order. Foundational schema/identity first; each step names the items it unblocks.

Phase A — Private Trusted Testers

  1. operators table + Google-OAuth operator sessions. The keystone — nothing else can be tenant-scoped until an operator is a real principal. (depends_on: none)
  2. operator_id + status on agents; operator_id on documents (stamp + backfill to a bootstrap operator). The structural ownership anchor. (depends on 1)
  3. projects (default project, resolver-function ownership) + documents.project_id + project_agents; per-project slug uniqueness. The org/ACL-scoping tier. (depends on 1, 2)
  4. share_policy enum + default_share_policy + polymorphic shares table. The policy inputs to access control. (depends on 2, 3)
  5. can_access() in src/access.ts as the sole data path; route every read/write/list/search/fetch through it. The isolation mechanism — owner shortcut → deleted_at → owner-state → policy switch → explicit share. (depends on 2–4; the central dependency)
  6. Scope writes & listings by operator; owner-account-state fail-closed at auth; per-version provenance. The operational half of isolation. (depends on 5)
  7. append-only audit_log + write hook (incl. denied attempts) + source-IP reader. Capture from day one or the data is unrecoverable. (depends on 1, 5)
  8. per-key rate/creation caps + per-operator quota gate + the 429/Retry-After error model (HTTP + MCP descriptions). Runaway-agent backstop. (depends on 2)
  9. soft-delete (deleted_at) + restore (auto-rename) + write_idempotency / Idempotency-Key + 409 reuse + 422 violations[]. Lifecycle + write correctness. (depends on 5; soft-delete is the 1b gate)
  10. retain pre-sanitized original_key + pre-parse SAX depth/node/attr gate. Cheap now, impossible to backfill; structural DoS first filter. (depends on: none — do early so originals start accruing)
  11. /v1 two-tier namespace + operator key/agent management (re-home + scope) + share endpoints + version-history/rollback + connector OAuth handoff + GET /v1/contract + account-deletion soft phase. The operator-facing surface + MCP operator_id threading. (depends on 1, 5, 6)

Phase B — Public Release

  1. Counsel engagement (start in parallel, early). DMCA designated agent, CSAM/NCII runbooks, DSA, ToS/AUP, privacy, and the actual retention durations the two-clocks model needs. Gates launch and feeds several builds. (depends on: nothing in code; do not start it last)
  2. scheduled-handler / Cron infrastructure + split version body (sanitized_key/original_key + sizes). Shared plumbing + the storage shape every destruction/quota item consumes. (depends on Phase A original_key)
  3. platform_admins principal + admin_can() + statement_of_reasons + enforcement_actions log. The authz substrate every admin verb writes to. (depends on Phase A identity)
  4. legal_holds + versions.held + write-time inheritance + hold-wins guard on every destruction path + nightly reconciler. Must precede any hard-delete/prune or held bytes get destroyed. (depends on 2, 3)
  5. documents.withheld_at takedown + operator suspension + unified cache-purge routine + the full admin verb surface (one target per call) + four-eyes seam. (depends on 3, 4)
  6. reports table + always-202 rate-limited intake; wired CSAM auto-hold + auto-withhold (authenticated-only) flow; DMCA strikes-as-query. The one end-to-end T&S flow. (depends on 4, 5)
  7. two-counter quota (live/physical_bytes) + hard-delete/erasure with audit pseudonymization + background pruning + account terminal purge + audit retention rollover. Storage hygiene + GDPR/CCPA. (depends on 2, 4)
  8. RLS / database-layer backstop (likely the D1→Postgres metadata migration) mirroring can_access. The pre-public second wall against an app-layer can_access bug. (depends on Phase A can_access)
  9. per-document origin isolation + content-origin header Worker + signed-token mint gated on access. Cross-document serve isolation at scale. (the origin and the Worker move together)
  10. Adversarial mXSS corpus in CI + sanitizer-update runbook/advisory watch/parser differential + lazy & targeted re-sanitization + structural CSS allowlist + global sanitization concurrency cap. Sanitizer hardening for an open population. (re-sanitization depends on the split body; corpus/runbook are independent — start early)
  11. WAF + DDoS posture + network rate-limit backstop + Logpush; intake rate-limit; (additive) per-key anomaly detection over the audit log. Edge + observability, final pre-launch IaC sign-off. (depends on Phase A rate limiter; gated with RLS under counsel sign-off)

Phase A is an isolation problem solvable by a small team on the existing stack; the single hardest item is the can_access + identity core that everything else hangs from. Phase B is a host-of-untrusted-content problem whose cost is dominated by the admin/T&S service, the RLS/store decision, and counsel — none of which should block onboarding a vetted tester population.
