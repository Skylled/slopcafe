Agent Web Host — SOLO v1 Spec Conformance Audit

Audit date 2026-05-29 · commit b6009fe · spec agent-knowledge-host-spec-SOLO-v1.md
Methodology: compared against the codebase via a multi-agent gap analysis; each finding independently verified against source.

Executive summary

The codebase is a faithful SOLO subset on the load-bearing security primitives — capability-URL entropy, HMAC-under-pepper key verification, the revoke kill switch, and the scriptless sandbox + strict CSP serve wall are all present and correct. The drift is concentrated in three areas: the slug-privacy model, the serving-isolation model, and a cluster of retained-original / re-sanitization seams the spec expects to exist.

Headline items, in order of weight:

  1. Public /d/by-slug/:slug endpoint contradicts §4's "there is no public /d/by-slug/<slug> endpoint to probe." A guessable slug becomes an enumeration surface that resolves to the secret public_id — directly collapsing the "slugged and unslugged docs are equally private" invariant. (high)
  2. Single-origin serving rather than the per-document isolated origin §5 mandates. Heavily mitigated by a sandbox that grants neither allow-scripts nor allow-same-origin, but a forbidden-by-spec deviation on the serve boundary. (high)
  3. No doc:<slug> write-time link resolution. The slug's sole stated purpose (inter-document links) is unbuilt — leaving the slug a partly-orphaned feature whose only shipped consumer is the spec-forbidden public redirect. (medium)
  4. No original-bytes retention / no lazy re-sanitization. Convert-and-discard drops the §5 "old versions self-heal on fetch" story; a future profile hardening cannot auto-heal already-stored versions. (medium)
  5. Operator auth is a static OPERATOR_TOKEN bearer, not the Google-OAuth session §1 names. A reasonable single-operator simplification, but a divergence from the named mechanism. (medium)

Several of these (slug-release-on-revoke, missing doc: resolution, convert-and-discard, the OAuth door) are defensible SOLO decisions — but the spec text says the opposite in each case, so the right close-out for most is an explicit, recorded divergence, not silent drift. The one item that should be treated as a genuine open security gap regardless of intent is the public by-slug endpoint.

1. Principals & authentication (§1)

Operator authenticates via a static OPERATOR_TOKEN, not Google OAuth drifted medium

Current: authenticateOperator extracts a Authorization: Bearer value and constant-time-compares it to env.OPERATOR_TOKEN. Every operator surface — admin endpoints, the Door A consent screen, the document-revoke form — wraps its input into a synthetic Bearer request and calls the same function. No session, no cookie, no OAuth/OIDC provider; src/admin.ts:3 states verbatim "for v1 we use a single shared secret (no Google OAuth yet)."

Spec: §1 — operator "authenticates via Google OAuth (Auth.js / Lucia or equivalent)"; line 35 contrasts "operator (session)" vs "agent (key)."

Evidence: src/auth.ts:74-78; src/admin.ts:3; src/authorize.ts:82-87; src/admin-oauth.ts:35-40

Recommendation: Two paths — (a) accept as a v1 simplification and amend §1 to state the operator authenticates via a static bearer token, so spec and code stop diverging; or (b) build the OAuth/OIDC session §1 names. The static approach is reasonable for a single operator, but note the operational consequences vs a session: the token is long-lived (no expiry/refresh), is sent plaintext on every admin call and pasted into the public /authorize and /d/:id/revoke forms, and has no per-action MFA. If retained, document the rotation procedure and ensure it is never logged.

Agent key secret is 24 CSPRNG bytes, below the spec's "≥32-byte" floor drifted low

Current: newApiKey mints the secret from new Uint8Array(24) (192 bits), base64url-encoding to 32 chars; the inline comment reads "32 b64url chars," suggesting the "32" was read as encoded chars, not raw bytes. The 8-byte prefix is separate. Verified secret material is 24 bytes.

Spec: §1 — agent key is "a ≥32-byte CSPRNG secret."

Evidence: src/ids.ts:49-55

Recommendation: Bump secretBytes to new Uint8Array(32) to meet the literal floor, or amend the spec to "≥32-character / ≥192-bit." Security impact is minimal — 192 bits is far beyond brute-force — hence a conformance nit, not a vulnerability.

Key verification: HMAC-under-pepper, constant-time, fail-closed present-as-intended info

Current: authenticateAgent looks up by indexed key_prefix, recomputes HMAC-SHA256(secret, HMAC_PEPPER) as lowercase hex, and compares via timingSafeEqual. Returns null (uniform 401) on missing token, parse failure, or missing pepper — an unset pepper fails closed, not open. Schema stores only the hash + indexed prefix. This is a key-verification construction (HMAC), not a slow password hash, exactly as §1 specifies.

Evidence: src/auth.ts:26-37,53-71; src/admin.ts:137,241; migrations/0001_init.sql:23-32

Recommendation: None — matches spec as intended.

Revoked key fails closed on the next request present-as-intended info

Current: if (!row || row.revoked_at) return null runs before the HMAC recompute, so a revoked key fails closed without reaching the crypto check. Read live on every request (no Door B token caching), so setting revoked_at closes the key on the very next request. Door B (static bearer on /mcp) reuses authenticateAgent verbatim via resolveExternalToken, so revocation is uniform across HTTP and MCP.

Evidence: src/auth.ts:65; migrations/0001_init.sql:28; src/oauth.ts:76-87

Recommendation: None — the rogue-agent kill switch matches spec.

Anonymous read gated solely by possession of the high-entropy public_id present-as-intended info

Current: newPublicId produces 16 CSPRNG bytes = 128 bits, base64url to 22 chars, meeting §4's "≥128 bits, non-enumerable." No reader credential or per-document access matrix exists; URL possession is the capability — consistent with the SOLO collapse to "mine + link-published."

Evidence: src/ids.ts:30-34

Recommendation: None — matches spec as intended.

OAuth 2.1 + PKCE "Door A" for agents (additive, not divergent) present-as-intended info

Current: Beyond the static awh_ key, agents can authenticate to /mcp via OAuth 2.1 authorization-code + PKCE: allowPlainPKCE: false, 15-min tokens, no DCR (pre-registration only). One client pinned per agent, operator-only mint, client_id↔agent_id join in D1. Both doors converge on the identical AwhProps { agentId, via }; consent is gated behind the same OPERATOR_TOKEN; deleteClient cascades token invalidation on revoke.

Spec: §1 names only "a per-agent API key" and does not describe an OAuth door; §0 requires SOLO be a strict subset of the platform spec (carried-over/simplified, never divergent).

Evidence: src/oauth.ts:42-89; src/admin-oauth.ts:49-105,136; src/authorize.ts:85,114-123; src/mcp-auth.ts:19-22; migrations/0002_oauth_clients.sql:14-19

Recommendation: Treat as additive and reasonable — it's the transport hosted Claude/Cowork require, resolves to the same per-agent revocable identity, minting is operator-gated, tokens are short-lived with synchronous cascade-revoke. Not a security gap. Spec-hygiene only: update §1 to acknowledge the OAuth door so the "strict subset" lineage stays auditable, and confirm OAuth-grant agents are write-capable like API-key agents (no read-only tier). No code change.

2. Document model & two names (§2, §3, §4)

Public unauthenticated /d/by-slug/:slug contradicts §4's "slug is NEVER externally visible" drifted high

Current: GET /d/by-slug/:slug is wired as a public, no-auth route that validates the slug shape and 302-redirects to /d/:public_id. The handler's own doc comment states it is "intentionally public, no auth required." findDocumentBySlugCore does a bare slug lookup with no principal check. The redirect leaks the secret public_id in the Location header to anyone who can guess or type a slug. (See also §3 finding below — this is the same surface from the capability-URL angle.)

Spec: §4 line 108 — verbatim "There is no public /d/by-slug/<slug> endpoint to probe — so a memorable, guessable slug like north-island-report is NOT an enumeration surface. (Slugs resolve only for authenticated principals... never for an anonymous reader.)"

Evidence: src/index.ts:166-170; src/serve.ts:483-489,500-512; src/core.ts:932-948

Recommendation: A direct contradiction of the core privacy invariant — a guessable slug becomes an enumeration surface that maps to the unguessable public_id, collapsing "slugged and unslugged docs are equally private." Either (a) require operator/agent auth on this endpoint (resolve only for authenticated principals, per §4), or (b) remove the public endpoint entirely and resolve slugs only at write time per §4's link-resolution model. If a public slug lookup is a deliberate post-spec product decision, record it as an explicit divergence from §4 — the spec was written to forbid exactly this, and it changes the read-security model the whole §4 boundary rests on. This is the audit's top item.

Revoke releases the slug to NULL; §3/§4 require a revoked doc to keep its slug claimed drifted medium

Current: revokeDocumentCore sets slug = null in the same UPDATE that flips revoked_at, explicitly so the slug is "immediately available to a future publish." A POST claiming a previously-revoked slug therefore succeeds rather than 409ing.

Spec: §3 — "A revoked document keeps its slug claimed... it prevents a new document from hijacking a revoked one's identity and inbound links." §4 repeats "its slug stays claimed."

Evidence: src/core.ts:1186-1195 (slug=null at :1190); src/core.ts:1153-1158; migrations/0005_document_slug.sql:11-17

Recommendation: Decide intentionally. The spec's hijack-of-inbound-links reasoning only bites if inter-document slug links exist — and they do not (see the doc: finding below), so the attack surface the spec guards against currently has no instance. Reasonable to keep release-on-revoke as a deliberate SOLO simplification, but flag it as a conscious divergence (the spec says the opposite), and revisit if/when doc:<slug> resolution is built.

No PUT /d/by-slug/<slug> create-or-update verb; updates are public_id + If-Match only drifted low

Current: The only update verb is PUT /d/:public_id, keyed by the unguessable id and gated by a required If-Match. /d/by-slug/:slug is GET-only and never accepts PUT. updateDocumentCore returns not_found when the public_id doesn't resolve — it cannot create.

Spec: §3 — "PUT /d/by-slug/<slug> → create-or-update at this name... PUT is idempotent on the slug." §6 restates "PUT /d/by-slug/<slug> (or PUT /d/<id>) writes a new version." Intended model: "POST to claim a name; PUT to revise the doc at a name."

Evidence: src/index.ts:164,166-170,423-446; src/core.ts:495-503,515-517

Recommendation: Materially different addressing/verb model — arguably a cleaner strong-ETag optimistic-concurrency design, but a divergence from §3/§6, not a subset. Either add the create-or-update PUT /d/by-slug/<slug> verb, or record the public_id + If-Match model as the intended replacement and update §3/§6. Note: a PUT-by-slug verb would be an authenticated surface, which the spec permits — unlike the public GET redirect above.

No doc:<slug> write-time link resolution — the slug's sole stated purpose is unimplemented missing medium

Current: No doc:<slug> grammar, write-time slug→public_id rewriting, or dropped-link advisory exists anywhere. The write path (prepareForStorage) runs convert → sanitize → advisories with no link-resolution step; the sanitizer has no doc: scheme handling, so an <a href="doc:section-1"> href is silently stripped by ammonia (not in its scheme allowlist) with no advisory naming the dropped link. Slugs are used only for lookup/filter and the by-slug redirect. src/advisories.ts:42-44 reserves this exact case in a TODO.

Spec: §3 — the slug "exists for one reason: inter-document hyperlinks." §4 — links "authored by slug, resolved to public_id at WRITE time... stores the already-resolved URL," with the no-forward-links drop-and-advise (missing/revoked target → link dropped to plain text + non-fatal advisory). §9 lists this as a thing to lift.

Evidence: src/core.ts:147-167,68-74,882-887,932-948; src/serve.ts:500-512; src/advisories.ts:42-44 (grep for doc: grammar returned no matches)

Recommendation: A deliberate v1 descope consistent with CLAUDE.md framing — but it leaves the slug a partly-orphaned feature: §3 says inter-doc links are its whole reason to exist, yet the only shipped consumer of the slug is the (spec-forbidden) public by-slug redirect. Record an explicit ADR/plan note that write-time doc: resolution + drop-and-advise (and the matching advisory field) are deferred, so the slug's privacy contract is evaluated against actual current uses rather than the spec's intended one. If descoped, the public by-slug endpoint is then doubly unjustified.

No original_key column — convert-and-discard drops the lazy re-sanitization §5 requires drifted low

Current: The versions table stores only the sanitized R2 blob; there is no original_key column. Pre-sanitized bytes are explicitly discarded ("convert-and-discard is the v1 storage model"). No re-sanitization-on-fetch path exists — readDocumentCore just GETs the stored bytes. (This is the same root cause as the §5 finding below, seen from the schema side.)

Spec: §2 lists "original_key ... the pre-sanitized input (retained for re-sanitization; §5)." §5 — "stale stored versions are re-sanitized lazily on fetch (regenerate sanitized_key from original_key when a fetched version's sanitizer_profile_version is stale)."

Evidence: migrations/0001_init.sql:50-58; migrations/0003_source_format.sql:3-4; src/core.ts:46,388,568,677-717

Recommendation: Consequential but defensible tradeoff: without original_key, the §5 self-heal-on-fetch story is unavailable — if the Ammonia profile later closes a hole, already-stored versions keep serving old-profile bytes until rewritten. Given v1's non-adversarial-author model and the load-bearing CSP/sandbox wall, residual risk is low. Keep convert-and-discard if storage pressure justifies it, but record it as an explicit divergence and note the mitigation gap, so a future profile hardening triggers a deliberate corpus rewrite rather than silent stale-bytes exposure.

Schema deltas: sanitizer_v rename, missing sanitized_hash/model; nullable slug; no label partial info

Current & spec, three sub-items (all faithful / for-the-record):

  • slug nullable vs §2 NOT NULL. Implemented as nullable TEXT with a partial unique index excluding NULL; publish inserts NULL when no slug supplied. §2's schema block declares slug ... not null, but §3 prose contemplates slugless docs ("standalone documents... need none"). The implementation correctly follows the prose; the NOT NULL is a spec internal inconsistency. Worth a one-line note so future schema audits don't re-tighten it. migrations/0005_document_slug.sql:30,36; src/metadata.ts:192
  • sanitizer_profile_version → sanitizer_v. A harmless rename, populated from sanitizerVersion() and surfaced on reads/writes. Present as intended. migrations/0001_init.sql:55; src/core.ts:398,575
  • sanitized_hash and model absent. model is a nullable provenance nicety (no X-Doc-Model header / write field; safe SOLO drop). sanitized_hash pairs with the re-sanitization story above; its absence is consistent with convert-and-discard. No action beyond recording the two deliberate omissions.
  • label column not implemented. §2/§3 explicitly reserve label as a deferred, cosmetic, no-uniqueness nice-to-have. Its absence is a faithful descope, not drift. migrations/0001_init.sql:37-45; src/metadata.ts:175-193

Recommendation: No code change. Note the rename and the deliberate omissions so a future schema diff against §2 doesn't read them as accidental.

public_id is 128-bit CSPRNG, 22-char base64url, format-gated everywhere present-as-intended info

Current: newPublicId() draws 16 bytes (128 bits) from crypto.getRandomValues and base64url-encodes to 22 chars. PUBLIC_ID_RE = /^[A-Za-z0-9_-]{22}$/ gates every read/update/revoke path. documents.public_id is TEXT NOT NULL UNIQUE. No sequential ids.

Evidence: src/ids.ts:13-18,30-34; src/serve.ts:31; migrations/0001_init.sql:39; src/core.ts:487,681,1164

Recommendation: None — meets §4's load-bearing entropy requirement exactly.

3. Capability-URL security model (§4)

The public by-slug endpoint and the missing doc: resolution appear above under §2; the §4 view is the same surface seen through the read-security boundary. The remainder of this cluster is correct-by-design.

Authenticated slug→public_id resolution (MCP find_document_by_slug + list/search filters) present-as-intended info

Current: findDocumentBySlugCore returns the full listing incl. public_id; it backs the MCP find_document_by_slug tool (agent-auth) and the operator-auth admin list/search slug filters. These are spec-permitted because authenticated. The shared core function is fine; only the anonymous serveBySlug caller violates §4 (see §2).

Spec: §4 line 108 — slugs "resolve only for authenticated principals — operator/agent via API." §5 — agent read accepts GET /d/<public_id-or-slug>.

Evidence: src/core.ts:932-948,882-887,1047-1050; src/index.ts:122-127,141-144; src/mcp.ts (find_document_by_slug)

Recommendation: None — keep. The authenticated lookups are spec-blessed; the fix belongs at the anonymous caller, not the shared core.

Born-live, no draft state present-as-intended info

Current: publishDocumentCore inserts the document and version 1 and sets current_ver = 1 in one D1 batch, so the public_id is reachable on create. No per-document public/private toggle anywhere; serve paths gate only on revoked_at. The deferred draft/private seam (§4) is reserved, not built.

Evidence: src/core.ts:411-442,453; src/serve.ts:224

Recommendation: None — born-live as specified; absence of draft state is the intended v1 descope.

Revoke flips revoked_at before R2 purge; all serve paths 404 opaquely present-as-intended info

Current: revokeDocumentCore sets revoked_at, nulls current_ver, releases slug, and deletes the FTS row in a D1 batch before the R2 delete — so the doc is unreachable instantly even if R2 lags. Every serve path 404s on revoked_at; the opaque notFound() is identical to the missing-row 404 (a revoke can't be distinguished from a never-existed id); all serve routes carry Cache-Control: no-store. Versions rows persist as audit trail.

Spec: §4 line 117; §5 line 134 ("a revoked public_id must stop resolving at the edge"). Note: the slug-release half of this batch is the docmodel divergence above; the kill-switch half is correct.

Evidence: src/core.ts:1186-1199; src/serve.ts:224,357,432,35,125-130; src/core.ts:701

Recommendation: None — kill switch implemented as specified, including the revoke-before-purge ordering and opaque 404.

rel="noopener noreferrer" + no-referrer header/iframe close Referer leakage present-as-intended info

Current: The sanitizer forces rel="noopener noreferrer" on anchors via link_rel(); external anchors get target="_blank". Defense in depth: Referrer-Policy: no-referrer on all serve responses and referrerpolicy="no-referrer" on both render iframes. The secret /raw (and /d/:id) URL does not leak to outbound destinations.

Evidence: sanitizer/src/lib.rs:173,184,241-272,971-976; src/serve.ts:37,297,395

Recommendation: None — closed both ways. (Fragment/relative/mailto links get no target so ammonia's link_rel skips them, but they don't leave the origin; the no-referrer header is the catch-all.)

4. Sanitize & serve (§5)

Single-origin same-domain serving, not a per-document isolated origin drifted high

Current: All documents render on the one Worker origin. The shell embeds <iframe sandbox src="/d/:public_id/raw"> and the raw bytes are served from the same origin; RAW_CSP uses frame-ancestors 'self'. There is no per-public_id origin/subdomain derivation anywhere in serve.ts.

Spec: §5 (carrying platform §7.1) — the document is served inside a sandboxed iframe from a per-document isolated origin, with the origin host derived from / gated by the public_id (which is "both the origin discriminator and the capability secret"). Per-document origins isolate documents from each other.

Evidence: src/serve.ts:297,395,68-76 (RAW_CSP, frame-ancestors 'self'); compensating control src/serve.ts:119 (SANDBOX = "allow-popups allow-popups-to-escape-sandbox")

Recommendation: The largest §5 drift. The compensating control is real: the sandbox grants neither allow-scripts nor allow-same-origin, so framed documents cannot run JS or act as the origin — removing most of the cross-document-isolation value the per-document origin was protecting. Decide explicitly: either document this as a deliberate SOLO simplification justified by the no-script sandbox (and downgrade), or implement public_id-derived origins (e.g. <public_id>.docs.<host>) if script execution is ever enabled. Flagged high because it is a forbidden-by-spec deviation on the load-bearing serve boundary; security impact is heavily mitigated by the scriptless sandbox.

RAW_CSP omits explicit connect-src 'none' partial low

Current: RAW_CSP is default-src 'none'; img-src 'self' data:; style-src 'unsafe-inline' data:; font-src 'self' data:; frame-ancestors 'self'; base-uri 'none'; form-action 'none'. No explicit connect-src — it falls back to default-src 'none'.

Spec: §5 explicitly names connect-src 'none' (no exfiltration) in the serve-time CSP.

Evidence: src/serve.ts:68-76 (grep for connect-src in src/ returns no matches)

Recommendation: Functionally satisfied — default-src 'none' covers connect-src, and with no allow-scripts there is no script to issue fetch/XHR/WebSocket. Optionally add explicit connect-src 'none' for defense-in-depth and to match the spec's named directive list verbatim. The fallback already enforces it.

No original bytes retained, no lazy re-sanitization on stale-profile fetch missing medium

Current: Convert-and-discard stores only sanitized HTML at <docId>/v<n>; no original_key column. versions.sanitizer_v is stamped at write but no read path consults it to detect staleness. serveRaw and readDocumentCore return stored bytes verbatim with zero re-sanitization. (Schema-side view of this is in §2 above.)

Spec: §5 — store sanitized + original + sanitizer_profile_version; when the profile hardens, stale versions are re-sanitized lazily on fetch (regenerate from original_key), updating hash/version in place without advancing version_no.

Evidence: src/core.ts:388,393-401; migrations/0001_init.sql:55; src/serve.ts:420-447

Recommendation: Self-healing is partially present — new writes always use the current profile, so the corpus heals forward as documents are re-PUT. But a document never rewritten after the sanitizer hardens stays frozen at the old profile, and the lazy path is impossible to build as specced because original_key is not retained. To match §5, either (a) start retaining original bytes + add the stale-profile re-sanitize-on-fetch path, or (b) document the convert-and-discard tradeoff as a deliberate descope and note that profile-hardening requires operator-driven re-PUT rather than auto-heal. CLAUDE.md already documents convert-and-discard as intentional, so this is likely an intended descope that nonetheless diverges from the spec text.

SVG <use> is allowlisted (with href/xlink:href); §5 requires it dropped drifted medium

Current: <use> is in SVG_TAGS and both href and xlink:href are granted on it; <symbol>/<marker> are also allowlisted. <foreignObject> is correctly dropped (with a pinning corpus test), as are <animate*>. No corpus negative test pins <use> removal.

Spec: §5 (platform §6.3) — SVG is an explicit allowlist, and both <use> and <foreignObject> are dropped. <use href> can reference external/fragment definitions and is a known SVG sanitizer-bypass vector.

Evidence: sanitizer/src/lib.rs:70 (use in SVG_TAGS); sanitizer/src/lib.rs:153-157 (href + xlink:href on use); contrast §5 line 129

Recommendation: Remove use from SVG_TAGS (and its per-tag href grant), or restrict <use> to same-document fragment refs only and document the deviation. Add a corpus negative test pinning <use> removal, mirroring strips_foreign_object. Medium because the scriptless sandbox + default-src 'none' largely defang an external/data reference — but it is an explicit spec-forbidden construct surviving the allowlist, and §5 names the allowlist (not the CSP) as the security boundary.

No killable-worker timeout, concurrency bound, or pre-parse depth/node gate missing medium

Current: The only resource bound is the 5 MiB raw-input byte cap (MAX_INPUT_BYTES), checked before conversion. sanitize() runs ammonia synchronously in-isolate with no timeout, no depth/node-count pre-parse gate, and no concurrency cap.

Spec: §5 (platform §6.6) — per-document size cap (present) AND a cheap pre-parse depth/node/attribute gate before the heavy parser AND sanitize in a killable worker with a global concurrency bound, because a compromised/injected agent can still submit a pathological document.

Evidence: src/core.ts:40,351-354,528-531; sanitizer/src/lib.rs:182-185

Recommendation: Note the runtime difference: the platform §6.6 concern is the Node event loop blocking; on Cloudflare Workers each request runs in its own isolate with platform-enforced CPU-time limits, which partially substitutes for the killable worker + concurrency bound. The pre-parse depth/node/attribute gate, however, has no platform substitute and is genuinely absent — a deeply-nested 5 MiB document goes straight into ammonia (the size cap alone does not bound nesting depth). Either add a cheap SAX-style depth/node gate before clean(), or document the per-isolate CPU limit as the accepted SOLO substitute for the killable-worker requirement — but add the pre-parse gate regardless.

Agent Markdown read is a separate /text endpoint, not Accept content negotiation drifted low

Current: GET /d/:public_id content-negotiates on the Authorization header, not Accept — a valid agent key returns raw sanitized HTML, no header returns the shell. The Markdown projection is a distinct route GET /d/:public_id/text (plus the read_document_text MCP tool). Nothing inspects Accept for representation selection.

Spec: §5 — agent read is GET /d/<public_id-or-slug> with Accept content negotiation; text/html default, text/markdown reserved for the future projection.

Evidence: src/serve.ts:153-181,462-478

Recommendation: Mechanism drift, same intent — the dual representation (HTML + derive-on-read Markdown from sanitized bytes, never raw) is delivered and in fact exceeds the "reserved for future" projection. The deviation is a separate URL suffix + Authorization-based switching instead of Accept negotiation on one resource. No security/correctness impact. Optionally add Accept: text/markdown handling on GET /d/:public_id for authenticated agents to match the standards-correct shape, or document /text as the chosen SOLO mechanism.

target="_blank" injected by post-clean() byte-splice rather than inside the profile drifted low

Current: rel="noopener noreferrer" is forced inside the ammonia profile via link_rel() (correct). But target="_blank" is added by add_blank_target_to_external_links, a byte-splice that runs on the output of clean() — after the sanitizer's final serialization.

Spec: §5 — "no markup mutation after the sanitizer's final serialization (re-serialization after the last security decision manufactures mutation-XSS). Forced link attributes (rel, target) are applied inside the sanitizer profile (a hook), never as a post-pass."

Evidence: sanitizer/src/lib.rs:182-185,241-272

Recommendation: Low because the implementation is carefully constrained to avoid the mutation-XSS the rule guards against: it operates on ammonia's already-normalized serialization, inserts only a fixed literal target="_blank" into matched <a…> start tags, copies all other bytes verbatim (no re-parse, no SVG/text re-serialization), and is idempotent (corpus test). Still a literal post-sanitize markup mutation, which the spec forbids. Either move target injection into an ammonia hook/profile transform to satisfy the letter of the rule, or document in code why this byte-splice is exempt (it does not re-serialize and cannot manufacture new parse contexts). The extensive corpus tests mitigate regression risk.

5. Write semantics, storage cap, publisher posture (§6, §7, §8)

Idempotency-Key header + 48h sweep is entirely absent missing low

Current: No code path reads an Idempotency-Key header; no dedup store, no 48h sweep (repo-wide grep returns zero matches). POST /d and PUT /d/:public_id parse content-type, body, metadata headers, and If-Match only; core write functions take no key argument.

Spec: §6 / §9 (lifting platform §11.3) — "Idempotency-Key header dedupes retries (ephemeral, swept after 48h). New attempt ⇒ new key; retry of the same attempt ⇒ same key."

Evidence: src/index.ts:287-353,423-519; src/core.ts:335-465,477-649

Recommendation: Either implement the path (a short-TTL KV/D1 dedup keyed on (route, key) → cached result with a 48h sweep) or, if deliberately descoped for SOLO, document the omission explicitly. The spec lists it as a carried-over requirement, so right now it is undocumented drift rather than a stated descope. Risk is low at single-operator scale (a retried POST creates a duplicate doc with a fresh public_id; a retried PUT creates an extra version) but non-zero.

POST /d does not accept or store the optional model field partial info

Spec: §6 — "Create: POST /d with { html, slug, model? }." §2 reserves a model column (provenance: which model authored the version).

Evidence: src/index.ts:300,302-303; src/core.ts:416-428; no model column in migrations

Recommendation: Deliberate descope working as intended for SOLO. The request shape diverges from the literal { html, slug, model? } (the impl uses a text body + headers, the same information minus model), and model provenance is simply not captured — a low-value field. Update addressed by public_id + If-Match; no PUT /d/by-slug create-or-update drifted info

Recommendation: Appears to be a deliberate SOLO simplification (slug is never externally writable; the public by-slug surface is a redirect only). Current: PUT /d/:public_id requires If-Match (428 when absent), parses only strong "v<n>" tags or the * wildcard, and returns 412 on version mismatch. The response ETag is "v<version>" on both POST (201) and PUT (200); version_no is the monotonic per-document int.

Single storage cap enforced at write via best-effort SUM(size_bytes) present-as-intended info

Revoke is operator-only (HTTP + form) and hard-purges R2 bytes present-as-intended info

Current: DELETE /d/:public_id gates on authenticateOperator (401 otherwise) before revokeDocumentCore; the form POST /d/:public_id/revoke is also operator-token-gated. Recommendation: None — correct, including operator-only gating, the revoke-before-purge ordering, and the hard byte purge. The kill switch the §8 posture relies on is present and functioning.

Current: Documents are born-live; the document INSERT writes only id/public_id/created_by/slug — no visibility/draft/published_at flag (repo-wide grep finds none; the sole "visibility" hit is prose in an MCP tool description). Ordered implementation plan

Loose priority. Wave 1 closes security-affecting drift on the read/serve boundary; Wave 2 reconciles spec text with deliberate SOLO decisions and hardens the sanitizer/resource model; Wave 3 mops up contract and conformance nits. Each step names the finding(s) it closes.

Wave 1 — security-affecting drift (do first)

  1. Resolve the public by-slug endpoint. Either gate GET /d/by-slug/:slug behind agent/operator auth (resolve only for authenticated principals, per §4) or remove it entirely and resolve slugs only at write time. If a public lookup is a deliberate product decision, record it as an explicit, signed-off divergence from §4 and update skills/publishing.md. (closes the §2/§3 and §4 by-slug findings — the top item)
  2. Decide the serving-isolation posture and write it down. Either formally accept single-origin as a SOLO simplification justified by the no-script sandbox (and downgrade the finding in the spec), or implement public_id-derived origins (<public_id>.docs.<host>) before any allow-scripts is ever considered. Capture the rule "script execution requires per-document origins first." (closes single-origin serving)
  3. Drop SVG <use> from the allowlist (and its per-tag href/xlink:href grant), or restrict it to same-document fragment refs with a documented rationale. Add a corpus negative test pinning <use> removal, mirroring strips_foreign_object. (closes SVG <use> drift)
  4. Add a cheap pre-parse depth/node/attribute gate before clean(), and document Workers' per-isolate CPU limit as the accepted substitute for the killable-worker + concurrency bound. The size cap alone does not bound nesting depth. (closes the resource-bounds finding)

Wave 2 — reconcile deliberate descopes & harden the serve hooks

  1. Build, or formally defer, doc:<slug> write-time link resolution + drop-and-advise. If building: resolve to the target's public_id inside the existing sanitizer pass, drop missing targets to plain text with a new advisory field, and surface it through the MCP tool descriptions + skills/publishing.md. If deferring: record an ADR noting the slug is currently a lookup handle only, which then makes step 1's "remove the public endpoint" option the cleaner choice. (closes the doc: findings; interacts with step 1)
  2. Decide on original-bytes retention. Either add an original_key column + the §5 lazy re-sanitize-on-fetch path, or formally record convert-and-discard as a deliberate divergence and document that profile hardening requires an operator-driven re-PUT/corpus rewrite (not auto-heal). Either way, ensure a future Ammonia hardening triggers a deliberate action rather than silent stale-bytes exposure. (closes the original_key / lazy re-sanitization findings)
  3. Move target="_blank" injection inside the sanitizer profile (an ammonia hook), or add an in-code justification for why the post-clean() byte-splice is exempt from the no-post-mutation rule (operates on normalized serialization, fixed literal, no re-parse, idempotent). (closes the post-sanitize byte-splice finding)
  4. Decide slug-on-revoke. Keep release-on-revoke as a recorded SOLO divergence from §3/§4 (it is safe while no doc: links exist), or — if step 5 builds doc: resolution — switch to keep-claimed-on-revoke so revoked names can't be hijacked. (closes the slug-release-on-revoke finding; depends on step 5)

Wave 3 — contract & conformance reconciliation

  1. Reconcile the operator-auth and OAuth-door spec text. Amend §1 to describe the static OPERATOR_TOKEN bearer (with a documented rotation procedure and never-log discipline) and to acknowledge the OAuth 2.1 + PKCE agent door, so the "strict subset" lineage stays auditable. (closes operator-auth drift + the OAuth-door hygiene note)
  2. Add the Idempotency-Key path or document its omission. A short-TTL KV/D1 dedup keyed on (route, key) with a 48h sweep, or an explicit SOLO descope note in §6/§9 — currently it is undocumented drift, not a stated descope. (closes the idempotency finding)
  3. Settle the addressing/verb model. Either add the PUT /d/by-slug/<slug> create-or-update verb (an authenticated surface the spec permits), or record the public_id + required-If-Match model as the intended SOLO replacement and update §3/§6. (closes the create-or-update verb findings)
  4. Tidy the schema/conformance nits, ideally in one pass: bump the agent-key secret to 32 bytes (or amend the spec to "≥192-bit"); add explicit connect-src 'none' to RAW_CSP; note the deliberate omissions of sanitized_hash/model/label, the sanitizer_profile_version→sanitizer_v rename, and the intentionally-nullable slug, so a future schema diff doesn't read them as accidental. Optionally add Accept: text/markdown handling alongside the existing /text route. (closes the key-entropy, connect-src, schema-delta, and content-negotiation findings)

Two of these (the public by-slug endpoint and single-origin serving) are deviations from boundaries the spec explicitly forbids; the rest are either correct-by-design, recorded descopes, or low-impact nits. The fastest material risk reduction is steps 1–3.
