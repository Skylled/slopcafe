# Byte-exact publishing — design note

**Status:** implemented. The integrity handshake + curl recipe docs landed
first (§5). After operator review the credential-delivery question was
resolved in favor of a **short-lived minted key via MCP** — see §7 for the
decision and what shipped. §1–§6 are preserved as the original analysis;
§7 supersedes §4.3–§4.4 and §6's "safe on-demand disclosure".

## 1. The problem

Publishing a large HTML document (~66 KB, ~85 KB observed) through the MCP
`publish_document` tool requires the agent to put the entire document in the
`html` tool-call **argument**. A tool-call argument is model output: the LLM
regenerates the whole file token-by-token to make the call. For large bodies
that is

- **expensive / slow** — 85 KB of HTML is ~20–30 K output tokens, emitted
  serially; and
- **fidelity-risky** — the model can silently truncate, or insert a
  placeholder (`…`, `[unchanged]`, `<!-- rest omitted -->`) and "finish" a
  call that looks successful but stored the wrong bytes.

In the session that surfaced this, fidelity happened to hold (read-back
confirmed `size_bytes` ≈ source minus stripped comments, head/tail anchors
present). The risk is structural to "large content as a model-generated tool
argument," not a one-off.

## 2. What is the binding constraint? (measure first)

The brief asks whether MCP transport/argument-size limits are the binding
constraint or whether it is purely regeneration fidelity. From the code:

- `/mcp` is Streamable HTTP via `createMcpHandler` (Cloudflare Agents SDK).
  The request body is a JSON-RPC envelope parsed by the SDK; Workers accept
  multi-MB request bodies and there is no small per-argument cap in the
  transport.
- Our own ceiling is `MAX_INPUT_BYTES = 5 MiB` (`src/core.ts`), enforced
  identically on both doors. 85 KB is ~1.6 % of that.

So **transport is not the constraint at 66–85 KB.** The constraint is that the
`html` argument is *generated by the model*. The only way to remove the cost
and the fidelity risk together is a path where the bytes reach the server
**without flowing through model token generation**.

That single criterion is what separates the options below.

> An empirical confirmation is cheap and worth doing once: publish a known
> 85 KB file via the MCP tool and via curl, compare output-token count and
> wall-clock, and diff the read-back. The integrity handshake added in this
> change (§5) gives us the read-back hash tooling to make that diff exact.

## 3. Options

| Option | Bytes bypass the model? | Works without a shell (pure-MCP / claude.ai) | New trust surface | Verdict |
|---|---|---|---|---|
| **A. Disclose a key → curl** | ✅ (shell streams `@file`) | ❌ needs a shell + the file on disk | **High** — hands a long-lived, fleet-wide secret to the model | Right *mechanism* (curl), wrong *delivery* by default — see §6 |
| **B. Fetch-by-URL** | ✅ (server pulls) | ✅ *if* the bytes are already hosted | Medium — SSRF | Viable secondary; defer (§6) |
| **C. Chunked / append publish** | ❌ model still emits every byte across N calls | ✅ | Low | **Reject as the primary fix** — see below |
| **D. Integrity handshake** | n/a (complements A/B and the existing path) | ✅ | None (additive) | **Implement now** (§5) |

### Why chunked-append does *not* solve this

It is the intuitive answer and it is wrong for the stated goal. Splitting the
body into `begin → append → commit` makes each call smaller and individually
more reliable, but the model still **generates every byte** — across more
calls, with more overhead. Token cost is the same or worse. It mitigates
per-call truncation but does nothing for the core problem ("without an LLM
regenerating the content"). It also adds real machinery: a staging area in R2
or D1, a session/upload id, a commit/abort lifecycle, GC for abandoned
uploads, and four new tools. High cost, doesn't meet the goal. Skip.

### The two execution contexts (this is the crux)

"Byte-exact from a file" only means something when there is a canonical byte
source that is *not the model*:

1. **Agent with a shell + the file on disk** (Claude Code, Cowork — this is
   what hit the limit). The byte-exact path **already exists**: `POST /d` with
   `curl --data-binary @file.html`. The shell reads the bytes; the model only
   emits the ~100-token command. The MCP server cannot read the agent's local
   disk over the network, so there is no "publish from path" tool that helps
   here — a non-MCP tool that streams from disk (`Bash` + `curl`) is the
   *only* mechanism. The single missing piece is a **credential at the shell**.

2. **Pure-MCP agent, no shell** (claude.ai web connector). Has only the MCP
   tools; cannot run curl. Its large HTML almost always *originated as model
   output* (it generated the doc in an artifact / earlier turn), so "byte-exact
   from a file" is moot — the bytes were model tokens to begin with, and the
   only concern is re-emission fidelity on the publish call. Fetch-by-URL helps
   this context only when the content is already hosted somewhere fetchable.
   There is no MCP primitive for streaming a file into a tool argument; for
   genuinely large *existing* content this context is simply the wrong tool.

The honest conclusion: **the byte-exact win is the curl/REST path, and it is
already built.** What's missing is (a) telling agents to use it for large
bodies, (b) a safe way to get a credential to the shell, and (c) a tripwire
that makes a botched large upload fail loudly instead of silently.

## 4. Recommendation

1. **Make the existing curl/REST path the documented byte-exact route.**
   Tell shell-capable agents to prefer `POST /d --data-binary @file` over
   regenerating `html` as an MCP argument for any sizable existing file.
   *(Implemented: skill recipe + MCP tool-description pointers.)*

2. **Ship the integrity handshake now** — optional `X-Content-SHA256` on
   `POST /d` / `PUT /d/:id`. Cheap, additive, no new trust surface, and it
   converts the actual observed risk (silent truncation/placeholders) into a
   loud `422`. *(Implemented: §5.)*

3. **Credential delivery: prefer out-of-band, not model-visible.** The
   single-tenant operator already owns every key. The safest delivery is to
   put an `awh_` key in the shell's environment (e.g. `.dev.vars`, a CI
   secret, the Cowork session's env) where the *shell* can read it for curl but
   it never enters the model's context, tool results, or transcript. This keeps
   the OAuth door's core property intact: the model never handles a raw
   long-lived secret.

4. **Do *not* build MCP key-disclosure as specced.** Returning a long-lived,
   full-power, fleet-wide `awh_` key from an MCP tool inverts the entire reason
   the OAuth door exists. If on-demand disclosure is still wanted, build the
   *safe* version (§6) — operator-gated, minting a fresh scoped+expiring key —
   not a handout of the static bearer.

5. **Fetch-by-URL: defer** until there's a concrete hosted-bytes use case;
   it carries SSRF surface that needs its own hardening (§6).

## 5. What this change implements

A content-integrity handshake on the HTTP write path, plus docs.

- `src/integrity.ts` — `sha256Hex`, `normalizeExpectedSha256`,
  `verifyContentIntegrity`. Pure functions; unit-tested in
  `test/integrity.test.mjs`.
- `POST /d` and `PUT /d/:id` accept an optional **`X-Content-SHA256`** header
  (64 lowercase hex, optional `sha256:` prefix). The server hashes the **raw
  received body** — *before* sanitization — and returns
  - `400 bad_integrity_header` if the header is malformed, or
  - `422 integrity_mismatch` (with `expected_sha256`, `actual_sha256`,
    `received_bytes`) if the bytes don't match.
- The check is on the **pre-sanitization** bytes deliberately: it verifies the
  wire ("the bytes I sent arrived intact"), independent of the sanitizer's
  transformations. `modified` / `size_bytes` continue to describe the
  post-sanitize result; integrity is a separate, earlier gate.

**Byte-length is intentionally subsumed by the hash** — a truncated or
placeholder-injected body produces a different digest — so the surface is a
single header. A length-only `X-Content-Length` (cheaper, no hashing) could be
added later if a use case wants it.

**HTTP-only by design.** The value comes from the hash being produced by the
same deterministic tool (the shell) that streams the file. A model cannot
reliably hash 85 KB it is emitting as a tool argument, so there is no coherent
MCP equivalent: an MCP agent with a shell should use curl; one without can't
benefit. The MCP tool descriptions say this explicitly so an agent on that
door knows not to look for it.

Recipe (shell-capable agent):

```sh
sha256sum report.html          # → <hex>  report.html
curl -X POST https://slopcafe.com/d \
  -H "Authorization: Bearer $AGENT_WEB_HOST_KEY" \
  -H "Content-Type: text/html" \
  -H "X-Content-SHA256: <hex>" \
  --data-binary @report.html
# 201 → published byte-exact; 422 integrity_mismatch → the upload was truncated, resend
```

## 6. Deferred / needs operator sign-off

### Safe on-demand key disclosure (if §4.3 out-of-band isn't enough)

If the operator wants an agent to obtain a credential *at runtime* rather than
pre-provisioned, the safe shape is **not** returning the existing static
bearer. It is:

- **Operator-gated.** Disclosure is a privileged action. Gate it on
  `OPERATOR_TOKEN` (or Door A operator consent), never "any agent bearer can
  ask for a key." The MCP tools today close over `props.agentId` and never see
  a raw key — disclosure is a brand-new trust surface and must be the
  operator's explicit choice.
- **Fresh + scoped + short-lived.** Mint a new key on demand, not an existing
  one, scoped to publish/update only and with a short TTL. This needs schema
  work the codebase doesn't have yet: `agent_keys` has no `expires_at` and no
  scope column, and keys are currently full-power (publish/read/list/search
  across the whole fleet). `authenticateAgent` would need an expiry check and a
  scope check.
- **Revocable + logged-by-id.** Reuse `DELETE /admin/keys/:id`. Honor the
  logging discipline (CLAUDE.md): log key *id*/event only, never the secret,
  never the token.

Even done safely, the secret still lands in the model's context and any
transcript/log of the tool result — which is exactly what out-of-band delivery
(§4.3) avoids. Recommend out-of-band first; build this only if there's a
delivery scenario it can't cover.

### Fetch-by-URL (`publish_document_from_url`)

Server pulls bytes from a caller-supplied URL — bytes bypass the model and it
works in the pure-MCP context, *if* the content is hosted. SSRF is the cost:
a Worker `fetch()` can be steered at internal/private services and used as a
relay. Hardening required before shipping: https-only scheme allowlist,
private/loopback/link-local IP blocking, redirect limits, a response-size cap
(reuse `MAX_INPUT_BYTES`), and a timeout — possibly an operator-approved host
allowlist. Defer until a concrete need (e.g. mirroring an existing public
page) justifies the surface.

## 7. Decision & what shipped (credential delivery)

§4.3 recommended delivering the curl credential **out-of-band** (an `awh_` key
in the shell env). That turned out not to fit the real deployment: **Claude's
connector settings have no field for a static bearer**, and — the deeper
blocker — **the connector's OAuth access token is held by the Claude client
transport and is not exposed to the model or the shell**, so neither
"out-of-band key" nor the literal "curl with the OAuth token" is reachable.
(Confirmed live: a session can hold both the Slopcafe MCP tools and a Bash
shell yet have no way to read the bearer to hand to curl.)

Two things also made "make `/d` accept the OAuth token" the wrong build:
the OAuth wrap matches routes by **path prefix, not method**, so making `/d`
an `apiRoute` would 401 the public `GET /d/:id` render path; and the provider
exposes no standalone token-validation method to call from inside the existing
handler.

**Resolution: the already-OAuth-authenticated MCP session mints a short-lived
key on demand.** New MCP tool `create_publish_credential` → `mintEphemeralKey`
(`src/admin.ts`) inserts a normal `awh_` key with a non-NULL `expires_at`
(default 15 min = the OAuth token TTL, max 60, clamped); `authenticateAgent`
rejects expired keys like revoked ones. The agent then curls the **unchanged**
`/d` path with it. Migration `0007_key_expiry.sql` adds the column (NULL =
never expires, so every existing key is untouched).

Why this is sound: the minted key **grants nothing beyond what that MCP
session already wields** through the write tools — it only repackages those
powers into a form `curl` can send — so a short TTL + revocability
(`DELETE /admin/keys/:id`) is the whole containment story. No separate operator
gate is needed (the operator already approved the OAuth client at consent).

**Rejected — presigned one-time upload URL.** Cleaner per-operation, but it's
single-use and publish-specific: the moment a second curl operation is needed
(an update, a second file, a read-back) it doesn't generalize, inviting a
second presigned variant per operation. A short-lived key is the *general*
primitive — one mint covers any number of writes in the window and reuses the
exact documented curl recipe (`--data-binary`, `If-Match`, `X-Content-SHA256`)
with no new endpoint or mental model. The using-agent's ergonomics decided it.

**Deferred — key scope.** v1 ships TTL-only (full-power, short-lived). A
write-only scope was considered and dropped for now: single-tenant, the
damaging vector (overwriting docs) isn't blocked by a write scope anyway, it
would make a natural curl read-back 401 confusingly, and the key already can't
exceed the agent's MCP authority. Scope-limiting is the additive follow-up if
the trust model goes multi-tenant or a minted key should survive a leak with
less blast radius.
